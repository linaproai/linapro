# 测试与验证规则

## 适用范围

本规则约束单元测试、E2E 测试、反馈修复测试覆盖、测试 helper、fixture、页面对象、runner 配置、治理验证和任务完成前的验证门禁。

## 强制遵守场景

以下任一场景命中时，必须先读取并遵守本规则：

- 新增或修改单元测试、E2E 测试、测试 helper、fixture、页面对象、runner 配置或测试数据
- 修复 bug、处理反馈、改变功能行为或用户可观察结果
- 修改权限、缓存、数据权限、插件桥接、接口联动或端到端流程
- 纯治理类反馈需要选择 OpenSpec 校验、静态扫描、文件检查或格式检查作为验证方式

## 单元测试要求

- 单元测试必须自包含且顺序无关。
- 每个测试方法必须自行构造和清理所需场景与数据。
- 禁止依赖其他测试方法先运行或遗留的副作用。
- 需要共享逻辑时只能共享 helper/fixture，并由当前测试显式调用。
- 清理逻辑必须使用当前测试注册的`defer`、`t.Cleanup`或等价机制完成。
- 涉及全局状态时，当前测试必须保存原值并恢复。

## E2E 测试要求

- 测试用例必须完整覆盖业务模块的各项操作，例如增删改查，保证功能完整性和可用性。
- 所有 E2E 用例需要在`tasks.md`中有工作记录，并且使用`lina-e2e`技能生成和管理对应测试用例。
- 修复 bug 或新增功能涉及用户可观察行为变化时，必须编写或更新对应 E2E 测试用例。
- 涉及用户可观察页面、路由、表单、表格、权限交互、接口联动或端到端工作流的 bugfix，必须使用 E2E 测试。
- 纯内部重构且无 UI 变化可豁免新增 E2E，但需运行已有测试套件或更贴合的验证确认无回归。
- 使用 Playwright 等测试工具创建截图或者临时文件时，文件必须放置到项目根目录`temp/`目录下。测试期间临时创建的文件必须以日期时间作为文件名前缀，例如`20260101-120000-screenshot.png`，以避免文件名冲突和覆盖。

## E2E 文件组织要求

- E2E 测试文件名必须使用模块本地递增的`TC{NNN}-{brief-name}.ts`前缀。
- 同一模块目录内编号从`TC001`开始连续递增。
- 源码插件专属 E2E 必须保留在插件自己的`hack/tests/e2e/`目录。
- 插件专属 POM/helper 必须保留在同插件的`hack/tests/pages/`和`hack/tests/support/`目录，不得回流到宿主测试目录。

## E2E 质量审查要求

- 审查涉及用户可观察页面、路由、表单、表格、权限交互、接口联动、端到端工作流、E2E 测试资产或功能行为类反馈修复时，必须执行 E2E 质量审查。
- E2E 质量审查关注测试是否能够稳定证明业务行为正确，只约束测试结果和审查证据，不规定具体定位方式、封装模式或编码实现方式。
- 本次变更涉及的核心用户路径必须有对应 E2E 或等价自动化验证；覆盖缺口影响本次变更验收时，判定为严重问题。
- 新增、修改或修复的业务行为必须覆盖成功路径和必要失败路径；功能行为类 bugfix 必须覆盖原问题复现场景和修复后的预期结果，缺少有效验证时判定为严重问题。
- E2E 断言必须能够证明业务结果、数据状态、权限结果或用户可观察状态变化。仅验证页面元素存在、弹窗关闭、按钮可点击或页面跳转，且不能证明业务结果正确时，判定为质量问题；如果该断言是本次变更的唯一验收依据，判定为严重问题，否则判定为警告。
- 启用 `i18n` 的插件或宿主页面，E2E 断言必须覆盖关键用户可见文案的翻译正确性。仅使用双语正则（如 `/中文|English/i`）定位元素不能证明翻译正确，因为正则在原始 i18n key 或错误语言下也可能匹配。必须至少在核心路径中断言目标语言的实际文案出现。
- 表单选项（RadioGroup、Select、Checkbox）、下拉菜单、按钮文案、表格列标题等通过 `$t()` 动态渲染的 i18n 内容，若属于本次变更范围，必须有断言验证其显示为翻译后的文案而非原始 key（如 `plugin.xxx.common.enabled`）。缺少此类断言时，判定为警告；若该选项文案是本次变更的唯一用户可观察输出，判定为严重问题。
- E2E 不得依赖其他测试用例的执行结果、残留数据、执行顺序或与业务语义无关的偶然结构和状态。稳定性问题导致测试不可独立运行或结果不可信时，判定为严重问题，否则判定为警告。
- 表单、删除、批量操作、状态切换、权限拒绝、唯一约束、导入导出等场景，若属于本次变更范围，必须覆盖关键异常路径。本次变更引入或修复的异常路径缺少验证时，判定为严重问题；与本次变更无直接关系的异常路径缺口判定为警告或改进建议。
- E2E 质量审查报告必须说明是否触发质量审查及触发原因，并说明覆盖判断依据，例如 OpenSpec 任务、增量规范、代码变更、页面行为、接口联动或现有测试证据。
- 质量审查发现问题时，必须给出文件路径、行号、严重性和原因；覆盖缺口应描述建议补充的业务场景，不得要求采用某种具体实现方式作为唯一修复方案。
- 审查范围不涉及 E2E 测试资产、用户可观察行为、端到端工作流或功能行为类反馈修复时，必须记录未触发 E2E 质量审查及跳过原因。

## E2E 截图验证要求

以下要求适用于执行 E2E 测试的代理或工具支持多模态图片识别能力的场景：

- 涉及用户可观察页面的 E2E 测试，在关键交互步骤后必须捕获浏览器截图，并使用多模态图片识别能力审查截图内容。截图可以发现文本断言无法捕获的问题，包括但不限于：`i18n` key 未翻译（页面显示 `plugin.xxx.common.enabled` 而非"启用"）、表单标题或按钮文案换行截断、表格列挤压重叠、操作后接口报错的 toast/notification 文案、空状态或异常状态页面展示错误。
- 截图必须覆盖以下关键节点：页面首次加载完成、表单抽屉/弹窗打开后、表单提交或操作执行后、列表筛选或搜索后、异常路径（如接口报错、权限拒绝）触发后。
- 多模态审查截图时，必须检查：页面文案是否为预期语言的翻译文本而非原始 `i18n` key；表单布局是否正常（无换行截断、无元素重叠）；操作反馈是否正确展示（成功提示、错误提示、loading 状态）；数据是否正确渲染（列表数据、详情内容、统计数字）。
- 若执行代理不支持多模态图片识别，必须在审查结论中明确说明跳过截图验证的原因，并以文本断言作为替代验证手段。
- 截图验证发现的问题必须标注为警告或严重问题，附上截图文件路径和问题描述。

## Bugfix 反馈测试要求

- 所有以`lina-feedback`或等价反馈流程处理、且涉及功能行为、业务逻辑、用户可观察结果、运行时接口、缓存、权限、数据权限、插件桥接等可执行行为变化的 bugfix，必须新增或更新能够复现问题并验证修复有效性的自动化测试。
- 后端纯逻辑、服务层、工具函数、缓存、权限、数据权限、插件桥接等内部行为优先使用单元测试。
- 用户可观察行为或跨模块工作流缺陷必须使用 E2E 测试。
- 测试应覆盖原始问题的失败场景和修复后的预期行为。
- 功能行为类 bugfix 缺少单元测试或 E2E 测试不得标记完成，也不得通过`lina-review`。

## 治理类反馈验证要求

- 纯项目治理类反馈不要求新增单元测试或 E2E。
- 适用验证方式包括`openspec validate`、静态扫描、文件存在性检查、格式检查或审查结论。
- 治理类反馈必须说明为什么不需要运行时测试，并提供匹配的治理验证证据。

## 任务完成门禁

- 修改完成后必须运行相关新增或更新测试、识别出的回归测试或对应治理验证，并确认通过，再标记任务完成。
- 如果回归测试失败且与当前变更相关，必须直接修复。
- 如果发现独立问题，应记录为新的反馈任务。

## 审查要求

- 审查必须判断变更是否涉及可执行行为变化。
- 审查必须确认功能行为类 bugfix 具备单元测试或 E2E 覆盖。
- 审查必须确认治理类反馈具备匹配治理验证证据。
- 审查必须确认测试可单独运行、顺序无关且清理完整。
- 审查必须确认 E2E 文件编号和插件测试目录归属合规。
- 审查必须按 E2E 质量审查要求确认覆盖有效性、断言有效性、稳定性、异常路径和审查输出证据。
- 审查涉及启用 `i18n` 的页面时，必须确认 E2E 断言覆盖关键文案的翻译正确性，而非仅依赖双语正则定位。
