## Feedback

- [x] **FB-6**: `make upgrade` 的实现必须收敛到仓库根目录 `hack/upgrade-source/`，并且只读取 `apps/lina-core/hack/config.yaml` 中的数据库连接与升级元数据，禁止继续依赖运行时配置文件。
- [x] **FB-7**: `init` / `mock` 必须支持按阶段切换 SQL 资产来源，运行时默认读取 embedded FS，开发态 `Makefile` 显式读取本地 SQL 文件。
- [x] **FB-5**: 框架元数据中的 `homepage` 应表示官网地址，并新增独立的仓库地址字段供系统信息展示和升级命令使用。
- [x] **FB-4**: `internal/cmd` 的单元测试文件应按命令组件职责重新内聚，避免保留已删除 helper 文件对应的测试文件名。
- [x] **FB-3**: `internal/cmd` 中与 `init`、`mock`、`upgrade` 相关的非测试逻辑应尽量内聚到对应命令文件或 `cmd.go`，减少零散辅助文件。
- [x] **FB-1**: 源码升级不再引入数据库升级状态表、升级记录表和 SQL 游标表。
- [x] **FB-2**: `make upgrade` 执行宿主 SQL 时从第一条 SQL 开始全量执行，不再依赖数据库中的执行位置。
- [x] **FB-8**: 开发态升级工具目录应从 `hack/upgrade-framework` 重命名为更准确的 `hack/upgrade-source`，并保证组件根目录只保留 `main.go`，其余功能源码全部下沉到 `internal` 独立组件中。
- [x] **FB-9**: 源码插件升级治理实现必须从 `internal/service/plugin` 主包中抽离为独立宿主侧组件，由 `make upgrade` 与宿主启动校验复用同一套核心逻辑；`pkg` 层只保留稳定导出契约与必要门面，不再承载真实升级实现细节。
- [x] **FB-10**: 补齐源码插件升级治理的自动化验证，新增宿主侧 `sourceupgrade` 组件/门面单元测试，并补充 `TC0106` 覆盖“源码发现更高版本后不会自动切换，插件列表与当前用户菜单仍保持当前生效版本”的端到端场景；显式升级切换路径继续由宿主/命令侧单元测试验证。
