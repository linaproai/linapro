## Context

用户已经明确，这一轮要解决的是**源码阶段升级**，而不是运行中升级。源码阶段的版本历史天然由 Git 管理，因此不需要在数据库里再维护一套框架升级状态、升级记录和 SQL 游标。真正需要落地的是一个足够直接、稳定、可执行的命令：

1. 读取当前项目源码中的框架版本；
2. 读取目标标签中的框架版本；
3. 目标版本更高时，用目标标签代码覆盖当前项目；
4. 再把宿主 `manifest/sql/` 从头执行一遍完成数据库对齐。

因此，本轮设计从之前的“数据库基线驱动升级”调整为“源码元数据 + 全量 SQL 重放”的 P0 方案。

## Goals / Non-Goals

**Goals**
- 为框架源码升级提供一个正式且可执行的 CLI 命令。
- 在升级开始前完成备份提醒、Git 工作区脏检查和版本比较。
- 通过当前项目 `metadata.yaml` 与目标标签 `metadata.yaml` 完成版本判断。
- 在升级时用目标标签代码覆盖本地框架代码，并从第一条宿主 SQL 开始按顺序全部执行。
- 让系统信息页顶部项目卡片直接展示 `metadata.yaml` 中的框架元数据。

**Non-Goals**
- 本轮不实现业务系统运行时版本治理（业务版本号、业务升级包目录、业务 SQL 账本等 P1 范围）。
- 本轮不新增数据库升级状态表、升级记录表或 SQL 执行游标表。
- 本轮不实现自动回滚；升级失败后只提供命令级错误输出，供人工处理。
- 本轮不自动清理上游版本已删除的本地旧文件。

## Decisions

### 1. 当前框架版本以本地 `metadata.yaml` 为准

源码升级面向的是开发阶段项目，因此“当前项目正在使用哪个框架版本”直接取自当前仓库中的 `apps/lina-core/manifest/config/metadata.yaml`。这比依赖数据库更符合源码升级的真实语义，也避免引入多余状态同步问题。

### 2. 不引入数据库升级元数据表

本轮不新增框架升级状态表、升级记录表、SQL 步骤表。原因是：

- 源码阶段版本历史应由 Git 管理；
- 升级是否已完成，可以通过本地代码版本和命令执行结果判断；
- 再引入一套数据库账本，只会增加复杂度和初始化成本。

### 3. 目标版本来自目标标签代码中的 `metadata.yaml`

升级命令执行时：

- 若未显式指定目标标签，则默认取远端最新语义化标签；
- 将目标标签代码拉到临时目录；
- 读取目标标签中的 `apps/lina-core/manifest/config/metadata.yaml`，得到目标框架版本；
- 将本地版本与目标版本做语义化比较。

只有目标版本更高时，升级才继续执行。

### 4. 升级前必须完成安全检查

升级命令开始前必须完成以下检查：

- 输出备份提醒；
- Git 工作区必须干净；若存在未提交或未暂存的修改，直接拒绝升级；
- 当前项目和目标标签中的框架版本都必须是合法 semver；
- 目标版本必须高于当前版本。

这样可以避免在脏工作区里强行覆盖代码，导致冲突和不可控结果。

### 5. 升级采用“先覆盖代码、再全量执行宿主 SQL”的顺序

本轮遵循用户指定的升级顺序：

1. 获取目标标签代码；
2. 将目标标签代码覆盖到当前仓库；
3. 从 `manifest/sql/` 第一条 SQL 开始按文件顺序执行全部宿主 SQL；
4. 命令完成后，由当前源码中的 `metadata.yaml` 自然体现新版本。

这里不再做“从某个 SQL 游标继续执行”的逻辑，所有宿主 SQL 都依赖幂等性保障重复执行安全。

### 6. `init` 保持 bootstrap 语义，不承担升级基线职责

`init` 命令仍然只是初始化命令，不是升级命令，也不负责写入任何升级状态。它只负责执行宿主 bootstrap SQL；正式升级统一由 `upgrade` 命令承担。

### 7. 框架元数据统一收口到 `metadata.yaml`

系统信息页顶部项目卡片展示的名称、版本、介绍、官网、仓库地址、协议，不再由前端单独维护，而统一由 `apps/lina-core/manifest/config/metadata.yaml` 中的框架元数据提供。其中 `homepage` 表示官网地址，`repositoryUrl` 表示源码仓库地址，再由后端接口返回给前端展示。

这样可以保证升级后前后端展示信息一致，并且升级命令也能直接读取同一份版本元数据。

## Risks / Trade-offs

- **全量重放 SQL 依赖幂等性**：宿主 `manifest/sql/` 中的 DDL 和 Seed DML 必须保持可重复执行，否则升级时会失败。
- **覆盖代码但不自动删除上游已移除文件**：可以降低误删用户文件的风险，但会留下少量需要人工清理的旧文件。
- **升级失败后代码可能已被覆盖、SQL 部分执行**：本轮不做自动回滚，因此必须在升级前提醒用户备份。
- **默认上游仓库取框架仓库地址**：如果下游项目的上游仓库不等于主页地址，操作者需要通过命令参数显式传入仓库地址。

## Migration Plan

1. 先为宿主新增框架元数据段，并更新系统信息接口/页面展示。
2. 新增 `upgrade` 命令和 `make upgrade` 入口。
3. 实现目标标签解析、Git 脏检查、版本比较、代码覆盖与宿主 SQL 全量执行。
4. 增加版本比较、目标标签解析、Git 工作区检查等自动化测试。

## Open Questions

- 是否要在后续迭代中增加“删除上游已移除文件”的可选开关，本轮先不做。
- 业务系统运行时升级（P1）何时独立立项推进，本轮不阻塞源码升级落地。
