## Context

当前仓库已经对插件安装、升级、迁移记录和失败回滚建立了较完整的治理模型，但宿主与业务系统层面仍缺少对应的正式升级机制。现状中：

- `make init` / `go run main.go init` 主要承担宿主数据库初始化语义，适合新装或重置，不适合正式升级。
- `manifest/sql/` 目前更接近“宿主默认建表 + Seed 数据”的启动资产，而不是可审计、可恢复、可分阶段执行的正式升级账本。
- 对开源用户而言，最早遇到的升级问题往往不是生产运行时数据库，而是“我基于旧版本框架做了二开，如何合并新版本而不被核心改动击穿”。
- 对已部署业务系统而言，框架后续必须提供一套通用的业务版本升级机制，帮助业务项目声明版本、组织升级 SQL、记录执行历史，并在失败时能够诊断与恢复。

本次变更因此拆分为两个升级模型，并明确优先级：先解决源码升级治理，再定义运行时升级内核。两者共享版本治理、变更分级、审计和预检查原则，但落点不同：源码升级偏契约与流程，运行时升级偏框架机制与元数据管理。

## Goals / Non-Goals

**Goals:**
- 定义“源码升级（给二开项目）”与“运行时升级（给已部署系统）”两套正式升级模型，并明确 P0/P1 优先级。
- 为源码升级建立稳定的兼容性承诺、扩展边界分级、升级说明交付要求与升级前检查机制。
- 为运行时升级建立统一的版本号、升级资产、迁移记录、幂等执行、失败诊断与恢复治理模型。
- 明确 `init` / `mock` 与正式升级命令的职责分离，避免启动命令与升级命令混用。
- 为后续实现提供可执行的规范基础，使升级能力可进入 CLI、数据库元数据、CI 验证与发布流程。

**Non-Goals:**
- 不在本次设计中承诺任意版本跨度的直接升级；默认只保证受支持升级路径内的平滑升级。
- 不在本次设计中承诺通用的数据库降级 SQL 或任意双向回滚；运行时升级默认采用前向迁移与失败恢复策略。
- 不把宿主所有内部目录都定义为稳定扩展面；只有明确列出的扩展点、公共契约与文档声明边界才承诺兼容。
- 不复用插件升级表直接承载业务系统升级元数据；业务系统升级需要独立治理模型。

## Decisions

### 1. 采用“共享治理原则 + 双升级模型”的整体架构

本次变更将升级问题拆成两个能力域：

- `source-upgrade-governance`：面向二开项目源码升级，关注版本兼容策略、推荐扩展方式、破坏性变更治理、文档和预检查。
- `runtime-upgrade-governance`：面向已部署业务系统运行时升级，关注业务版本号、框架版本号、升级资产、迁移记录、锁、失败恢复。

这样拆分的原因是两类用户面对的升级对象不同：前者升级的是“代码基线”，后者升级的是“运行中系统状态”。若合并为单一模型，容易把文档/兼容策略与数据库迁移/执行编排混为一谈，导致规范难以落地。

**备选方案：** 使用一套统一的“升级规范”同时覆盖源码与运行时。  
**放弃原因：** 关注点差异过大，任务优先级与落地节奏不同，容易让 P0 的源码升级治理被运行时细节拖慢。

### 2. 源码升级优先级定为 P0，并以“稳定扩展边界 + 发布交付”作为核心契约

源码升级是开源发布前必须先解决的问题，因此优先级高于运行时升级。P0 方案包含四个核心约束：

- 框架版本必须使用明确的版本分级策略，并说明支持的升级路径。
- 框架必须把可兼容扩展面分为“稳定扩展面 / 限定兼容面 / 内部实现面”。
- 二开项目必须优先通过插件、扩展点、配置与适配层进行定制，而不是直接侵入宿主核心目录。
- 每次发布都必须同步交付升级说明、兼容性说明和破坏性变更说明。

源码升级的本质不是“保证所有文件永远不变”，而是“清楚地声明哪些东西可以依赖、哪些东西不要依赖，以及每次版本升级用户应该怎么做”。

**备选方案：** 优先实现运行时升级命令与数据库账本，源码升级先靠 README 约定。  
**放弃原因：** 开源用户最先感知到的痛点往往是源码合并与定制漂移，而不是生产数据表升级；若不先定义源码升级契约，开源后很快会积累不可控分叉。

### 3. 将正式升级与初始化命令彻底分离

`init` / `mock` 继续保留其“初始化/演示数据”职责，不承担正式升级语义。正式升级必须引入独立能力与命令，例如后续的 `upgrade` / `migrate` / `doctor` 入口，由它们负责：

- 版本前置检查；
- 升级计划解析；
- 分布式锁或全局互斥；
- 迁移记录落库；
- 失败停止与恢复。

这项决策的核心是防止用户误把“全量初始化命令”当成“升级命令”使用，从而造成重复执行、意外写入或运行时资产被误扫。

**备选方案：** 继续复用 `init`，通过参数模式区分初始化和升级。  
**放弃原因：** 命令语义模糊，容易导致误操作，而且初始化扫描逻辑与升级账本逻辑完全不同。

### 4. 运行时升级采用“前向迁移 + 版本账本 + 可恢复执行”模型

运行时升级不再依赖“重新执行一遍全部 SQL”，而是为业务系统提供正式升级账本。核心模型如下：

- 业务系统声明自己的 `businessVersion`。
- 框架在升级元数据中记录当前业务版本、目标业务版本、当前框架版本、兼容框架版本范围与升级状态。
- 每个业务版本绑定独立升级资产目录，目录中维护按顺序执行的 SQL 文件和元数据声明。
- 框架为每个升级步骤记录文件名、校验值、执行顺序、执行状态、执行时间和错误信息。
- 已成功执行的步骤不得重复执行；失败步骤必须保留上下文，供后续恢复或人工介入。

这与插件生命周期中的 release / migration 思路一致，但对象变为“业务系统版本”，不能直接借用插件表结构。

**备选方案：** 继续依赖 `manifest/sql/*.sql` 顺序执行，或每次部署重新执行一遍全量 SQL。  
**放弃原因：** 缺少版本维度、审计能力、失败恢复语义与校验值治理，无法满足业务系统平滑升级要求。

### 5. 运行时升级资产采用独立目录与版本绑定，而不是复用宿主 bootstrap SQL 目录

为避免与宿主 `manifest/sql/` 启动资产混淆，运行时升级将使用独立的业务升级资产目录；推荐模型为：

- 以业务版本为目录单元；
- 每个版本目录内维护升级元数据和按顺序执行的 SQL 文件；
- SQL 文件命名继续采用零填充序号前缀，且必须幂等；
- 升级执行时只扫描目标版本链路内尚未成功的步骤。

这样可以同时满足“按版本组织升级包”和“按步骤记录执行历史”的要求。

**备选方案：** 沿用宿主 `manifest/sql/` 目录并通过更复杂的文件命名表达版本。  
**放弃原因：** 宿主 bootstrap SQL 与业务运行时升级资产职责不同，复用目录会让初始化与升级边界再次模糊。

### 6. 版本治理至少区分框架版本、业务版本和迁移步骤三个维度

运行时升级至少要追踪以下三个维度：

- `frameworkVersion`：当前部署所依赖的 LinaPro 框架版本；
- `businessVersion`：业务系统自身版本；
- `migrationStep`：当前版本包中的具体升级步骤。

源码升级则主要依赖框架版本与兼容性分级，但也需要记录“当前项目基于哪个框架版本开始升级”。通过这三个维度，框架可以表达“业务版本要求的最小/最大框架版本范围”以及“升级已执行到哪个步骤”。

**备选方案：** 只记录一个“当前版本”字段。  
**放弃原因：** 无法区分框架升级与业务升级，无法表达步骤级审计，也无法做兼容性校验。

### 7. 失败恢复采用“停止 + 保留上下文 + 支持续跑”，不承诺通用 down migration

运行时升级遇到失败时，系统必须立即停止后续步骤，保留以下内容：

- 当前业务版本与目标版本；
- 已成功/失败的步骤；
- 对应校验值；
- 错误信息与时间。

后续恢复默认采用“修复后续跑”或“人工介入 + 重试”，而不是要求每个业务版本都提供完整降级 SQL。对于存在破坏性结构调整的场景，框架鼓励业务采用 expand/contract 两阶段演进。

**备选方案：** 要求每个升级包同时提供一套降级 SQL。  
**放弃原因：** 对开源业务项目负担过高，而且大量真实场景难以保证安全降级。

### 8. 升级能力必须进入 CI 与发布清单，而不是只停留在文档

升级治理一旦成为正式能力，就必须进入验证流程。后续实现阶段至少需要覆盖：

- 新装初始化验证；
- 从受支持旧版本到新版本的增量升级验证；
- 升级失败后的恢复/续跑验证；
- 源码升级前检查与文档清单校验。

这样可以保证升级能力不是发布时临时补的说明，而是框架本身的一部分。

## Risks / Trade-offs

- **[稳定扩展边界定义过宽]** → 仅把已文档化、具备长期治理价值的扩展点列为稳定扩展面，其余默认保持内部实现语义。
- **[运行时升级元数据模型设计过重]** → 第一阶段采用单业务系统、单版本链路模型，只保留必要字段，后续再扩展多应用/多租户场景。
- **[SQL 幂等约束执行成本上升]** → 通过目录约定、校验值、执行记录和 lint/测试约束降低人工遗漏风险。
- **[不提供通用 down migration 引发部分用户顾虑]** → 在规范中明确推荐备份、分阶段变更和失败续跑策略，避免给出难以兑现的“万能回滚”承诺。
- **[发布文档容易与实现漂移]** → 把升级文档、兼容性说明和 breaking changes 纳入版本发布清单与 CI 校验范围。

## Migration Plan

1. 先落地源码升级治理规范，明确版本分级、扩展边界、发布交付物和升级前检查要求。
2. 在数据库启动命令规范中明确 `init` / `mock` 与正式升级语义隔离，防止新增运行时升级能力后出现误用。
3. 再落地运行时升级治理规范，定义业务版本、升级资产目录、迁移记录和失败恢复原则。
4. 实现阶段按优先级推进：先文档与契约，再 CLI/元数据，再 CI 验证与样例。
5. 发布时要求至少提供一次“旧版本二开项目升级演练”和一次“业务版本增量升级演练”作为验收基线。

## Open Questions

- 运行时升级资产目录最终命名采用 `manifest/upgrades/` 还是更强调业务语义的专用目录名，需在实现阶段统一。
- 升级前检查是否只提供 CLI 入口，还是同时提供管理后台可视化入口，当前不阻塞规范制定。
- 运行时升级元数据表的精确命名与字段拆分仍可在实现阶段细化，但能力边界和必须记录的字段范围已在本设计中锁定。
