## ADDED Requirements

### Requirement: 框架必须为业务系统提供正式的运行时版本治理模型
框架 SHALL 为已部署业务系统提供正式的运行时版本治理模型，至少能够区分并记录业务版本号、框架版本号和升级状态，用于升级前校验、执行过程跟踪和升级后审计。

#### Scenario: 记录当前运行版本信息
- **WHEN** 业务系统完成初始化或一次正式升级
- **THEN** 框架必须能够查询当前业务版本号与当前框架版本号
- **AND** 必须保留该状态用于后续升级判断与审计

#### Scenario: 升级目标与当前状态不一致
- **WHEN** 业务系统准备升级到新的业务版本
- **THEN** 框架必须能够表示当前版本、目标版本和当前升级状态
- **AND** 升级流程必须基于这些元数据决定是否允许执行

### Requirement: 框架必须支持与业务版本绑定的升级资产目录
框架 SHALL 为业务系统定义与业务版本绑定的正式升级资产目录与文件约定，使每个目标业务版本都能声明自己的升级元数据和按顺序执行的 SQL 步骤，并与宿主 bootstrap SQL 目录隔离。

#### Scenario: 新业务版本提交升级资产
- **WHEN** 业务项目新增一个待发布的业务版本
- **THEN** 该版本必须在独立的升级资产目录中声明版本元数据和升级 SQL 文件
- **AND** 这些资产不得与宿主 `manifest/sql/` 初始化目录混放

#### Scenario: 升级 SQL 文件需要顺序执行
- **WHEN** 一个业务版本包含多个升级步骤
- **THEN** 框架必须按照稳定顺序解析这些 SQL 文件
- **AND** 升级记录必须能标识每个步骤对应的文件名或稳定步骤键

### Requirement: 运行时升级 SQL 必须满足幂等和可审计要求
框架 SHALL 要求业务系统的运行时升级 SQL 满足可重复执行且结果一致的幂等约束，并在执行时记录步骤级校验值、执行状态、执行时间和错误信息，防止重复执行或状态漂移。

#### Scenario: 已成功步骤重复进入升级流程
- **WHEN** 某个升级步骤已经被记录为成功执行
- **THEN** 框架不得再次重复执行同一校验值的成功步骤
- **AND** 升级账本必须能够据此跳过已完成步骤

#### Scenario: 升级步骤执行失败
- **WHEN** 某个升级 SQL 步骤执行失败
- **THEN** 框架必须记录失败步骤、校验值、时间和错误信息
- **AND** 必须立即停止后续步骤执行

### Requirement: 框架必须为运行时升级提供前置检查、互斥执行与失败恢复机制
框架 SHALL 在运行时升级开始前执行版本兼容性与状态预检查，并通过互斥机制保证同一业务系统同一时刻只能有一个升级流程执行；当升级失败时，系统 MUST 保留上下文并支持后续恢复或续跑。

#### Scenario: 升级前置检查失败
- **WHEN** 目标业务版本与当前框架版本不兼容，或当前系统存在未完成升级
- **THEN** 框架必须拒绝开始新的升级流程
- **AND** 返回明确的失败原因

#### Scenario: 并发触发多次升级
- **WHEN** 同一业务系统在同一时间被多个进程或节点触发升级
- **THEN** 框架必须保证只有一个升级流程获得执行权
- **AND** 其他请求必须被拒绝、等待或提示稍后重试

#### Scenario: 升级失败后恢复执行
- **WHEN** 某次升级在部分步骤执行后失败
- **THEN** 框架必须保留当前业务版本、目标版本和每个步骤的执行状态
- **AND** 后续恢复流程必须能够基于账本决定从何处继续
