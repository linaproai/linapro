## Why

框架源码升级发生在二开项目的开发阶段，不是业务系统运行中的在线升级。这个阶段最核心的问题只有两个：

1. 当前项目源码正在使用哪个框架版本；
2. 当切到更高版本标签后，如何把宿主 SQL 从头重新执行一遍完成数据库对齐。

这里不需要再额外引入数据库升级状态表、升级记录表或 SQL 游标表。源码历史本来就应当由 Git 管理；升级是否执行成功，应当通过命令执行结果和 Git 变更来判断，而不是再给宿主数据库增加一套框架升级账本。

因此，本次迭代调整为更直接的 P0 方案：当前版本以项目内 `metadata.yaml` 为准，升级时比较本地版本与目标标签版本，目标版本更高才执行代码覆盖；覆盖完成后，从宿主 `manifest/sql/` 的第一条 SQL 开始按顺序全部执行一遍。业务系统运行时升级仍然只保留为 P1 方向说明，不在本轮实现。

## What Changes

- 在 `apps/lina-core/manifest/config/metadata.yaml` 中新增框架元数据，统一维护：名称、版本号、介绍、项目官网、仓库地址、开源协议。
- 系统信息接口与系统信息页面顶部项目卡片改为读取上述框架元数据，避免前后端各自硬编码。
- 新增 `upgrade` 命令和 `make upgrade` 入口，作为正式的框架源码升级命令。
- 升级流程固定为：
  - 先提醒备份；
  - 检查当前 Git 工作区是否干净；
  - 读取当前项目 `metadata.yaml` 中的框架版本；
  - 解析目标标签版本并比较大小；
  - 若目标版本不高于当前版本，则直接退出；
  - 拉取目标标签代码覆盖本地框架代码；
  - 从 `manifest/sql/` 第一条 SQL 开始按顺序执行全部宿主 SQL。
- 不新增任何框架升级状态表、升级记录表或 SQL 游标表。
- `init` 与 `mock` 继续保持显式确认机制，但 `init` 不承担建立升级账本的职责。

## Capabilities

### New Capabilities
- `source-upgrade-governance`: 提供面向二开项目的源码升级能力，包括版本比较、Git 工作区检查、目标标签代码覆盖和宿主 SQL 全量重放。

### Modified Capabilities
- `database-bootstrap-commands`: `init` 与 `mock` 继续保持显式确认机制；其中 `init` 只负责宿主 bootstrap 初始化，不再维护任何框架升级元数据。

## Impact

- `apps/lina-core` 需要新增升级服务和 CLI 命令，但不需要新增升级元数据表、DAO/DO/Entity。
- `apps/lina-core/manifest/config/metadata.yaml` 需要承载统一框架元数据。
- `apps/lina-vben` 的系统信息页需要改为展示后端返回的框架元数据。
- 宿主 SQL 需要继续满足可重复执行且结果一致的幂等性要求，以支撑升级时从头执行全部 SQL。
- 当前这轮变更只覆盖 P0 源码升级；P1 业务系统运行时升级机制在后续迭代推进。
