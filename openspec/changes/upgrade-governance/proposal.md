## Why

当前框架已经具备初始化数据库、插件生命周期和模块化交付能力，但仍缺少面向开源用户的正式升级治理方案。若在开源发布前不明确源码升级与运行时升级的边界、约束、命令语义与兼容性承诺，后续用户基于旧版本框架二开或将业务系统升级到新版本时，极易遭遇非兼容变更、迁移失败和回滚困难，直接影响用户体验与社区信任。

本次变更需要在框架开源前把“如何升级”定义为正式能力，而不是零散文档约定。与此同时，升级需求存在两个不同对象：一类是基于框架继续研发的二开项目，另一类是已部署业务系统的运行时版本演进；两者需要共享治理原则，但必须拆分为不同模型与任务优先级，其中源码升级优先级更高。

## What Changes

- 定义双升级模型，并明确任务优先级：
  - **P0：源码升级（给二开项目）**，定义框架版本兼容策略、扩展边界、推荐定制方式、升级路径、破坏性变更治理、升级前检查与升级文档交付要求。
  - **P1：运行时升级（给已部署系统）**，定义框架为业务系统提供的版本号、迁移 SQL 绑定、迁移历史记录、幂等执行、执行顺序、审计与失败恢复机制。
- 为源码升级建立稳定治理契约，明确哪些目录、扩展点与接口属于推荐扩展面，哪些属于宿主内部实现，不承诺兼容。
- 为运行时升级建立统一治理机制，要求框架能够管理业务系统版本号、框架版本号、迁移文件执行状态，以及与版本绑定的升级元数据。
- 明确宿主 `init` / `mock` 命令仅用于初始化与演示数据加载，不再承担正式升级语义；正式升级需使用独立升级能力与命令语义。
- 建立开源发布前必须交付的升级文档与版本说明基线，确保每个后续版本都能为用户提供可执行的升级指南。

## Capabilities

### New Capabilities
- `source-upgrade-governance`: 定义二开项目的源码升级模型、兼容性承诺、扩展边界、升级路径与发布交付要求。
- `runtime-upgrade-governance`: 定义业务系统运行时升级模型，包括业务版本管理、迁移文件治理、执行记录、幂等与失败恢复机制。

### Modified Capabilities
- `database-bootstrap-commands`: 调整宿主数据库命令边界，明确 `init` / `mock` 只负责初始化与演示数据加载，不承担正式升级语义，并为后续独立升级命令预留规范边界。

## Impact

- `openspec/specs/` 需要新增源码升级治理与运行时升级治理规范，并更新数据库启动命令规范。
- `apps/lina-core` 后续需要新增或调整升级相关命令、版本元数据、迁移记录存储与执行编排能力。
- 宿主与业务系统的 SQL 文件交付规范、版本号管理方式、升级验证流程与发布文档要求会被统一治理。
- 后续开源发布流程需要同步输出升级说明、兼容性策略与破坏性变更说明，避免用户在升级时依赖猜测或人工比对。
