## Why

上一轮 `upgrade-governance` 已经把**框架源码升级**收敛为开发阶段显式执行的能力：通过 `make upgrade` 完成版本比较、代码覆盖和宿主 SQL 全量重放。这解决了二开项目升级框架时的核心问题，但**插件升级治理**仍然不完整，尤其是源码插件。

当前插件升级存在两个明显缺口：

1. **源码插件缺少正式升级路径**。源码插件已经随宿主编译集成，但当前仅有“发现/安装/卸载”语义；当数据库中记录的已安装版本还是旧版本，而源码树里的 `plugin.yaml` 已经提升到更高版本时，系统缺少一个开发阶段显式执行的升级命令。
2. **源码插件的当前生效版本与源码发现版本被混在一起**。现有扫描同步流程会直接把注册表版本更新为源码里的新版本，导致“数据库当前生效版本”和“源码树发现到的新版本”无法清晰区分；一旦开发者直接启动宿主，就会出现“代码已变新、治理状态未升级完成”的不一致状态。

动态插件已经具备基于 staged artifact / active release 的运行时升级基础，因此本轮不再把插件升级泛化为统一的运行时升级平台，也**不引入回滚机制**。本轮只收敛以下边界：

- **框架升级**：继续保留现有开发态升级方案；
- **源码插件升级**：新增与框架升级并列的开发态显式升级能力，并要求在宿主启动前完成；
- **动态插件升级**：继续保留现有运行时 upload + install/reconcile 升级语义，但补齐边界文档，明确不纳入 `make upgrade`。

## What Changes

- 扩展现有 `make upgrade` 开发态入口，增加升级范围参数，支持：
  - `scope=framework`：升级框架；
  - `scope=source-plugin`：升级指定源码插件或全部源码插件。
- 现有仓库根目录开发态升级工具继续作为统一入口使用，但需要扩展为同时支持框架升级与源码插件升级的计划/执行能力。
- 源码插件的治理模型调整为：
  - `sys_plugin.version` / `release_id` 只表示**当前生效版本**；
  - 源码扫描发现的更高版本只写入对应的 `sys_plugin_release`，作为**已发现但未应用的 release**；
  - 未执行升级前，不得直接覆盖当前生效版本。
- 宿主启动时必须校验是否存在“已安装源码插件发现了更高版本但尚未升级”的情况；若存在，启动 MUST 失败并提示先执行 `make upgrade`。
- 新增源码插件升级执行流程：版本比较、升级计划输出、`phase=upgrade` 迁移记账、菜单/权限/资源引用同步，以及 release / registry 状态切换。
- 动态插件继续通过运行时 upload + install/reconcile 完成升级；本轮会同步补正文档和接口说明，明确其与 `make upgrade` 的边界。
- 本轮**不实现插件回滚**，升级失败仅保留失败状态、日志和人工修复入口。

## Capabilities

### Modified Capabilities
- `source-upgrade-governance`: 从“仅框架源码升级”扩展为“统一的开发态升级入口治理”，同时覆盖框架升级和源码插件升级。

### New Capabilities
- `plugin-upgrade-governance`: 提供源码插件的版本发现/生效版本拆分、开发态显式升级命令，以及宿主启动前的升级前置校验能力。

## Impact

- 仓库根目录 `make upgrade` 及其开发态升级工具需要从单一框架升级扩展为多范围升级。
- 插件注册表与 release 元数据同步逻辑需要调整，避免源码扫描直接覆盖当前生效版本。
- 宿主启动阶段需要新增源码插件升级前置校验，防止“新源码 + 旧治理状态”直接启动。
- 源码插件升级将复用现有 `sys_plugin_release`、`sys_plugin_migration` 与资源引用治理表，不额外引入插件升级专用账本。
- 动态插件运行时升级的边界需要在 OpenSpec 文档、命令帮助和后续 README/接口说明中同步澄清。
- 业务系统运行时升级仍然只保留为后续方向，本轮不进入实现范围。
