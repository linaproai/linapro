## MODIFIED Requirements

### Requirement:运行时错误不得用 panic 替代显式错误处理
生产后端代码 SHALL 仅在启动、初始化、不可恢复的关键路径、`Must*` 语义构造器或未知 panic 重抛场景使用 `panic`。普通请求、导入/导出流程、动态插件输入、运行时配置读取和可恢复的资源处理路径必须使用显式 `error` 返回、统一错误响应或受控降级。

#### Scenario:启动不可恢复错误使用快速失败
- **当** 后端在进程启动、驱动注册、命令树初始化或源码插件静态注册期间检测到不可恢复错误时
- **则** 代码可使用 `panic` 快速失败进程
- **且** panic 调用点必须在白名单中并有保留原因，或对官方插件工作区命中约定的注册 fail-fast 模式自动归类

#### Scenario:普通业务请求返回错误
- **当** 普通 HTTP 请求、文件导入/导出、Excel 生成或资源关闭操作遇到可恢复错误时
- **则** 服务或控制器必须返回 `error`，使统一错误处理链能生成响应
- **且** 不得使用 `panic` 代替返回错误

#### Scenario:动态插件输入验证失败
- **当** 动态插件产物、清单、hostServices 声明或授权输入无效时
- **则** 宿主必须返回带上下文的验证错误
- **且** 插件提供的动态输入不得触发生产代码 panic

#### Scenario:无效运行时配置值显式返回
- **当** 读取快照时受保护的运行时配置值有解析错误
- **则** 后端必须通过显式 `error` 返回或统一错误响应暴露配置问题
- **且** 写路径必须保持严格验证，使正常管理入口无法保存无效值

#### Scenario:新 panic 受静态检查约束
- **当** 开发者在生产后端 Go 代码中添加 `panic` 调用时
- **则** 自动检查必须要求调用点匹配白名单或约定的自动放行模式
- **且** 宿主代码的白名单条目必须记录其分类和保留原因

#### Scenario:官方插件 init 注册 fail-fast 按模式自动放行
- **当** 官方插件工作区已初始化
- **且** `apps/lina-plugins/*/backend/plugin.go` 的包级 `init` 中，panic 均为对 error-returning 注册 API 错误结果的 fail-fast
- **则** 生产 panic 治理测试自动归入 `plugin-registration` 类别并通过
- **且** 宿主测试不得为该模式维护按插件 ID 枚举的 Path/Count 清单

#### Scenario:非常规插件 panic 被拒绝
- **当** 官方插件生产代码出现字面量 panic、非 `init` 生产 panic，或非注册 fail-fast 模式的 panic
- **则** 生产 panic 治理测试失败
- **且** 不得因该 panic 位于插件目录而默认放行

#### Scenario:宿主 panic 仍精确白名单
- **当** `apps/lina-core` 生产代码中的 panic 调用点变化（新增、删除或同一函数内次数变化）
- **则** 精确白名单必须同步更新并通过计数校验
- **且** 不得依赖插件模式自动放行宿主代码中的 panic
