## Context

插件管理页已经统一通过安装审查弹窗承载源码插件安装确认和动态插件宿主服务授权审查，但当前仍要求管理员在安装成功后回到列表，再通过状态开关完成启用。现有后端已经把安装与启用拆分为稳定的生命周期动作，动态插件在安装阶段形成授权快照后，后续启用也支持直接复用，不需要再重复弹出授权窗口，因此这次更适合做成前端层面的复合快捷动作，而不是再引入新的生命周期或存储模型。

这项改动会同时影响插件管理列表、安装弹窗、权限可见性和 E2E 覆盖，但不涉及数据库结构、插件 manifest 结构或后端治理数据表变更。

## Goals / Non-Goals

**Goals:**
- 在安装弹窗中提供“安装并启用”快捷动作，减少管理员往返列表的操作成本。
- 保留现有“仅安装”路径，避免改变已有治理语义。
- 复用既有安装与启用接口，保证源码插件与动态插件继续遵循同一套生命周期顺序。
- 明确组合动作的权限边界与部分成功提示，避免出现误操作或状态误判。
- 补充插件管理相关自动化测试，覆盖动态插件授权审查、源码插件快捷启用和权限可见性场景。

**Non-Goals:**
- 不新增数据库表、SQL 脚本或新的插件状态字段。
- 不引入新的后端复合接口或新的插件生命周期状态。
- 不把“安装后默认启用”改成系统级默认行为；管理员仍然显式选择安装或安装并启用。
- 不追求安装与启用两步之间的强原子事务；若启用失败，不自动回滚成功的安装结果。

## Decisions

### 1. 采用前端复合动作，顺序调用既有安装与启用接口

安装弹窗新增“安装并启用”按钮后，前端在一次用户操作内按顺序执行：先调用安装接口，安装成功后再调用启用接口。这样可以直接复用现有后端安装、启用、权限校验、菜单刷新和插件运行时收敛逻辑，不需要额外增加后端 DTO、控制器和服务编排。

不选择新增后端 `install-and-enable` 复合接口，原因是当前链路已经足够稳定，新增复合接口只会扩大接口面和回归范围，但对用户体验收益有限。

### 2. 组合动作继续复用安装审查弹窗，不新增列表级独立按钮

快捷动作仍放在现有安装弹窗底部，而不是在列表操作列上再增加一个“安装并启用”按钮。这样可以保证源码插件详情确认和动态插件授权审查仍在同一个上下文里完成，避免跳过必要的插件信息核对流程。

弹窗需要同时支持：
- 仅安装
- 安装并启用

其中“安装并启用”只在当前用户同时具备 `plugin:install` 与 `plugin:enable` 权限时显示；若用户只有安装权限，则仍可正常执行“仅安装”。

### 3. 动态插件在组合动作里复用安装阶段形成的授权快照

动态插件在安装审查弹窗中提交的授权结果仍只在安装请求中确认一次。若管理员选择“安装并启用”，后续启用步骤直接复用安装阶段已经持久化的授权快照，不再额外弹出第二次授权窗口，也不要求管理员重复确认。

这样可以保持当前动态插件授权治理模型不变：授权快照在 install 阶段形成，enable 阶段只消费最终快照；历史上未确认的 release 仍然沿用原有启用补确认逻辑。

### 4. 组合动作采用“真实状态优先”的部分成功反馈语义

若安装成功但启用失败，系统不自动回滚安装结果。前端必须刷新插件列表，并明确提示当前插件已处于“已安装、未启用”状态，管理员可以稍后重试启用。

这样做的原因是安装与启用本来就是两个独立治理动作；启用失败并不代表安装结果无效，强行自动卸载或回滚会放大副作用，也会破坏现有生命周期事件与治理审计的一致性。

### 5. 用 E2E 覆盖组合动作、权限边界与失败提示

本次改动的主要风险在 UI 编排和权限边界，因此验证重点放在端到端：
- 动态插件：安装审查弹窗直接“安装并启用”，并验证不重复弹授权审查。
- 源码插件：安装后直接进入启用状态。
- 权限边界：仅具备安装权限时看不到“安装并启用”；同时具备安装和启用权限时才能使用该按钮。
- 失败反馈：模拟第二步启用失败时，界面仍能正确显示“已安装、未启用”。

## Risks / Trade-offs

- [部分成功会增加提示分支] -> 通过统一的“安装成功，但启用失败”提示文案和列表刷新，确保管理员看到真实状态而不是模糊失败提示。
- [前端权限判断容易误用 any-of 语义] -> 为组合动作单独实现“同时拥有安装与启用权限”的判断，避免直接复用现有单权限 helper。
- [弹窗 footer 动作增加后可能造成主次不清] -> 明确按钮文案与层级，让“安装并启用”承担快捷入口，“仅安装”保留为保守路径。
- [动态插件授权流被误改为重复确认] -> 组合动作的启用步骤默认复用安装阶段快照，不重新打开授权审查弹窗，并补充 E2E 回归验证。

## Migration Plan

1. 先更新 OpenSpec 增量规范，明确安装弹窗支持复合快捷动作与权限边界。
2. 调整插件管理安装弹窗 footer，新增“安装并启用”动作并接入复合提交流程。
3. 补充插件页面对象与插件管理 E2E 用例，覆盖组合动作、权限可见性和动态插件授权复用。
4. 回归现有插件管理相关测试，确保原有“仅安装”“启用开关”“卸载”路径不受影响。
