# 设计说明

## 背景

主框架此前提供匿名`GET /api/v1/health`和`health.timeout`配置，把数据库探测、集群主从模式和服务可用性包装为框架级健康语义。实际业务交付中，健康检查通常需要覆盖业务数据库、队列、外部服务、模型服务、插件状态和业务降级策略，宿主通用健康接口无法表达这些差异。

## 决策

- 删除宿主内建匿名健康检查 API、DTO、控制器、路由绑定和 apidoc 翻译资源。
- 删除`health.timeout`配置段、`config.Service.GetHealth`和相关静态配置缓存。
- 不为旧`/api/v1/health`路径提供兼容或重定向，避免继续暗示宿主提供业务健康语义。
- 保留`GET /api/v1/system/info`作为受认证和权限控制的系统诊断入口，用于集群 coordination 等运维诊断。
- 业务应用、交付镜像或业务插件如需健康检查，应在自身边界定义健康语义和路由。

## 影响分析

| 领域 | 影响 |
|------|------|
| 架构边界 | 宿主不再承载业务健康语义，减少通用框架与具体交付场景的耦合。 |
| API 契约 | 移除匿名`GET /api/v1/health`，不新增替代匿名 API。 |
| 配置 | 移除`health.timeout`配置入口，配置服务不再暴露健康配置读取契约。 |
| 诊断 | 集群和 Redis coordination 诊断改用已认证`GET /api/v1/system/info`。 |
| `i18n` | 删除宿主健康 API 的 apidoc 翻译资源，不新增运行时 UI 文案。 |
| 数据权限 | 无新增数据读写或匿名数据可见性；系统信息诊断继续走认证和权限链路。 |
| 缓存一致性 | 删除能力不涉及缓存权威数据源、失效、TTL 或跨实例同步。 |

## 验证策略

- 使用 OpenSpec 严格校验验证规范和任务记录结构。
- 使用 Go 包测试覆盖配置服务、HTTP 路由绑定、权限审计、角色测试替身、集群诊断替代路径、apidoc 和 packed manifest。
- 使用 E2E 资产校验和 TypeScript 编译验证删除健康端点用例后的测试编号与类型正确性。
- 使用静态检索确认生产实现、交付配置和测试脚本中不再引用宿主健康 API、`GetHealth`、`HealthConfig`或`health.timeout`。
