## Context

当前项目已经具备一部分“零散的国际化前置条件”，但距离“框架级 i18n 能力”仍有明显差距：

1. 前端已有 `vue-i18n` 基础。`apps/lina-vben/packages/locales/` 与 `apps/lina-vben/apps/web-antd/src/locales/` 已经支持静态语言包加载，工作台请求也会自动携带 `Accept-Language`。
2. 后端尚未形成语言上下文。`apps/lina-core` 目前没有统一的 `LocaleResolver`、翻译服务或请求级语言解析链路，多数接口仍然直接返回库表中的单语文本。
3. 动态元数据仍是单语。菜单、字典、系统参数名称/备注、登录页公共配置、系统信息页元数据、插件 `plugin.yaml` 中的 `name/description/menu.name` 都是直接存储和直接返回。
4. 业务内容多语言模型缺失。当前若业务项目要支持标题、描述、正文等多语言内容，只能自行加字段或在业务侧重复造轮子。
5. 维护方式不统一。既没有统一的翻译键约定，也没有“文件基线 + 运行时覆写”的资源治理机制，导致交付项目很难低成本维护多语配置。

由于项目是全新项目，不需要考虑历史兼容包袱，因此本次设计直接建立统一国际化基础设施，不做历史接口兼容层，也不为旧格式保留长期双轨方案。

## Goals / Non-Goals

**Goals:**
- 提供宿主级国际化基础能力，使 LinaPro 交付项目能够以框架内建方式支持多国语言。
- 建立统一的语言解析链路，让前后端围绕同一语言上下文协作。
- 建立统一的翻译资源组织方式，覆盖宿主、交付项目和插件，并支持按语言聚合为运行时翻译包。
- 支持动态元数据国际化，包括菜单、字典、系统参数元数据、登录页公共配置、系统信息与插件元数据。
- 为业务模块提供通用的多语言内容存储模型，避免各业务自行扩展多套字段。
- 让配置和维护尽可能简便，通过稳定翻译键约定、文件基线、数据库覆写、导入导出和缺失校验降低交付成本。
- 保持前端静态文案与后端动态元数据的职责分层，避免所有翻译都挤到同一层处理。

**Non-Goals:**
- 不在本次变更中引入自动机器翻译、AI 自动补全文案或外部翻译平台对接。
- 不要求所有现有业务表立即改造成多语言内容表，只提供统一能力和接入方式，由业务按需接入。
- 不在首期为每个业务模块都建设完整的翻译管理页面；首期优先保证资源模型、接口、导入导出与运行时链路成立。
- 不把所有错误文案一次性全部改造完成；首期先建立翻译服务和新增文案约束，历史硬编码按模块逐步收敛。
- 不在本次范围内引入用户级持久化语言偏好存储；首期以 `lang` 参数、`Accept-Language` 和系统默认语言为主。

## Decisions

### 决策一：采用“三层国际化模型”，拆分静态界面文案、动态元数据和业务内容

**选择**：将国际化文本按来源和生命周期分为三层：
- **静态界面文案**：前端框架与页面本地静态文案，继续由 `vue-i18n` 本地 JSON 管理；
- **动态元数据**：菜单、字典、系统参数元数据、系统信息、插件描述等，由后端按语言投影；
- **业务内容**：公告标题、正文、描述等业务内容，采用通用多语言内容表按业务模块按需接入。

**原因**：
- 静态 UI 文案和动态元数据的来源不同，混在一起会让维护边界模糊。
- 菜单、字典、系统配置元数据本身属于后端治理数据，应该由后端按语言投影，而不是要求前端硬编码所有组合。
- 业务内容通常带有数据主键和发布生命周期，最适合抽象成通用多语言内容模型，而不是给每张业务表都加一组 `_en/_ja/...` 字段。

**备选方案**：
- 把所有翻译都放到前端静态 JSON。拒绝，因为动态菜单、字典、插件元数据无法稳定同步，且后端错误消息与外部 API 也无法覆盖。
- 给每张业务表增加多语言列。拒绝，因为会导致模型膨胀，且不适合作为框架通用能力。

### 决策二：采用“文件基线 + 数据库覆写”的资源治理模式

**选择**：国际化资源以文件为基线、以数据库为覆写：
- 宿主与交付项目维护基础语言包文件；
- 插件在各自目录声明翻译文件；
- 运行时允许通过数据库维护启用语言、覆写消息和业务内容；
- 翻译包聚合时按优先级合并文件资源与数据库资源。

**原因**：
- 文件最适合交付项目随代码版本管理，便于开发者批量维护与 code review。
- 数据库覆写适合线上补丁、运营改文案和租户级/项目级差异化配置。
- 两者组合既能保证交付便捷，又能避免所有修改都必须重新发版。

**资源优先级**：
1. 数据库覆写消息
2. 项目级文件资源
3. 插件文件资源
4. 宿主默认文件资源
5. 默认语言回退值

**备选方案**：
- 只用数据库存储所有翻译。未采用，因为开发交付体验差，且不利于插件随代码交付。
- 只用文件，不支持数据库覆写。未采用，因为上线后的配置与局部修订成本过高。

### 决策三：后端建立请求级语言解析链路，并将语言写入业务上下文

**选择**：在宿主新增 `LocaleResolver` 与请求级上下文注入机制，首期语言解析优先级为：
1. 查询参数 `lang`
2. 请求头 `Accept-Language`
3. 系统默认语言

解析出的 `locale` 在控制器、服务和插件宿主桥接层统一可取，用于菜单、字典、配置、系统信息、错误消息和运行时翻译包聚合。

**原因**：
- 前端当前已经发送 `Accept-Language`，宿主只需补上解析链路即可形成前后端协同闭环。
- `lang` 查询参数适合下载、公开页面、调试和站外嵌入场景，优先级应高于请求头。
- 统一写入业务上下文后，避免每个服务组件重复读取请求头。

**备选方案**：
- 完全依赖前端本地语言，不让后端感知。拒绝，因为动态元数据与错误消息都来自后端。
- 首期就把用户语言偏好写入 `sys_user`。未采用，后续可以扩展，但不是建立基础能力的前置条件。

### 决策四：定义稳定翻译键约定，尽量避免人工配置映射关系

**选择**：使用稳定业务键自动推导翻译键，减少额外配置。运行时 UI 资源文件允许使用层级 JSON 或扁平 dotted key 编写；宿主加载后统一归一化为“扁平 key”作为治理形态：
- 菜单标题：`menu.<menu_key>.title`
- 字典类型名：`dict.<dict_type>.name`
- 字典标签：`dict.<dict_type>.<value>.label`
- 配置名称/备注：`config.<config_key>.name`、`config.<config_key>.remark`
- 插件名称/描述：`plugin.<plugin_id>.name`、`plugin.<plugin_id>.description`
- 系统信息组件说明：`systemInfo.component.<section>.<name>.description`
- 公共前端配置文案：`publicFrontend.<group>.<field>`

**原因**：
- 项目已经在菜单、插件、配置等模型中具备稳定业务键或可推导键，适合用作翻译锚点。
- 约定式键生成比显式维护映射表更容易落地，也更适合插件生态和交付项目批量维护。
- 层级 JSON 更适合作为人工维护格式，可减少重复前缀并提升代码审查可读性。
- 扁平 key 更适合作为治理格式，便于数据库建模、缺失翻译检查、导入导出和 `GoFrame gi18n` 这类后端翻译组件复用。
- 运行时 API 仍可在输出阶段按需转换为前端友好的嵌套消息树，因此不会牺牲前端接入体验。
- API 文档翻译资源同样允许使用层级 JSON，并可按 `manifest/i18n/apidoc/<locale>/**/*.json` 拆分为多个文件；宿主加载时仍归一化为稳定结构化 key。
- 对标准响应包装字段、分页字段和通用时间字段等重复 OpenAPI 元数据，使用 `core.common.*` fallback key 维护公共翻译，具体结构 key 存在时仍优先使用具体翻译。

**备选方案**：
- 在所有业务表中增加单独的 `i18n_key` 字段。未采用，因为多数场景已有稳定锚点，额外字段会增加维护负担。

### 决策五：新增统一翻译资源模型，分别处理“消息翻译”和“业务内容翻译”

**选择**：新增三类核心表：
- `sys_i18n_locale`：语言注册表，维护语言编码、名称、是否启用、是否默认；
- `sys_i18n_message`：消息翻译表，维护 `key + locale + value + scope/source` 等信息，用于 UI/元数据翻译与数据库覆写；
- `sys_i18n_content`：业务内容多语言表，维护 `business_type + business_id + field + locale + content`，用于业务正文类内容。

**原因**：
- 消息翻译和业务内容翻译在生命周期、查找方式和缓存策略上不同，拆表更清晰。
- `sys_i18n_content` 能成为业务项目的统一扩展模型，避免重复为公告、文章、页面配置等各自设计多语言结构。

**备选方案**：
- 所有翻译共用一张大表。未采用，因为消息级键值与内容级锚点的查询模式差异很大。

### 决策六：后端负责动态元数据翻译，前端负责静态文案与运行时消息合并

**选择**：
- 后端负责菜单、字典、配置元数据、系统信息、插件清单、公共前端配置等动态数据的本地化投影；
- 前端保留现有 `vue-i18n` 体系，新增运行时加载器，将宿主运行时翻译包、插件翻译包与本地静态语言包合并后注入；
- 语言切换时，前端同时刷新运行时消息、公共前端配置、动态菜单/路由以及依赖动态元数据的页面。
- 翻译文件可使用层级 JSON 或扁平 dotted key 维护，数据库覆写、导入导出、缺失检查和来源诊断统一使用扁平 key；运行时接口在输出给前端时再转换为嵌套消息对象。

**原因**：
- 动态元数据掌握在后端，后端翻译可以让 Web、插件页、OpenAPI 消费端共用同一结果。
- 前端已经有 `vue-i18n`，继续复用最稳妥，只需扩展加载链路而不是重建体系。

**前后端分工**：
- **后端**：解析语言、聚合翻译资源、提供运行时翻译包 API、输出本地化动态数据、提供导入导出/缺失检查。
- **前端**：加载与缓存翻译包、切换语言、刷新页面依赖、保证页面组件统一使用 `$t` 与运行时本地化数据。

### 决策七：为插件定义标准国际化资源目录，并纳入生命周期管理

**选择**：插件新增标准资源目录 `manifest/i18n/<locale>.json`。宿主在以下时机处理插件国际化资源：
- 源码插件同步发现时注册资源；
- 动态插件安装/升级时写入对应 release 的资源快照；
- 插件启用时纳入运行时翻译聚合；
- 插件停用/卸载时从运行时聚合结果中移除。

**原因**：
- 插件国际化资源应该跟随插件版本一起交付，而不是散落在宿主项目中手工维护。
- 宿主统一管理资源生命周期，才能保证插件启停与语言包可见性一致。

**备选方案**：
- 在 `plugin.yaml` 中直接嵌长文本多语言结构。拒绝，因为 manifest 会快速膨胀且不利于维护。

### 决策八：提供运行时翻译包接口，而不是让前端逐个模块自行拼装

**选择**：宿主提供统一的运行时翻译包接口，按当前语言或指定语言返回聚合结果。前端启动和切换语言时只请求一个聚合包，再按需要叠加应用本地静态语言包。聚合链路内部将文件资源和数据库覆写统一归一化为扁平 key，接口输出阶段再转换为嵌套对象。

**原因**：
- 一个接口即可覆盖宿主、交付项目、插件和数据库覆写结果，避免前端分别请求菜单、配置、插件资源后再手工合并。
- 更容易加缓存、版本戳和缺失键诊断信息。

**备选方案**：
- 为每个模块单独提供翻译接口。未采用，因为会增加前端协同复杂度。

### 决策九：维护策略优先“约定式配置”，并提供导入导出与缺失检查能力

**选择**：
- 首期通过稳定键约定、文件目录约定和模板化 JSON 降低新增语言成本；
- 宿主提供语言启停、消息导入导出、缺失键检查与回退报告接口；
- 管理台可在后续迭代补全可视化管理页，但首期接口与模型必须先到位。

**原因**：
- 用户的核心诉求是交付项目能便捷配置国际化；在这件事上，“约定优于配置”比先做重型 CMS 更有效。
- 导入导出和缺失检查是交付阶段最常见的维护需求，优先级高于复杂编辑器。

## Risks / Trade-offs

- [Risk] 国际化范围跨越宿主、前端、插件和业务模型，首期实现面较大。→ Mitigation：按“基础设施 -> 动态元数据 -> 插件接入 -> 业务内容”的顺序分阶段落地，并在任务中拆分明确边界。
- [Risk] 动态元数据与运行时翻译包都可能缓存，若语言切换或插件启停后缓存失效不及时，会出现文案不一致。→ Mitigation：为运行时翻译包、公共前端配置和菜单数据统一建立版本戳/失效机制，并在前端语言切换时主动刷新。
- [Risk] 若缺少稳定翻译键约定，交付项目容易在多处重复定义同义文案。→ Mitigation：通过规范明确键推导规则，并提供缺失/重复检查工具。
- [Risk] 历史硬编码错误消息无法在短期内一次性收敛。→ Mitigation：首期新增翻译服务并要求新增用户可见文案必须走翻译键，历史模块逐步整改。
- [Risk] 插件资源与宿主资源来源不同，若聚合优先级不清晰，可能出现覆盖混乱。→ Mitigation：明确资源优先级并在导出/诊断结果中展示实际命中来源。

## Migration Plan

1. 在宿主新增国际化基础设施，包括语言解析、翻译服务、资源聚合与数据库模型。
2. 为宿主提供运行时翻译包接口、语言列表接口、导入导出接口与缺失检查接口。
3. 改造菜单、字典、配置、登录页公共配置、系统信息和插件清单的动态元数据投影逻辑，使其按语言返回本地化值。
4. 扩展默认管理工作台的 `vue-i18n` 加载链路，接入运行时翻译包，并在切换语言时刷新公共配置、菜单和插件页面。
5. 为插件增加 `manifest/i18n/` 资源组织规则，并纳入安装、升级、启停、卸载的宿主管理流程。
6. 为业务模块提供 `sys_i18n_content` 接入规范，后续业务项目按需接入多语言内容字段。
7. 增加导入导出、缺失检查和回归测试，确保交付项目可以持续维护多语言资源。

## Open Questions

- 首期是否需要直接提供完整的国际化管理后台页面，还是先交付 API + 导入导出能力即可？当前设计优先后者，以保证基础能力先稳定。
- 业务内容多语言是否需要支持富文本、Markdown 和附件关联的统一结构？首期先以文本/长文本为主，复杂富文本可在业务接入时追加约束。
- 后续是否需要把用户语言偏好沉淀到用户资料并参与解析优先级？本次先不纳入强制范围，待用户中心能力成熟后再扩展。
