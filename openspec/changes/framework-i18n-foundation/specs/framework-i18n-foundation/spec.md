## ADDED Requirements

### Requirement: 宿主必须提供统一的语言解析与回退能力
宿主系统 SHALL 在每个请求中解析当前语言，并将解析结果写入统一业务上下文，供控制器、服务、插件桥接层和运行时翻译聚合链路复用。语言解析优先级 MUST 为 `lang` 查询参数、`Accept-Language` 请求头、系统默认语言；当请求语言不可用时，系统 MUST 回退到默认语言。

#### Scenario: 通过查询参数覆盖请求语言
- **WHEN** 客户端请求任意支持国际化的接口并显式传入 `lang=en-US`
- **THEN** 宿主使用 `en-US` 作为本次请求的有效语言
- **AND** 不再继续使用同一请求中的 `Accept-Language` 结果

#### Scenario: 请求语言未启用时回退默认语言
- **WHEN** 客户端请求了未启用或不受支持的语言
- **THEN** 宿主回退到系统默认语言
- **AND** 本次请求中的动态元数据和运行时翻译包都使用默认语言返回

### Requirement: 宿主必须提供统一的翻译资源聚合与覆盖优先级
宿主系统 SHALL 支持从宿主默认资源、项目级资源、插件资源和数据库覆写中聚合翻译消息，并按统一优先级生成运行时可用结果。运行时 UI 文件资源 MAY 使用层级 JSON 或扁平 dotted key 编写；宿主加载后 MUST 统一归一化为扁平 key 管理。数据库覆写、导入导出、缺失检查和来源诊断 MUST 使用扁平 key；数据库覆写 MUST 高于文件资源；当指定语言缺失翻译时，系统 MUST 回退到默认语言或默认值。

#### Scenario: 数据库覆写覆盖文件基线
- **WHEN** 某个翻译键在项目文件中已有基线值，同时数据库中存在相同 `key + locale` 的覆写值
- **THEN** 运行时聚合结果返回数据库中的覆写值
- **AND** 导出或诊断结果能够标识该值来自数据库覆写

#### Scenario: 语言缺失时回退默认语言值
- **WHEN** 当前请求语言的聚合资源中缺少某个翻译键
- **THEN** 系统回退使用默认语言中的同名翻译键
- **AND** 若默认语言也缺失，则回退到约定的默认值或原始基线文本

### Requirement: 宿主必须提供运行时翻译包分发能力
宿主系统 SHALL 提供运行时翻译包接口与语言列表接口，按语言返回聚合后的消息资源以及当前可用的语言描述信息，供默认管理工作台和宿主嵌入式插件页面加载。运行时翻译包 MUST 能同时包含宿主、项目和当前已启用插件的国际化消息，并在输出时转换为前端可直接消费的嵌套消息对象。

#### Scenario: 默认工作台加载运行时翻译包
- **WHEN** 前端以 `zh-CN` 请求运行时翻译包
- **THEN** 宿主返回该语言下的聚合消息集合
- **AND** 结果中包含宿主资源、项目级资源以及已启用插件资源的合并结果

#### Scenario: 前端获取宿主支持的语言列表
- **WHEN** 前端请求运行时语言列表
- **THEN** 宿主返回当前支持的语言编码、默认语言标记、展示名称与原生名称
- **AND** 展示名称按当前请求语言返回，原生名称保持为对应语言自身文案

#### Scenario: 运行时语言包支持层级维护并保持扁平治理
- **WHEN** 宿主从文件、插件或数据库加载翻译资源
- **THEN** 宿主允许运行时 UI 文件资源使用层级 JSON 或扁平 dotted key 编写
- **AND** 宿主内部统一以扁平 key 形式聚合和覆写消息
- **AND** 运行时接口返回给前端的结果为嵌套对象结构，以便直接并入前端 `vue-i18n` 消息树

#### Scenario: 插件停用后不再暴露其翻译资源
- **WHEN** 某个插件被停用或卸载
- **THEN** 后续运行时翻译包结果中不再包含该插件贡献的翻译消息
- **AND** 其他宿主与已启用插件资源保持可用

### Requirement: 宿主必须为动态元数据定义稳定翻译键约定
宿主系统 SHALL 使用稳定业务键推导动态元数据翻译键，避免菜单、字典、配置、插件和系统信息等模块手工维护多套映射关系。菜单、字典、配置、插件和系统信息的翻译键规则 MUST 在框架规范中统一定义，并在宿主和插件实现中保持一致。

#### Scenario: 菜单按稳定业务键推导翻译键
- **WHEN** 宿主需要为某个菜单返回本地化标题
- **THEN** 宿主基于该菜单的稳定业务键推导翻译键
- **AND** 同一菜单在菜单树、路由标题和下拉树中的本地化结果保持一致

#### Scenario: 插件元数据按插件标识推导翻译键
- **WHEN** 宿主需要返回插件名称或插件描述的本地化结果
- **THEN** 宿主基于 `plugin_id` 推导插件翻译键
- **AND** 不要求插件在 `plugin.yaml` 中重复声明额外的翻译键字段

### Requirement: 宿主必须提供通用的业务内容多语言存储模型
宿主系统 SHALL 提供独立于具体业务表的多语言内容存储模型，用于管理标题、摘要、描述、正文等业务内容的多语言值。业务内容多语言模型 MUST 至少支持 `business_type`、`business_id`、`field`、`locale` 和 `content` 维度，并支持按默认语言回退。对于部门、岗位、角色、字典、参数、通知、调度等可在管理后台编辑的业务主数据，管理页列表、详情、编辑回填和选择器默认 MUST 保持数据库原值；除非该条记录的具体字段已经显式接入业务内容多语言存储，否则系统不得仅依据稳定键、种子映射或前端启发式规则改写其展示值。菜单治理能力属于宿主导航元数据，菜单树列表、父级菜单展示、角色菜单树与相关只读选择器 SHALL 继续基于稳定 `menu_key` 返回本地化标题，但菜单详情表单中的可编辑 `name` 字段与编辑回填仍 MUST 保持数据库原值。唯一例外是框架内置且在当前治理页中受保护、不可编辑删除的交付型治理记录；此类记录 MAY 在只读列表展示位基于稳定业务锚点提供名称本地化投影，但详情、编辑回填和选择器仍 MUST 保持数据库原值。

#### Scenario: 业务模块读取多语言标题
- **WHEN** 某业务模块按当前语言读取某条业务记录的标题字段
- **THEN** 系统优先返回 `business_type + business_id + field + locale` 命中的内容值
- **AND** 若当前语言缺失，则回退到默认语言内容

#### Scenario: 业务模块未接入多语言内容时保持默认值
- **WHEN** 某个业务模块尚未为某字段写入多语言内容记录
- **THEN** 系统仍可返回该模块默认语言或原始字段值
- **AND** 不会因缺少多语言记录导致读取失败

#### Scenario: 管理页中的可编辑主数据未接入内容翻译时保持原值
- **WHEN** 管理员在英文环境中查看部门、岗位、角色、字典、参数、通知或调度等管理页中的可编辑业务记录
- **THEN** 列表、详情、表单回填和树形/下拉选择器显示数据库中的原始名称、备注、标题或正文
- **AND** 系统不会仅因存在 `dict_type`、`config_key` 等稳定键就自动把这些记录投影为英文

#### Scenario: 菜单治理页在英文环境中显示本地化标题但保持可编辑字段原值
- **WHEN** 管理员在 `en-US` 环境中查看菜单管理列表、父级菜单展示或角色菜单树
- **THEN** 菜单标题按稳定 `menu_key` 返回英文等本地化结果
- **AND** 菜单详情表单中的可编辑 `name` 字段与编辑回填继续显示数据库原值

#### Scenario: 英文环境下内置受保护的超级管理员角色允许在列表中显示英文
- **WHEN** 管理员在 `en-US` 环境中打开角色管理页，并查看框架内置且受保护的超级管理员角色
- **THEN** 角色列表中的该条记录名称可基于稳定 `role.key` 显示英文投影
- **AND** 同页中其他仍可编辑的角色记录继续显示数据库原值
- **AND** 超级管理员角色的详情、编辑回填和任何选择器语义仍保持数据库原值

### Requirement: 默认管理工作台必须完整本地化框架交付页面与内建展示内容
宿主系统 SHALL 确保默认管理工作台在已启用语言下，对随框架交付的宿主页面、宿主共享组件、内建插件页面、菜单/页签/路由标题，以及框架自带的 `seed` / `mock` / `demo` 展示内容提供完整本地化结果。该要求覆盖框架默认交付的治理元数据、示例文案和演示数据，但不应覆盖管理后台中仍以数据库字段原值为准的可编辑业务主数据。若某类展示内容尚未接入通用消息翻译，则系统 MUST 通过稳定业务锚点、插件资源或默认内容翻译记录接入现有国际化基础设施。

#### Scenario: 英文巡检默认工作台页面时不再出现中文系统文案
- **WHEN** 管理员将语言切换为 `en-US` 并逐个打开框架默认交付的宿主页面与内建插件页面
- **THEN** 菜单名称、页签标题、面包屑、搜索项标题、表头、列表标题、按钮、弹窗标题和空状态等系统展示文案均为英文
- **AND** 不再出现残留中文的宿主共享组件或路由标题

#### Scenario: 英文环境下默认示例与种子展示内容同步本地化
- **WHEN** 管理员在 `en-US` 环境中查看框架默认自带的仪表盘示例、定时任务默认数据、通知公告演示内容、插件示例页面和其他交付型种子数据
- **THEN** 这些由框架交付的默认展示内容返回英文或对应语言下的本地化投影
- **AND** 不需要用户手动修改框架内置 SQL、插件清单或前端源码才能消除默认中文展示
- **AND** 同一页面中若存在管理员可直接编辑的业务记录字段，则这些字段仍以数据库原值展示，除非该字段已显式维护对应的多语言内容

#### Scenario: 英文环境下长文案仍保持稳定可用的界面布局
- **WHEN** 管理员在 `1366px` 及以上桌面视口下切换到 `en-US`，并打开框架默认交付的列表页、搜索区、表单页、抽屉与弹窗
- **THEN** 搜索项标题、表头、表单标签、按钮和页签标题不会因英文文案更长而出现难以阅读的断裂、关键字段被遮挡或主要信息默认移出可见区域
- **AND** 对于受限区域，系统通过调整布局、增加标签/列宽、使用更短的默认英文文案或补充 tooltip 等方式保证可读性和可操作性

### Requirement: 系统接口文档必须使用英文源文案与独立 apidoc 翻译资源
宿主系统 SHALL 将宿主、源码插件和动态插件 API DTO 中的 OpenAPI 文档源文本统一维护为可读英文，包括路由分组、接口摘要、接口描述、请求参数、响应参数说明，以及能够进入接口文档展示的固定路由投影文案。接口文档本地化 SHALL 在渲染 `/api.json` 时基于当前请求语言执行，并使用独立的 `manifest/i18n/apidoc/<locale>.json` 与可选 `manifest/i18n/apidoc/<locale>/**/*.json` 资源；这些资源 MUST 与默认管理工作台运行时 UI 的 `manifest/i18n/<locale>.json` 资源解耦。宿主接口翻译 SHALL 由 `apps/lina-core/manifest/i18n/apidoc/` 维护；插件接口翻译 SHALL 由插件自己的 `apps/lina-plugins/<plugin-id>/manifest/i18n/apidoc/` 独立维护，lina-core 的 apidoc 模块 SHALL 在渲染时发现源码插件内嵌资源、动态插件运行时产物中的 apidoc 资源，并统一合并输出，禁止将插件 `plugins.*` 翻译键集中耦合到 lina-core apidoc 资源。非英文 apidoc 文件 MAY 使用层级 JSON 或扁平 dotted key 编写，宿主加载后 MUST 归一化为稳定结构化 key；公共响应包装、分页和时间字段等重复元数据 MAY 通过宿主 `core.common.*` fallback 键维护，具体结构键存在时 MUST 优先使用具体结构键。英文接口文档 SHALL 直接使用英文源文案，各级 `manifest/i18n/apidoc/en-US.json` SHALL 保持为空对象占位，避免重复维护英文映射。若 GoFrame、生成的 `entity` schema 或数据表注释元数据进入接口文档，系统 SHALL 按其数据源原文展示，禁止在 apidoc service 层维护中文到英文的临时转换表，也不得通过恢复 `en-US.json` 或维护非英文 apidoc JSON 映射兜底；需要改变其展示语言时，应修改对应数据源或生成源。系统不得在手写 API DTO 文档标签中写入中文源文本，也不得写入不透明 `i18n` key 占位符作为接口文档源文本。`eg/example` 示例值 SHALL 保持接口真实示例语义，不写入、不翻译 apidoc i18n 资源。

#### Scenario: 英文接口文档直接使用英文源文案
- **WHEN** 管理员在英文环境中打开系统接口文档或请求 `/api.json?lang=en-US`
- **THEN** 宿主、源码插件与动态插件的手写 API DTO 路由分组、接口摘要、接口描述、请求参数和响应参数说明均展示为英文
- **AND** 英文接口文档不依赖 `en-US` apidoc 翻译映射，`en-US.json` 仅作为空对象占位文件存在
- **AND** 生成的 `entity` schema、数据表注释与框架通用响应元数据不经过 apidoc service 层临时翻译，按其数据源原文展示
- **AND** 非英文 apidoc JSON 中不得维护 `internal.model.entity.*` 等生成 schema 翻译键

#### Scenario: 中文接口文档通过 apidoc JSON 投影
- **WHEN** 管理员在中文环境中打开系统接口文档或请求 `/api.json?lang=zh-CN`
- **THEN** API DTO 中维护的英文源文案通过 `manifest/i18n/apidoc/zh-CN.json` 及 `manifest/i18n/apidoc/zh-CN/**/*.json` 的稳定结构化 key 映射为中文
- **AND** 接口文档翻译资源不进入运行时 UI 语言包，也不依赖默认管理工作台前端静态语言包
- **AND** 插件接口的 `plugins.*` 翻译键由插件自己的 apidoc i18n JSON 提供，并由 lina-core apidoc 模块统一合并后应用到接口文档
- **AND** 标准响应字段、分页字段和通用时间字段可通过 `core.common.*` fallback 翻译，避免在每个接口结构下重复维护同一文案
- **AND** `eg/example` 示例值继续展示为 DTO 或 OpenAPI 投影中的原始示例值

#### Scenario: API 文案变更时校验 apidoc 翻译覆盖
- **WHEN** 开发者新增或修改宿主、源码插件或动态插件 API DTO 的 OpenAPI 文档标签
- **THEN** 开发者必须同步更新所属宿主或插件非英文 `manifest/i18n/apidoc` 翻译资源中的稳定结构化 key，禁止使用英文源文案作为 JSON key
- **AND** 自动化测试或审查规则必须阻断遗漏非英文翻译、中文源文案和不透明占位符进入接口文档，但不要求为 `en-US` 空占位文件或 `eg/example` 示例值提供翻译

### Requirement: 宿主必须提供国际化维护与校验能力
宿主系统 SHALL 提供语言启停、翻译消息导入导出、缺失翻译检查和资源来源诊断能力，以支持交付项目长期维护多语言资源。交付项目和插件新增国际化资源时 MUST 遵循统一目录约定和翻译键规范。

#### Scenario: 导出某语言的翻译资源
- **WHEN** 管理员或交付工具请求导出 `en-US` 的翻译资源
- **THEN** 系统返回该语言的聚合消息结果或可导入文件
- **AND** 导出内容能够用于后续批量维护和重新导入

#### Scenario: 检查缺失翻译键
- **WHEN** 管理员或交付工具执行缺失翻译检查
- **THEN** 系统返回当前语言相对默认语言缺失的翻译键列表
- **AND** 结果中能够定位缺失键所属的宿主模块、项目资源或插件资源范围
