## Context

### 现状

- 当前 `hack/tests/e2e/` 已接近 100 个 `TC*.ts` 文件,但目录结构仍大量沿用旧工作台分组方式,其中 `system/` 承载了多数宿主能力用例,与当前工作台稳定菜单边界已经不一致。
- 当前前端静态路由已经按稳定能力分组为 `IAM / 系统设置 / 任务调度 / 扩展中心 / 关于 / 仪表盘`,并且组织、内容、监控等能力已经大量插件化,测试目录若继续围绕旧 `system/monitor/plugin` 组织,后续定位和维护成本会持续升高。
- `hack/tests/playwright.config.ts` 当前仍是 `workers: 1`、`fullyParallel: false`; `hack/tests/fixtures/auth.ts` 中 `adminPage` 夹具会为每个测试重新走一遍 UI 登录。
- 现有页面对象和部分测试文件中存在大量 `waitForTimeout(...)`,固定等待既拉长整轮耗时,也会掩盖真实状态信号。仓库里目前已有数百处固定等待,高频集中在菜单、角色、字典、用户、配置等页面对象上。
- 当前 `hack/tests/e2e/` 中还混入了不符合用例规范的文件,例如调试脚本和共享 helper;同时已经出现重复 TC ID,说明目录与编号治理需要系统化约束,不能继续依赖人工维护。

### 约束

- 需要继续遵守 `openspec-e2e` 约定: `TC{NNNN}-{brief-name}.ts`、全局唯一 TC ID、每个文件可独立运行。
- `make test` / `pnpm test` 仍需保留“跑完整轮 E2E”的能力,不能因为提速改造丢失全量回归入口。
- 测试目录调整不能破坏现有高价值用例覆盖,尤其是插件生命周期、权限治理、任务调度、系统配置等高风险场景。
- 本项目是全新项目,不需要为旧测试目录做长期兼容;目录、脚本和治理规则可以一次性收敛到目标形态。

## Goals / Non-Goals

**Goals:**

1. 让 E2E 目录结构与当前工作台稳定能力边界和插件归属保持一致,避免继续使用过载的历史目录。
2. 让开发者可以按 `smoke / module / full` 三种层级获取反馈,不必每次都跑完整轮全量回归。
3. 复用认证态,消除高频夹具中的重复 UI 登录开销。
4. 用状态型等待替换高频固定等待,在不牺牲稳定性的前提下压缩整轮执行时间。
5. 在保证安全边界的前提下引入有限并行,让独立模块的用例可以同时执行。
6. 为目录归属、TC 唯一性、辅助文件位置和执行分层建立自动化治理,防止问题在后续迭代中反复出现。

**Non-Goals:**

1. 不更换 Playwright,也不引入第二套测试框架。
2. 不在本次变更中重写全部页面对象;优先治理最耗时、最常用的公共路径。
3. 不减少现有业务覆盖范围,也不以删除高价值用例作为提速手段。
4. 不把后端启动、数据库初始化或前端构建耗时优化纳入本迭代核心范围;本次聚焦 E2E 套件自身结构与运行效率。
5. 不为了并行而强行并行所有测试;存在共享全局状态的高风险用例可以继续保留串行池。

## Decisions

### D1. 测试目录按稳定能力边界重组,而不是继续按旧 URL 前缀归档

**决定**: 将 `hack/tests/e2e/` 收敛为围绕稳定能力边界的目录树,例如 `iam/`、`settings/`、`scheduler/`、`extension/`、`monitor/`、`org/`、`content/`、`dashboard/`、`about/`。允许在能力目录下继续按子域拆二级目录,例如 `scheduler/job/`、`monitor/operlog/`。

示意结构如下:

```text
hack/tests/
  e2e/
    auth/
    dashboard/
    about/
    iam/
      user/
      role/
      menu/
    settings/
      dict/
      config/
      file/
    org/
      dept/
      post/
      user-org/
    content/
      notice/
      message/
    monitor/
      operlog/
      loginlog/
      online/
      server/
    scheduler/
      job/
      job-group/
      job-log/
    extension/
      plugin/
```

**理由**: 测试目录的首要职责是帮助开发者快速定位“某个能力的回归面”。能力边界已经在前端路由、宿主菜单和插件目录层面稳定下来,测试目录继续依赖旧的 `system/monitor/plugin` 分组只会不断增加认知折损。

**备选方案**:
- 继续保留 `system/` 大目录,仅在内部追加注释或 README —— 被拒,因为目录本身仍然无法表达新的能力边界。
- 完全按插件/宿主物理目录组织测试 —— 被拒,因为用户视角与回归路径首先是业务能力,不是源码物理归属。

### D2. 把 helper、调试脚本和治理脚本移出 `e2e/`,建立专门支持目录

**决定**: `hack/tests/e2e/` 只保留真正的 `TC*.ts` 测试文件。共享 API helper、等待工具、数据构造器迁入 `hack/tests/support/` 或 `hack/tests/fixtures/`; 调试脚本迁入 `hack/tests/debug/` 或 `hack/tests/scripts/`。同时补充自动校验脚本扫描重复 TC ID、非法文件和错误目录。

**理由**: 当前 `e2e/` 中混入非测试文件,会削弱用例树的可读性,也会让后续扫描、统计、迁移和 review 更容易遗漏问题。把“可执行用例”和“测试支持资产”拆开,后续治理脚本和执行入口会更简单。

**备选方案**: 允许少量 helper 留在子目录旁边 —— 被拒,因为规则一旦放松,后续仍会回到同样的混乱状态。

### D3. 执行入口采用“全量默认保留 + 清单驱动的 smoke/module 快速入口”

**决定**:
- 保留 `pnpm test` 作为完整回归入口,与现有使用习惯和 `make test` 预期一致。
- 新增 `pnpm test:full` 作为显式全量入口。
- 新增 `pnpm test:smoke` 和 `pnpm test:module -- <scope>` 两类快速入口,由测试清单或范围映射文件驱动,而不是要求开发者手工拼接 globs。
- 在 `hack/tests` 下维护一份执行清单(如 `config/execution-manifest.json`),至少声明 `smoke` 文件列表、模块范围映射和串行池范围。

**理由**: 日常开发需要的是“尽快发现变更影响”,不是每次都等待完整回归结束。保留 `pnpm test` 的完整回归语义可以避免打断现有流程,再通过清单驱动引入更快的反馈入口。

**备选方案**:
- 把 `pnpm test` 改成 smoke —— 被拒,因为会悄悄改变现有习惯,风险高。
- 只靠 README 说明如何拼接目录参数 —— 被拒,因为可发现性差,也难以标准化落地。

### D4. 认证态通过预生成 `storageState` 复用,不再为每个测试重复 UI 登录

**决定**:
- 为 Playwright 增加一次性登录准备步骤,在单轮执行开始时生成管理员 `storageState` 文件。
- `adminPage` 夹具改为直接消费已认证页面上下文,仅在登录态缺失或失效时由准备步骤重新生成。
- 保留 `loginPage` 等显式登录场景用例的真实登录能力,但不再让普通后台用例默认重复执行登录流程。

**理由**: 绝大多数后台测试并不是在验证登录流程本身,而是在验证“登录后的能力页面”。重复 UI 登录是最显著的稳定性和耗时损耗之一,一次生成、多处复用是收益最高的提速手段。

**备选方案**:
- 改为 API 登录直接写 token/cookie —— 暂不采用,因为它更依赖登录接口细节,而 UI 生成 `storageState` 对前后端登录链路的耦合更小。

### D5. 固定等待收敛为状态型等待,并沉淀可复用等待工具

**决定**: 以页面对象和夹具为边界治理固定等待,抽取公共等待能力,例如:
- 表格 ready: 等待 loading 消失、目标行出现、首屏请求完成。
- 抽屉/弹窗 ready: 等待弹层可见、骨架屏消失、表单字段可交互。
- 提示反馈 ready: 等待 toast 出现并按需消失。
- 路由 ready: 等待目标 URL、生效标题、关键测试元素稳定渲染。

优先治理固定等待最密集、被最多测试复用的页面对象,例如菜单、角色、字典、用户、配置等页面。

**理由**: 固定等待是整轮耗时的线性税收,也是 flaky 的常见来源。把等待逻辑集中到页面对象和公共 helper 层,可以在较小改动面上释放广泛收益。

**备选方案**: 在现有测试里逐个替换 `waitForTimeout` —— 被拒,因为收益分散且难以持续维护。

### D6. 并行策略采用“文件内串行 + 文件间分池并行”,高风险场景保留串行池

**决定**:
- 保留 `fullyParallel: false`,继续保证单文件内部顺序执行。
- 将文件级执行从单 worker 提升到有限 worker 数,由配置和环境变量控制,并为高风险共享状态用例建立串行池。
- 插件生命周期、权限治理、运行参数、导入导出等共享全局状态明显的文件先纳入串行池;其余独立 CRUD / 只读列表 / 页面渲染用例进入并行池。
- `test:full` 通过运行器脚本按“并行池 + 串行池”两阶段执行,避免一刀切开大并发导致全套不稳定。

**理由**: 当前文件级隔离已经较强,直接维持 `workers: 1` 会人为压制吞吐;但全量并行也会把共享状态冲突一次性放大。分池执行是在稳定性和速度之间更现实的平衡点。

**备选方案**:
- 全量直接开多 worker —— 被拒,因为当前存在大量共享状态场景,高概率引入新 flake。
- 永远保持单 worker —— 被拒,因为无法满足提速目标。

### D7. 套件治理通过自动化盘点脚本固化,不再依赖人工 review 发现问题

**决定**: 为 `hack/tests` 增加自动盘点脚本,至少覆盖:
- TC ID 全局唯一性;
- `e2e/` 下是否混入非 `TC*.ts` 文件;
- 测试文件是否落在允许的能力目录下;
- 执行清单中的 smoke、serial、module 引用是否指向真实文件。

该脚本同时作为迁移期的验收工具和后续新增用例的守门器。

**理由**: 当前重复 TC ID 与非法文件混入已经证明,只靠人工约定无法长期保证套件整洁。自动化盘点成本低,但能显著提升治理效果。

## Risks / Trade-offs

- **[目录迁移会引发大量导入路径调整]** → 先完成目标目录映射和支持目录落点,再按模块批量迁移,每完成一批就执行对应模块回归。
- **[登录态复用可能掩盖真实登录问题]** → 保留 `auth/` 模块中的真实登录用例,只有非认证主题测试使用 `storageState`。
- **[并行池分类不当可能引入 flaky]** → 首轮采用保守分类,把明显共享状态场景放入串行池,并保留通过环境变量退回 `workers=1` 的兜底入口。
- **[固定等待替换不完整时收益有限]** → 优先治理最常用页面对象和等待最密集的公共路径,以“80% 频率先吃到收益”为策略推进。
- **[smoke 清单维护不当会偏离真实高风险路径]** → 以登录、菜单骨架、核心系统管理、插件治理等高价值路径为初始清单,后续随模块风险变化迭代调整。

## Migration Plan

1. 盘点现有 E2E 文件、辅助文件、重复 TC ID、固定等待热点和共享状态高风险文件,形成迁移映射表与执行清单初稿。
2. 创建目标目录与支持目录,迁移 helper/debug 文件,修复导入路径并清理非法文件放置。
3. 按能力模块迁移 `TC*.ts` 文件到新目录,同步修复重复 TC ID 和引用路径。
4. 新增执行清单、范围运行脚本和 `smoke/module/full` 入口,保留完整回归默认入口。
5. 引入 `storageState` 预登录与新 `adminPage` 夹具,验证 `auth` 之外的高频模块回归通过。
6. 对高频页面对象实施第一轮状态型等待治理,并在并行池/串行池划分后执行模块回归与全量回归。
7. 记录迁移前后全量运行耗时和稳定性对比,作为后续继续治理的基线。
