## Context

当前宿主已经具备两个与本次需求直接相关的基础能力：

1. 宿主主配置文件已经支持`plugin.autoEnable`，能够在启动阶段自动安装并启用指定源码插件。
2. 宿主已经通过`pluginhost.HTTPRegistrar`向源码插件发布了全局 HTTP 中间件注册能力，`monitor-operlog`等插件已经在使用这一接缝。

但现状仍有一个明显空白：缺少一个面向演示环境的统一写保护开关。若直接依赖管理员自觉避免点击写操作按钮，无法真正保障演示数据安全；若把能力硬编码在宿主中间件里，又会削弱源码插件架构的扩展边界。

另外，本次还需要处理一个关键约束：如果把演示控制开关做成运行时`sys_config`参数，那么当演示模式开启后，参数管理接口本身也会落入写操作拦截范围，容易出现“开得开不了、关得关不掉”的治理死锁。因此更适合把该开关定义为宿主主配置文件中的静态环境配置，并由自动启用的源码插件在请求链路中消费。

## Goals / Non-Goals

**Goals:**

- 提供一个默认关闭的演示控制配置项，用于显式声明当前宿主实例是否进入演示只读模式。
- 以官方源码插件`demo-control`承载演示能力，而不是把该逻辑散落在宿主核心中间件实现里。
- 在演示模式开启时，统一拦截系统 API 的非查询写操作，优先复用`RESTful API`的`HTTP Method`语义作为判断依据。
- 让`demo-control`在默认交付态即被宿主启动自动启用，避免演示环境仍需人工安装或启用插件。
- 保留最小必要白名单，确保演示环境仍可完成登录、登出等基础会话动作。

**Non-Goals:**

- 不在本次变更中增加演示模式专属前端页面、提示条、横幅或工作台视觉标识。
- 不在本次变更中把演示控制扩展为细粒度资源级白名单、角色级例外或按模块选择性放行。
- 不在本次变更中为动态插件设计独立的中间件声明模型；动态插件只通过既有`/api/v1`统一链路被演示控制插件覆盖。
- 不在本次变更中引入数据库表、安装 SQL 或新的运行时参数表结构。

## Decisions

### 决策一：演示控制开关放在宿主主配置文件，而不是`sys_config`

宿主新增静态配置结构`demo.control.enabled`，默认值为`false`，由`apps/lina-core/internal/service/config/`统一读取，并通过发布给源码插件的配置服务暴露只读访问能力。

这样设计的原因：

- 演示模式本质上是环境级治理策略，更适合作为部署配置，而不是运行时业务数据。
- 静态配置不会受到演示模式写保护自身的影响，避免出现“开启后无法通过系统接口关闭”的循环依赖。
- 宿主已经有完善的静态配置缓存与结构化读取机制，实现成本和行为边界都更清晰。

备选方案与取舍：

- 放入`sys_config`：虽然可在管理界面动态修改，但演示模式启用后会阻断自身写接口，形成治理死锁，放弃。
- 直接硬编码常量开关：无法按环境配置，也不符合宿主配置治理模式，放弃。

### 决策二：以官方源码插件`demo-control`承载请求拦截，而不是修改宿主内建中间件

新增`apps/lina-plugins/demo-control/`源码插件，并通过宿主发布的`registrar.GlobalMiddlewares().Bind("/api/v1/*", ...)`接入统一请求链路。

这样设计的原因：

- 需求本身就是一个典型的“环境治理能力”，适合作为一方插件示例，验证源码插件对宿主全局中间件接缝的复用方式。
- 宿主核心中间件链路保持通用，不直接绑定“演示模式”这一特定场景语义。
- 插件启停状态仍由既有插件生命周期治理，后续若需要临时摘除也更容易操作。

备选方案与取舍：

- 直接在`apps/lina-core/internal/service/middleware/`新增宿主逻辑：实现最简单，但会把可选治理能力回灌到宿主核心，放弃。

### 决策三：写保护以`HTTP Method`为主判断依据，并保持最小白名单

`demo-control`插件的全局中间件采用以下规则：

- 当`demo.control.enabled=false`时，所有请求直接放行。
- 当开关开启时，默认允许`GET`、`HEAD`、`OPTIONS`。
- 当开关开启时，默认拒绝`POST`、`PUT`、`DELETE`等写操作方法。
- 为保证演示环境可用，额外放行`POST /api/v1/auth/login`与`POST /api/v1/auth/logout`。

这样设计的原因：

- 用户已经明确建议以`RESTful API`的`HTTP Method`作为控制依据，现有项目也要求查询操作必须使用`GET`，因此这一策略与项目接口治理规则一致。
- 以方法为主的判断成本低、覆盖面广，能够同时保护宿主 API、源码插件 API 与动态插件扩展 API。
- 登录、登出不是业务数据写操作，但属于会话入口；若不白名单，演示环境将无法正常使用。

备选方案与取舍：

- 基于 URL 逐个接口白名单：精度更高，但维护成本过重，且与当前“按 RESTful 语义统一治理”的需求不匹配，放弃。
- 完全不设白名单：会阻断登录和登出，演示环境不可用，放弃。

### 决策四：`demo-control`默认加入宿主启动自动启用列表

宿主默认配置中的`plugin.autoEnable`将追加`demo-control`，使该插件在宿主启动时自动安装并启用；而真正是否拦截写操作，仍由`demo.control.enabled`决定。

这样设计的原因：

- 插件默认启用后，演示环境只需要切换一个静态配置开关即可进入只读保护，无需额外执行插件安装启用步骤。
- 开关默认关闭，因此即使插件自动启用，也不会影响普通开发或非演示环境的写操作行为。
- 这也能验证源码插件“自动启用 + 全局中间件”这一组合场景的有效性。

备选方案与取舍：

- 只在演示环境手工启用插件：会增加部署步骤，不符合“开箱即具备能力”的目标，放弃。

## Risks / Trade-offs

- [Risk] 依赖`HTTP Method`判断意味着任何违反`RESTful`规范的旧接口都可能绕过或被误拦截。→ Mitigation：当前项目明确要求所有 API 严格遵循`RESTful`语义，因此本次直接按规范实施，不为非规范接口增加兼容分支。
- [Risk] 默认自动启用插件会让所有环境都挂载额外一层全局中间件。→ Mitigation：开关默认关闭时中间件只做极轻量配置判断并立即放行。
- [Risk] 登录、登出白名单范围过窄或过宽都可能影响演示可用性或保护强度。→ Mitigation：本次仅保留最小会话入口白名单，并通过测试覆盖显式校验。

## Migration Plan

1. 在宿主主配置文件模板与默认配置中新增`demo.control.enabled`，默认值为`false`。
2. 扩展宿主配置服务与源码插件配置服务契约，向插件发布演示控制开关读取能力。
3. 新增`demo-control`源码插件并接入插件工作区、`go.work`与默认`plugin.autoEnable`列表。
4. 补充单元测试，验证配置默认值、配置覆盖值、允许方法、拒绝方法与登录白名单行为。
5. 发布后，如需在演示环境启用，只需把`demo.control.enabled`改为`true`并重启宿主；如需回滚，只需改回`false`并重启。
