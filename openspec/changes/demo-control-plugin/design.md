## Context

当前宿主已经具备两个与本次需求直接相关的基础能力：

1. 宿主主配置文件已经支持`plugin.autoEnable`，能够在启动阶段自动安装并启用指定源码插件。
2. 宿主已经通过`pluginhost.HTTPRegistrar`向源码插件发布了全局 HTTP 中间件注册能力，`monitor-operlog`等插件已经在使用这一接缝。

但现状仍有一个明显空白：缺少一个面向演示环境的统一写保护开关。若直接依赖管理员自觉避免点击写操作按钮，无法真正保障演示数据安全；若把能力硬编码在宿主中间件里，又会削弱源码插件架构的扩展边界。

另外，本次还需要处理一个关键约束：既然宿主已经提供了`plugin.autoEnable`这类启动期静态启用入口，就没有必要再为演示能力增加第二个布尔开关。否则会同时出现“插件是否启用”和“开关是否开启”两层状态，增加理解与运维成本。因此更适合直接把`demo-control`是否被列入`plugin.autoEnable`视为演示能力是否开启。

## Goals / Non-Goals

**Goals:**

- 以官方源码插件`demo-control`承载演示能力，而不是把该逻辑散落在宿主核心中间件实现里。
- 通过`plugin.autoEnable`控制`demo-control`是否在启动期自动启用，并以此作为演示能力的唯一开关入口。
- 在`demo-control`启用时，统一拦截系统 API 的非查询写操作，优先复用`RESTful API`的`HTTP Method`语义作为判断依据。
- 保留最小必要白名单，确保演示环境仍可完成登录、登出等基础会话动作。

**Non-Goals:**

- 不在本次变更中增加演示模式专属前端页面、提示条、横幅或工作台视觉标识。
- 不在本次变更中把演示控制扩展为细粒度资源级白名单、角色级例外或按模块选择性放行。
- 不在本次变更中为动态插件设计独立的中间件声明模型；动态插件只通过既有`/api/v1`统一链路被演示控制插件覆盖。
- 不在本次变更中引入数据库表、安装 SQL、额外的宿主布尔配置项或新的运行时参数表结构。

## Decisions

### 决策一：以`plugin.autoEnable`是否包含`demo-control`作为唯一开关入口

宿主不新增`demo.control.enabled`之类的独立布尔配置，而是直接复用现有`plugin.autoEnable`。行为语义为：

- `plugin.autoEnable`未包含`demo-control`时，演示控制能力默认关闭。
- `plugin.autoEnable`包含`demo-control`时，宿主在启动阶段自动安装并启用该插件，演示控制能力随之开启。

这样设计的原因：

- 用户只需要理解“插件启用即能力生效”这一层语义，不需要再维护第二个开关。
- `plugin.autoEnable`本身就是宿主已有的启动期静态配置入口，能够满足演示环境按部署配置启停的诉求。
- 关闭演示能力时，只需从`plugin.autoEnable`移除`demo-control`并重启宿主，治理路径一致且清晰。

备选方案与取舍：

- 新增`demo.control.enabled`静态布尔配置：虽然也可工作，但会形成与插件启用状态并存的双重开关，增加理解成本，放弃。
- 放入`sys_config`：虽然可在管理界面动态修改，但与演示保护自身存在循环治理问题，放弃。

### 决策二：以官方源码插件`demo-control`承载请求拦截，而不是修改宿主内建中间件

新增`apps/lina-plugins/demo-control/`源码插件，并通过宿主发布的`registrar.GlobalMiddlewares().Bind("/api/v1/*", ...)`接入统一请求链路。

这样设计的原因：

- 需求本身就是一个典型的“环境治理能力”，适合作为一方插件示例，验证源码插件对宿主全局中间件接缝的复用方式。
- 宿主核心中间件链路保持通用，不直接绑定“演示模式”这一特定场景语义。
- 插件启停状态仍由既有插件生命周期治理，而`plugin.autoEnable`可以直接承担环境级启停开关职责。

备选方案与取舍：

- 直接在`apps/lina-core/internal/service/middleware/`新增宿主逻辑：实现最简单，但会把可选治理能力回灌到宿主核心，放弃。

### 决策三：写保护以`HTTP Method`为主判断依据，并保持最小白名单

`demo-control`插件的全局中间件采用以下规则：

- 当开关开启时，默认允许`GET`、`HEAD`、`OPTIONS`。
- 当开关开启时，默认拒绝`POST`、`PUT`、`DELETE`等写操作方法。
- 为保证演示环境可用，额外放行`POST /api/v1/auth/login`与`POST /api/v1/auth/logout`。

这样设计的原因：

- 用户已经明确建议以`RESTful API`的`HTTP Method`作为控制依据，现有项目也要求查询操作必须使用`GET`，因此这一策略与项目接口治理规则一致。
- 以方法为主的判断成本低、覆盖面广，能够同时保护宿主 API、源码插件 API 与动态插件扩展 API。
- 登录、登出不是业务数据写操作，但属于会话入口；若不白名单，演示环境将无法正常使用。
- 插件仅在启用时才会接入并执行该规则；未启用时，请求链路保持现状。

备选方案与取舍：

- 基于 URL 逐个接口白名单：精度更高，但维护成本过重，且与当前“按 RESTful 语义统一治理”的需求不匹配，放弃。
- 完全不设白名单：会阻断登录和登出，演示环境不可用，放弃。

### 决策四：配置模板给出`plugin.autoEnable`示例，但默认保持未启用

这样设计的原因：

- 用户最初要求“默认关闭”，在移除独立布尔开关后，最自然的表达方式就是默认不把`demo-control`放入`plugin.autoEnable`。
- 同时在配置模板中提供明确示例，演示环境只需把`demo-control`加入`plugin.autoEnable`即可生效。
- 这样既保留默认安全的开发体验，也复用了已有的插件启动治理能力。

备选方案与取舍：

- 直接把`demo-control`放进默认`plugin.autoEnable`列表：会让所有默认启动环境都进入只读保护，不符合“默认关闭”，放弃。

## Risks / Trade-offs

- [Risk] 依赖`HTTP Method`判断意味着任何违反`RESTful`规范的旧接口都可能绕过或被误拦截。→ Mitigation：当前项目明确要求所有 API 严格遵循`RESTful`语义，因此本次直接按规范实施，不为非规范接口增加兼容分支。
- [Risk] 使用插件启用状态作为唯一开关，意味着切换演示模式需要修改配置并重启宿主。→ Mitigation：该能力本身就是环境级治理策略，复用启动期配置比引入第二个布尔开关更清晰。
- [Risk] 登录、登出白名单范围过窄或过宽都可能影响演示可用性或保护强度。→ Mitigation：本次仅保留最小会话入口白名单，并通过测试覆盖显式校验。

## Migration Plan

1. 新增`demo-control`源码插件并接入插件工作区、`go.work`与插件统一接线入口。
2. 在宿主配置模板中补充`plugin.autoEnable`示例，明确把`demo-control`加入列表即可启用演示保护。
3. 补充单元测试，验证未启用插件时不拦截、启用插件时允许查询并拒绝写操作，以及登录白名单行为。
4. 发布后，如需在演示环境启用，只需把`demo-control`加入`plugin.autoEnable`并重启宿主；如需回滚，只需移除该插件 ID 并重启。
