## Why

当前 LinaPro 的宿主服务已经承担了权限、组织、内容、监控、任务、开发工具等多类后台能力，随着功能持续增加，左侧菜单与宿主代码边界都在朝“大一统后台”方向膨胀。这会削弱 LinaPro 作为“AI 驱动的全栈开发框架”的定位：开发者下载项目后虽然可以快速开始，但很难清晰地区分哪些是必须保留的框架内核，哪些是可以按需裁剪、替换或独立演进的业务模块。

开源阶段需要优先完成一次宿主边界收敛和源码插件化重构：宿主只保留认证、权限、菜单、插件治理、任务调度等框架底座能力；组织管理、内容管理和系统监控中的非核心模块改为源码插件形式交付，让开发者能够按需安装和启用，默认获得更轻量的后台基础盘，同时保留后续叠加业务的稳定菜单结构与演进路径。

## What Changes

- 重构默认管理后台一级菜单结构，形成 9 个由宿主稳定提供的后台挂载点：`工作台(dashboard)`、`权限管理(iam)`、`组织管理(org)`、`系统设置(setting)`、`内容管理(content)`、`系统监控(monitor)`、`任务调度(scheduler)`、`扩展中心(extension)`、`开发中心(developer)`。
- 规定这些一级目录必须由宿主以稳定菜单记录显式创建和拥有，插件只能向这些目录下挂载菜单，不能自行发明新的一级目录体系。
- 明确宿主边界：`用户管理`、`角色管理`、`菜单管理`、`插件管理`、`任务调度`、认证会话内核、配置与字典等保留在宿主。
- 将 `组织管理` 规划为源码插件 `org-center`，承载部门管理与岗位管理。
- 将 `内容管理` 中的通知公告规划为源码插件 `content-notice`。
- 将 `系统监控` 下的 `在线用户`、`服务监控`、`操作日志`、`登录日志` 分别规划为 4 个独立源码插件：`monitor-online`、`monitor-server`、`monitor-operlog`、`monitor-loginlog`。
- 规定插件治理与导航分离：`扩展中心` 只承载插件治理入口，插件实际功能菜单必须挂到语义对应的宿主目录下。
- 规定空父目录通过导航投影自动隐藏，但宿主仍保留这些稳定目录记录，保证 `parent_key` 解析和后续扩展长期稳定。
- 为后续插件迁移补充宿主解耦要求：用户管理对组织能力降级、认证会话与在线用户治理解耦、登录日志与操作日志改为宿主事件发射 + 插件落库模式。
- 补充插件存储边界治理：宿主默认 `init/mock` 不再创建、写入或假定任何源码插件业务表；官方源码插件业务表统一迁移到插件作用域命名空间，宿主不再保留对应 ORM 工件与直接查表实现。

## Capabilities

### New Capabilities

- `core-host-boundary-governance`: 定义开源阶段宿主只保留框架内核与治理底座的边界约束。
- `module-decoupling`: 定义非核心管理模块以源码插件交付时的独立启停与宿主降级规则。
- `menu-management`: 为默认后台引入稳定一级目录结构、插件菜单语义挂载与空目录隐藏规则。
- `plugin-manifest-lifecycle`: 为官方源码插件补充固定挂载点、领域-能力式插件 ID 与宿主目录父键约束。

### Modified Capabilities

- `dept-management`: 部门管理能力后续由 `org-center` 源码插件交付。
- `post-management`: 岗位管理能力后续由 `org-center` 源码插件交付。
- `notice-management`: 通知公告能力后续由 `content-notice` 源码插件交付，并从 `内容管理` 挂载。
- `online-user`: 在线用户能力后续由 `monitor-online` 源码插件交付，宿主仅保留认证会话内核。
- `server-monitor`: 服务监控能力后续由 `monitor-server` 源码插件交付。
- `oper-log`: 操作日志能力后续由 `monitor-operlog` 源码插件交付，宿主改为发射统一审计事件。
- `login-log`: 登录日志能力后续由 `monitor-loginlog` 源码插件交付，宿主改为发射统一登录事件。
- `user-management`: 用户管理在组织插件缺失时需要隐藏部门/岗位相关筛选、字段和表单项并保持主体功能可用。
- `user-auth`: 认证链路需要发布登录生命周期事件，但不得再直接依赖具体登录日志持久化实现。

## Impact

### 后端影响

- 调整宿主菜单初始化 SQL 与菜单投影逻辑，提供稳定一级目录、精确父级 `menu_key` 与可隐藏的宿主目录骨架。
- 抽离组织、通知、监控相关模块的宿主依赖关系，为源码插件迁移做准备。
- 为登录、审计、在线会话、服务监控等链路补充“宿主内核 / 插件展示治理”边界。
- 新增官方源码插件目录与显式接线。
- 收紧插件存储边界：宿主不再持有插件业务表的 `dao/do/entity`、Mock 数据和直接查表逻辑，插件通过安装 SQL 与能力 Provider 自主管理业务存储。

### 前端影响

- 调整默认管理后台的左侧菜单层级与路由分组。
- 用户管理、监控等页面根据插件可用性进行 UI 降级与隐藏。
- 插件启停后动态刷新导航结果，保证空父目录自动收敛。

### 研发与交付影响

- 新增官方源码插件模板与命名约束。
- 新增插件自有表命名约束与默认数据库装载边界约束。
- 需要为插件启停、空目录隐藏、宿主降级与菜单刷新补充 E2E 覆盖。
- 后续模块实现必须优先判断该能力是否应落在宿主还是源码插件。
