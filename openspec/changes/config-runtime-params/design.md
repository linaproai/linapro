## Context

当前代码已经为参数设置模块落地了基础 CRUD 能力，但这次需求不是继续扩展一个通用键值表，而是把“已经被宿主真实消费、且适合运行时修改”的参数正式收敛到一个可维护的清单中。

本次变更同时涉及多个层面：
- 参数元数据初始化 SQL
- 宿主配置读取层（`internal/service/config`）
- 参数管理服务保护逻辑（`internal/service/sysconfig`）
- 认证、鉴权、在线会话、文件上传、用户重置密码等消费端行为
- Go 单测、Playwright E2E 与 OpenSpec 文档同步

仓库约束要求活跃变更先写 `openspec/changes/<change>/` 下的 proposal/design/tasks/specs，归档时再同步 `openspec/specs/` 基线规范。因此本次设计还需要把此前直接写入基线规范的内容收敛为 active change 的增量规范。

## Goals / Non-Goals

**Goals:**
- 明确当前真正适合纳入参数设置页的内置运行时参数范围，并形成稳定注册表
- 为这些参数补齐初始化元数据、默认值、格式说明与后端校验规则
- 确保参数改动能真实作用于宿主运行时行为，而不只是停留在页面展示层
- 在多实例部署下避免运行时热点链路反复直查 `sys_config`，同时保证参数变更能在各实例间有界传播
- 对关键内置参数增加防误删、防改键名和导入校验，避免宿主行为失控
- 为每一项纳管参数补齐单元测试或 E2E 测试，形成可回归验证链路
- 补齐本次功能对应的 OpenSpec proposal/design/tasks/specs，并恢复“先 change、后归档”的流程一致性

**Non-Goals:**
- 不把所有 `config.yaml` 中的配置都迁移到参数设置页
- 不纳管需要重启才能安全生效或安全风险较高的配置，例如 JWT secret、logger、cluster、存储路径等
- 不额外引入 Redis、消息队列或发布订阅中间件来同步参数变更
- 不重构参数设置页的整体交互模式，只在现有页面与现有 API 基础上补齐内置参数能力

## Decisions

### 1. 使用宿主配置服务维护统一的运行时参数注册表

在 `internal/service/config` 中维护内置运行时参数注册表，集中定义参数键名、展示名称、默认值、备注和格式校验规则。

这样做的原因：
- 避免把参数合法性分散在认证、会话、上传、用户等各个消费模块中
- 让 SQL 种子数据、参数页展示和运行时读取共享同一份契约
- 后续若再纳管新的运行时参数，只需在注册表中扩展即可

备选方案是把校验逻辑分别写在各消费模块或前端页面中，但那样会导致规则分散、文档难同步、导入场景难保证一致性，因此不采用。

### 2. 只纳管“已接入宿主且可安全运行时生效”的参数

本次纳管 5 个参数：
- `sys.jwt.expire`
- `sys.session.timeout`
- `sys.upload.maxSize`
- `sys.login.blackIPList`
- `sys.user.initPassword`

选择标准是：
- 宿主代码已经存在明确消费点
- 修改后无需重启，或无需额外运维操作即可生效
- 不会引入密钥、集群拓扑或底层存储路径这类高风险变更

备选方案是把更多配置项一并放入参数设置页，但这样会把“通用键值管理”和“运行时安全配置”混在一起，增加误操作风险，因此本次显式收敛范围。

### 3. 在 `sysconfig` 服务层统一执行内置参数保护

对内置运行时参数的保护放在 `internal/service/sysconfig` 服务层执行，包括：
- 创建/更新/导入时校验值格式
- 禁止修改已接入宿主的稳定键名
- 禁止删除内置运行时参数

这样做的原因：
- 服务层可以同时覆盖 API 修改、Excel 导入及后续其他入口
- 相比仅在前端置灰或隐藏按钮，后端约束能保证真正的数据一致性

备选方案是只在前端限制操作，但无法防住绕过 UI 的请求，因此不采用。

### 4. 运行时消费按“配置文件回退 + 数据库覆盖”分层实现

各消费点统一遵循以下策略：
- 先读取已有 `config.yaml` 默认值或现有硬编码缺省值
- 再尝试读取 `sys_config` 中对应参数并覆盖
- 当数据库没有记录或值为空时，自动回退到原默认配置

具体接入方式：
- JWT 有效期：在 `GetJwt` 中覆盖 `Expire`
- 会话超时：在 `GetSession` 中覆盖 `Timeout`
- 上传大小：在 `GetUpload` 中覆盖 `MaxSize`
- 登录黑名单：在认证服务登录前校验来源 IP
- 初始密码：前端打开重置密码弹窗时通过现有按键名查询接口读取

这样既能兼容历史配置，又能让参数设置成为新入口。

### 5. 会话超时在鉴权链路实时生效，而不是仅依赖定时清理

`sys.session.timeout` 不只用于定时清理任务，还会在认证中间件每次访问受保护接口时实时校验 `last_active_time`。若会话已超时，则当次请求直接返回 401，并清理会话记录。

这样做的原因：
- 仅靠 cron 清理会产生“会话已超时但在下一次清理前仍可访问”的窗口期
- 鉴权链路实时校验更符合管理员修改参数后应立即生效的预期

代价是每次鉴权需要读取并更新一次在线会话记录，但这是现有在线会话模型本就承担的职责，成本可接受。

### 6. 测试采用“页面元数据 + 运行时行为”双层覆盖

测试分层如下：
- 参数设置页 E2E：验证 5 个内置参数可在页面检索到
- 用户侧 E2E：验证 `sys.user.initPassword` 会影响重置密码弹窗默认值
- 配置层单测：验证时长/整数/IP 黑名单/密码等格式校验与运行时覆盖
- 业务层单测：验证登录黑名单拦截、会话超时校验、上传大小上限拦截、内置参数防删除/防改键名

这样可以同时覆盖“看得见”和“真正生效”两类风险。

### 7. 多实例场景采用“本地 gcache 快照 + 共享修订号 watcher”两层缓存

运行时参数读取不能在每次请求时都直查 `sys_config`，否则会把 JWT 鉴权、登录黑名单判断、会话超时校验和上传大小校验这些热点路径都绑定到数据库读压力上。与此同时，单机内存缓存又无法满足多实例部署下的参数同步需求。

因此本次反馈追加如下设计：
- **L1 本地快照**：每个进程在独立 `gcache` 实例中缓存一份已解析的运行时参数快照，请求链路只读本地缓存
- **同实例防击穿**：本地缓存 miss 时使用 `gcache.GetOrSetFuncLock` 串行化同一实例内的兜底加载，避免并发请求同时穿透到 `sys_config`
- **L2 共享修订号**：复用现有 `kvcache`/`sys_kv_cache` 维护跨实例共享 revision，而不是引入新的中间件；读路径新增纯读取整数值接口，不再通过 `Incr(delta=0)` 触发事务型更新
- **后台 watcher**：各实例通过 `cron` 中的 all-node job 每 `10s` 检查一次共享 revision；若 revision 未变化，则仅续租本地缓存的 `1h` 兜底 TTL
- **写后失效**：当 `sysconfig` 的创建、更新或导入真正改动了受保护运行时参数的值时，当前实例在事务内同步递增共享 revision，并立即清空本地快照，让后续请求按需重建
- **读时重建**：其他实例在下一次 watcher 发现共享 revision 已变化后，再从 `sys_config` 重新加载并解析快照；若 watcher 异常或节点冷启动，则首次 miss 仍可通过 `GetOrSetFuncLock` 兜底加载

这样既能把热点路径从“每次请求查库”降为“绝大多数请求只读内存”，也能在多实例之间提供约 `10s` 内的有界最终一致性，同时保留 `1h` 本地 TTL 作为 watcher 异常时的自愈兜底。

### 8. 配置文件静态项使用 once 对象缓存，热点链路按值解耦读取

在当前实现里，运行时参数已经通过本地快照缓存避免了热点链路直查数据库，但 `GetJwt`、`GetSession`、`GetUpload` 等配置方法仍然会在每次调用时重新构造配置对象并读取 `g.Cfg()`。对于 JWT 鉴权、在线会话校验等高频路径来说，这部分重复的静态配置读取和对象分配仍然没有必要。

因此本次反馈再追加如下设计：
- **静态配置对象缓存**：所有来自配置文件、且不需要运行时参数覆盖的基础配置项统一以对象形式缓存到宿主进程内，并使用 `sync.Once` 只初始化一次
- **返回副本而非共享指针**：对外暴露配置对象时返回缓存对象的副本，避免调用方误改共享内存
- **动态值后置叠加**：像 `sys.jwt.expire`、`sys.session.timeout`、`sys.upload.maxSize` 这类可运行时变更的值，不直接缓存最终结果，而是在读取静态对象后再叠加运行时快照
- **按用途拆分 getter**：对鉴权、会话、登录黑名单、上传校验等热点路径拆出更细粒度的方法，例如 JWT 校验只读取 `Secret`，会话校验只读取 `Timeout`，登录拦截直接判断黑名单命中，上传链路分别读取 `Path` 和 `MaxSize`，避免在热点链路里组装完整配置对象或重复解析规则

这样可以把“静态配置解析”与“动态运行时覆盖”分层处理：基础配置只做一次，热点路径按需拿值，运行时参数仍通过 shared revision + 本地快照机制保持多实例同步。

## Risks / Trade-offs

- **[部分配置仍未纳管]** → 本次只覆盖已确认可安全运行时生效的 5 个参数；其余配置继续保留在 `config.yaml`，并在 proposal/specs 中显式说明范围
- **[数据库异常导致读取失败]** → 配置读取层在读取不到运行时参数时回退到既有默认值，避免宿主因为参数页数据异常而无法启动或无法提供基础能力
- **[多实例之间存在短暂传播窗口]** → 接受基于共享 revision 刷新的最终一致性；本实例写后立即失效，其他实例在下一次 revision 同步周期内完成快照重建
- **[会话超时实时校验增加一次数据库读写]** → 保留在线会话表模型不变，并用定向单测验证该路径的正确性，接受这一点运行时开销以换取参数即时生效
- **[文档流程曾被提前写入基线规范]** → 在本次 change 中补齐 proposal/design/tasks/delta specs，并将提前写入 `openspec/specs/` 的内容回收，归档时再统一同步基线

## Migration Plan

1. 在 `007-config-management.sql` 中直接维护 5 个内置参数种子数据，并通过一次 `make init` 将其写入 `sys_config`
2. 部署宿主后端代码，使配置读取层和各消费点开始读取运行时参数
3. 在多实例环境中复用现有 `sys_kv_cache` 共享 revision，使任一节点修改运行时参数后可驱动其他节点刷新本地快照
4. 部署前端工作台，使重置密码弹窗改为读取 `sys.user.initPassword`
5. 运行相关 Go 单测、前端 lint 与 Playwright 用例，确认参数元数据、缓存刷新和运行时行为全部通过
6. 归档本次 change 时，再将 delta specs 同步到 `openspec/specs/` 基线规范
