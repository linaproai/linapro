## Context

当前动态插件安装/启用授权弹窗的核心信息源是插件列表行数据。前端弹窗在打开时直接读取 `SystemPlugin` 行对象中的 `requestedHostServices` 并渲染宿主服务卡片，因此页面交互足够轻量，但也意味着凡是弹窗要展示的新治理信息，都需要先由后端在插件管理数据投影阶段准备好。现状中，动态路由合同已经存在于动态插件 manifest/release 快照里，并且宿主已经能基于这些合同投影真实公开路径到 OpenAPI 文档；只是插件管理视图尚未把这部分信息暴露给管理员。

这次改动同时跨越后端 manifest 投影、插件管理 DTO、前端授权弹窗视图模型与测试链路，属于一个小型跨层治理增强。设计上需要保证：一是路由暴露面与 `hostServices` 授权边界清晰分离；二是授权弹窗继续保持“打开即展示”，不额外引入新的请求往返；三是展示内容使用宿主真实公开路径，而不是只暴露 guest 侧原始 `route.Path`。

## Goals / Non-Goals

**Goals:**
- 在动态插件安装/启用授权审查弹窗中新增独立的路由信息区，展示当前 release 声明的动态路由清单。
- 让管理员在同一个弹窗内同时看到“插件将访问哪些宿主资源”和“插件将暴露哪些对外入口”。
- 让授权弹窗优先展示核心宿主服务授权信息，路由暴露面后置显示，并在默认视图中只展开前两条路由。
- 让授权弹窗和详情弹窗中的板块标题具备明确的视觉分隔能力，而不是与正文内容混在一起。
- 由后端统一投影路由展示字段，前端直接消费结构化结果，避免弹窗侧重复拼装宿主真实公开路径。
- 保持现有授权提交模型不变，新增展示只影响审查信息，不引入逐条路由确认或新的授权快照结构。
- 在插件详情弹窗中复用与授权弹窗一致的路由审查结构，让安装后的治理复盘也能直接查看当前 release 的对外暴露面。
- 优化插件详情弹窗的基础信息布局，移除冗余的“授权要求”字段，并让“插件描述”以表格中的独立整行展示。
- 补齐后端投影测试与前端/端到端展示验证，覆盖 `public`、`login`、权限标识和空路由场景。

**Non-Goals:**
- 不改变动态路由的运行时鉴权、权限校验、OpenAPI 投影或执行器逻辑。
- 不为动态路由新增逐条授权、勾选、裁剪或二次确认机制。
- 不扩展源码插件静态路由清单展示，本次范围仅限动态插件授权审查窗口与插件详情弹窗中的动态插件视图。
- 不新建独立的插件详情拉取接口或专门的路由审查接口。

## Decisions

### 决策一：将路由信息作为授权弹窗中的独立审查区，而不是混入宿主服务授权卡片

**选择**：新增一组独立的“路由信息/注册路由列表”视图块，与宿主服务卡片并列展示，但在语义上明确区分“对外暴露面”与“宿主资源授权范围”。

**原因**：
- `hostServices` 描述的是插件向宿主申请的能力边界；动态路由描述的是插件向外暴露的访问入口。这两类信息面向的治理问题不同，混排会让管理员误以为路由也是需要授权确认的资源项。
- 现有宿主服务卡片已经围绕服务、方法、资源列表设计；强行把路由塞进去会破坏已有分组结构，也会让后续 cron、network、storage 等服务卡片的语义变得模糊。

**备选方案**：
- 把动态路由折叠为一种新的“服务卡片”。拒绝，因其会错误暗示路由属于 `hostServices` 授权对象。
- 仅在插件详情弹窗展示路由，不在授权弹窗展示。拒绝，因为管理员需要在安装/启用前完成审查，而不是事后补看。

### 决策二：后端统一投影动态路由展示模型，并直接返回宿主真实公开路径

**选择**：在插件管理后端 DTO 中新增动态路由展示字段，由后端基于动态插件 manifest/release 快照中的 `RouteContract` 生成结构化路由列表，并直接计算宿主真实公开路径。

**原因**：
- 动态插件真实可访问路径不是单纯的 `route.Path`，而是宿主前缀 `/api/v1/extensions/{pluginId}/...` 与合同路径拼接后的结果；这属于宿主治理语义，应该由后端统一计算，避免前端重复维护同一规则。
- 前端授权弹窗当前直接使用列表行数据打开，继续沿用这一模式可以避免新增弹窗打开时的二次请求，也降低交互复杂度。

**返回字段建议**：
- `method`
- `publicPath`
- `access`
- `permission`
- `summary`
- `description`（可返回，首版 UI 可按需展示）

**备选方案**：
- 前端根据 `pluginId + route.Path` 自行拼接公开路径。拒绝，因为路径规则应以宿主为准，且会导致多处重复实现。
- 新增独立查询接口按需拉取路由列表。拒绝，因为当前弹窗场景不需要额外请求，且会增加状态管理成本。

### 决策三：路由列表保持只读，并沿用声明顺序展示

**选择**：授权弹窗和插件详情弹窗都只读展示当前 release 的动态路由列表，不支持勾选、过滤或排序裁剪；默认沿用当前 release 中的声明顺序返回与展示，但首屏只展示前两条，剩余路由通过显式“展开”操作查看。

**原因**：
- 本次需求的核心是“补足审查可见性”，不是引入新的治理裁剪能力；继续保持只读能与现有宿主服务审查流程一致。
- 沿用声明顺序最接近“注册路由列表”的心智模型，也便于管理员对照插件作者声明、系统接口文档和后续调试信息。
- 路由声明较多时，如果一次性完全展开，会在授权弹窗中淹没更关键的宿主服务授权范围信息；默认显示前两条能同时兼顾信息优先级与可追溯性。

**备选方案**：
- 按 `public` 优先重新排序。未采用，虽然有助于风险前置，但会让展示顺序偏离作者声明；首版以声明顺序加显式访问级别标签即可。
- 支持按路由逐条确认。拒绝，因为会把当前默认整包确认的治理模型扩展成新的权限裁剪模型，超出本次范围。
- 默认全部展开。未采用，因为当插件声明较多路由时，会让授权和详情视图的信息密度过高，影响关键治理信息的首屏可读性。

### 决策四：插件详情弹窗复用同一套路由展示结构

**选择**：插件详情弹窗中的动态插件视图复用与授权弹窗一致的路由列表组件，同样把路由信息放在宿主服务信息之后。

**原因**：
- 详情弹窗的治理信息与授权弹窗关注点一致，只是一个用于安装前确认，一个用于安装后复盘；采用同一组件与层级结构能降低理解成本。

**备选方案**：
- 仅在授权弹窗显示路由，详情页不展示。拒绝，因为管理员在安装后仍可能需要复查当前 release 的暴露面。

### 决策五：板块标题需要显式承担信息分区作用

**选择**：在授权弹窗和详情弹窗中，为“宿主服务授权范围/宿主服务信息”“注册路由列表”等区块使用更大的字号、更高字重和带强调色的标题样式，让标题与正文形成明确视觉断层。

**原因**：
- 当前标题文字与正文接近，用户很难快速识别这是板块分隔标题，尤其在弹窗内容较多时更容易失焦。
- 这两个区块都是治理信息的一级分区，使用更清晰的视觉层级比继续沿用普通正文样式更符合信息架构。

**备选方案**：
- 仅略微加粗文字。未采用，因为对比度仍然不足，无法稳定建立板块标题心智。
- 通过增加大量空白分隔。未采用，因为会拉长弹窗整体高度，却仍然不能充分提升标题识别度。

### 决策六：详情弹窗移除“授权要求”，将描述回收到表格整行

**选择**：在插件详情弹窗中移除“授权要求”字段，把“插件描述”重新放回 `Descriptions` 表格，并作为 `span=2` 的独立整行展示。

**原因**：
- “授权要求”与“授权状态”语义接近，前者在详情页的独立展示价值较低，移除后基础信息表格更紧凑。
- 当前描述已经是基础元数据的一部分，回收到表格整行后仍然能完整展示，而且比额外插入一个独立说明块更统一、更简洁。

**备选方案**：
- 同时保留“授权要求”和独立描述块。未采用，因为会继续增加详情页上半区的信息密度。
- 直接删掉描述字段。拒绝，因为会损失插件基础信息完整性。

### 决策七：无动态路由时不渲染冗余列表块，但测试需要覆盖该空态

**选择**：当当前 release 未声明动态路由时，授权弹窗不渲染冗余的路由列表块；后端仍返回空数组或空值，前端按无数据处理。

**原因**：
- 该弹窗的首要任务是帮助管理员审查“实际存在的治理信息”；对没有路由的插件渲染空块会稀释重点。
- 前端已有按是否存在数据决定是否展示某一信息块的模式，保持一致更利于实现和维护。

**备选方案**：
- 永远展示“未声明动态路由”的空态提示。未采用，原因是会在多数只关注宿主服务授权的场景中增加视觉噪音。

## Risks / Trade-offs

- [Risk] 插件列表接口行数据体积增大，动态插件较多且单插件路由较多时会抬高响应体大小。→ Mitigation：首版仅返回授权弹窗实际需要的精简字段，不回传请求类型、内部路径等运行时细节。
- [Risk] 前端若把路由信息错误地设计成“授权项”，会让用户误解现有治理模型。→ Mitigation：在设计与规格中明确其为并列审查信息区，并在文案上避免使用“授权路由”表述。
- [Risk] 宿主真实公开路径拼装规则若在多处实现，后续前缀调整时容易漂移。→ Mitigation：统一复用后端已有的动态路由公开路径构建逻辑，由后端投影单一结果给前端。
- [Risk] 没有路由的动态插件场景容易在开发阶段被忽略。→ Mitigation：在后端投影测试和前端展示测试中都覆盖空路由清单场景。

## Migration Plan

1. 更新 `plugin-manifest-lifecycle` 的增量规范，明确授权弹窗新增路由暴露面审查要求。
2. 在后端插件管理投影层新增动态路由展示 DTO，并把当前 release 的路由合同映射为前端可直接消费的数据结构。
3. 更新前端动态插件安装/启用授权弹窗，新增独立的路由信息展示区并接入后端字段。
4. 补充后端单元测试、前端组件测试或 E2E 用例，验证典型路由组合与空路由场景。
5. 实现完成后，通过 `/opsx:apply` 进入编码与验证阶段；若上线后需要回滚，可先移除前端展示并回退后端新增字段，因本次不涉及数据库和持久化结构变更，回滚成本较低。

## Open Questions

- 首版路由列表是否需要同步展示 `description`，还是只展示 `summary` 即可满足管理员审查需求？当前设计默认后端返回两者，前端可先展示 `summary` 为主。
- 路由信息是否还需要在插件详情弹窗中复用展示，以便管理员在安装后继续审查？本次变更先聚焦授权弹窗，不把详情弹窗纳入强制范围。
