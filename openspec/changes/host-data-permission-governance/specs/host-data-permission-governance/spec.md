## ADDED Requirements

### Requirement: 宿主必须解析当前用户的有效数据权限范围

系统 SHALL 提供统一的宿主数据权限解析能力，根据当前请求用户、启用角色和角色 `dataScope` 计算有效数据范围。超级管理员 SHALL 获得全部数据范围。普通用户拥有多个启用角色时，系统 SHALL 采用最宽范围合并规则：全部数据优先于本部门数据，本部门数据优先于仅本人数据。没有登录用户上下文、没有启用角色或无法确认角色数据范围时，系统 SHALL 失败关闭，不得默认放宽为全部数据。

#### Scenario: 超级管理员获得全部数据范围

- **当** 内置超级管理员用户访问接入数据权限治理的宿主资源时
- **则** 系统将有效数据范围解析为全部数据
- **且** 后续资源查询和写操作不附加部门或本人范围限制

#### Scenario: 多角色采用最宽范围

- **当** 普通用户同时拥有 `本部门数据` 和 `仅本人数据` 两个启用角色时
- **则** 系统将有效数据范围解析为本部门数据
- **且** 不使用更窄的仅本人范围覆盖本部门范围

#### Scenario: 任一启用角色拥有全部数据

- **当** 普通用户任一启用角色的 `dataScope=1` 时
- **则** 系统将有效数据范围解析为全部数据

#### Scenario: 无用户上下文失败关闭

- **当** 接入数据权限治理的宿主资源操作缺少当前用户上下文时
- **则** 系统拒绝该操作或返回空结果
- **且** 不默认按全部数据范围执行

#### Scenario: 禁用角色不参与数据范围解析

- **当** 用户仅拥有禁用角色或无启用角色时
- **则** 系统不得从这些角色获得数据权限范围
- **且** 接入数据权限治理的宿主资源返回空结果或拒绝写操作

### Requirement: 宿主资源必须通过显式策略接入数据权限

系统 SHALL 要求每个接入数据权限治理的宿主资源声明显式资源策略。资源策略 SHALL 包含稳定资源键、适用表或业务资源、本人范围解析方式、部门范围解析方式和受控操作类型。宿主不得通过列名自动推断资源是否接入数据权限；未声明策略的系统治理接口不得被通用组件自动过滤。

#### Scenario: 声明资源策略后注入过滤条件

- **当** `system.file` 声明本人范围使用 `sys_file.created_by` 且部门范围使用上传人部门集合时
- **则** 文件列表、详情、按 ID 下载和删除操作按该策略注入数据权限边界
- **且** 上传文件 URL 访问 `/api/v1/uploads/*path` 不按该策略要求登录，但必须保留路径规范化和文件元数据记录校验

#### Scenario: 未声明策略的治理接口不自动过滤

- **当** 菜单管理、角色管理、字典管理或配置管理没有声明数据权限资源策略时
- **则** 系统不得因这些表存在 `created_at`、`updated_at` 或其他普通字段而自动套用数据权限过滤
- **且** 这些接口继续由功能权限和系统保护规则治理

#### Scenario: 资源策略覆盖读写操作

- **当** 一个宿主资源声明接入数据权限时
- **则** 该资源的列表、详情、导出、下载、更新、删除、状态变更和执行类操作必须使用同一资源策略
- **且** 批量操作必须在全部目标均通过数据权限校验后才执行

### Requirement: 部门数据范围必须通过组织能力解析

系统 SHALL 通过组织能力接口解析部门数据范围，而不是要求每个宿主业务表都持有 `dept_id` 字段。部门范围下，系统 SHALL 优先使用组织能力提供的数据库侧约束能力，将部门归属判断下推到数据库执行；当具体组织能力无法提供数据库侧约束时，才允许使用可见用户集合兜底，并必须避免在大数据量路径中生成无界长度的用户 ID `IN` 条件。当组织能力不可用时，全部数据范围不受影响；本部门数据范围 SHALL 降级为仅本人数据范围；仅本人数据范围 SHALL 继续按当前用户执行。

#### Scenario: 部门范围解析用户集合

- **当** 用户角色数据范围为本部门数据且访问文件列表时
- **则** 系统优先通过组织能力向文件列表查询注入数据库侧部门归属约束
- **且** 文件列表仅返回上传人与当前用户处于可见部门范围内的文件

#### Scenario: 用户管理部门范围使用数据库侧半连接

- **当** 用户角色数据范围为本部门数据且组织能力可用
- **且** 用户访问用户管理列表、详情、导出或写操作时
- **则** 系统不得先把部门范围内的全部用户 ID 拉回应用层再拼接到 `sys_user.id IN (...)`
- **且** 应通过数据库侧 `EXISTS`、`JOIN` 或等价半连接约束过滤目标用户

#### Scenario: 组织能力不可用时本部门范围降级为本人范围

- **当** 用户角色数据范围为本部门数据
- **且** `org-center` 未安装、未启用或组织能力 provider 不可用时
- **则** 系统按当前用户本人范围过滤列表、详情和导出结果
- **且** 写操作和执行类操作仅允许作用于当前用户本人可见数据

#### Scenario: 组织能力不可用时本人范围仍可执行

- **当** 用户角色数据范围为仅本人数据
- **且** 组织能力 provider 不可用时
- **则** 系统仍按当前用户 ID 过滤其本人资源

### Requirement: 首批宿主模块必须按适用性接入数据权限

系统 SHALL 将有明确用户或部门归属语义的宿主模块纳入首批数据权限治理。首批接入模块包括用户管理、文件管理、用户创建的定时任务、在线用户会话。用户消息模块 SHALL 保持当前用户自隔离语义。系统治理模块 SHALL 不纳入数据权限过滤。

#### Scenario: 用户管理接入数据权限

- **当** 普通用户访问用户管理列表、详情、导出或写操作时
- **则** 系统按用户自身和组织能力解析出的部门用户集合限制目标用户范围

#### Scenario: 文件管理接入数据权限

- **当** 普通用户访问文件列表、详情、批量信息、按 ID 下载或删除时
- **则** 系统按文件上传人和上传人所属部门限制目标文件范围
- **且** 公开的上传文件 URL 访问不作为文件管理权限入口，不得绕过路径规范化、元数据存在性校验或存储后端读取边界

#### Scenario: 用户创建任务接入数据权限

- **当** 普通用户访问定时任务管理中的用户创建任务时
- **则** 系统按 `sys_job.created_by` 和创建人所属部门限制目标任务范围
- **且** `sys_job.is_builtin=1` 的内置任务投影不按数据权限过滤

#### Scenario: 在线用户接入数据权限

- **当** 普通用户查询在线会话或强制下线时
- **则** 系统按会话用户和会话用户所属部门限制目标会话范围

#### Scenario: 系统治理模块不接入数据权限

- **当** 管理员访问菜单、角色基础治理、字典、配置、插件治理、i18n、健康检查、系统信息、公共配置、缓存、锁或集群协调接口时
- **则** 系统不应用角色 `dataScope` 过滤这些治理数据
- **且** 仍按现有功能权限、内置保护和组件一致性规则控制访问

### Requirement: 数据权限必须约束写操作和执行类操作

系统 SHALL 在写操作和执行类操作前校验目标记录是否位于当前调用方的数据权限范围内。目标记录不在范围内时，系统 SHALL 返回结构化业务错误，不得仅依赖前端隐藏按钮，也不得只在列表接口过滤。

#### Scenario: 拒绝更新不可见用户

- **当** 用户角色数据范围为仅本人数据
- **且** 该用户尝试更新其他用户资料或状态时
- **则** 系统拒绝操作并保留目标用户数据不变

#### Scenario: 拒绝删除不可见文件

- **当** 用户角色数据范围为本部门数据
- **且** 请求删除的文件上传人不属于该用户部门范围时
- **则** 系统拒绝删除
- **且** 不删除物理文件或数据库记录

#### Scenario: 批量操作必须全量校验

- **当** 调用方批量删除多个文件或用户
- **且** 任一目标记录不在当前调用方数据权限范围内时
- **则** 整个批量操作被拒绝
- **且** 不对任何目标记录执行部分删除

### Requirement: 数据权限缓存必须保持跨节点一致

系统 SHALL 将角色数据范围、用户角色关系和用户组织归属视为数据权限上下文的权威来源。若实现中缓存数据权限上下文或部门用户集合，系统 MUST 在相关权威数据变更后通过明确作用域的缓存修订号或访问拓扑失效机制使所有节点在允许窗口内丢弃旧快照。无法确认缓存新鲜度且超过故障窗口时，接入数据权限治理的宿主资源 MUST 失败关闭。

系统 SHALL 将有效角色数据范围并入登录令牌访问快照缓存，复用访问拓扑修订号控制新鲜度。接入数据权限治理的宿主资源在请求热路径中 SHOULD 从该快照读取有效数据范围，不应在每次判断数据权限时重新查询用户角色关系和角色表。部门范围的具体组织归属约束不得作为无界用户 ID 集合长期缓存；应优先通过组织能力下推到数据库侧执行。

#### Scenario: 角色数据范围变更后失效

- **当** 管理员修改角色 `dataScope` 时
- **则** 系统发布访问拓扑或数据权限缓存修订号
- **且** 后续请求不得无限期使用旧的数据权限范围

#### Scenario: 请求热路径复用 token 数据权限快照

- **当** 用户已经通过受保护 API 的权限校验并建立当前 token 的访问快照
- **且** 该用户后续访问接入数据权限治理的列表、详情或写操作接口
- **则** 系统从 token 访问快照读取有效角色数据范围
- **且** 不为每一次数据权限判断重复查询 `sys_user_role` 和 `sys_role`

#### Scenario: 用户角色关系变更后失效

- **当** 管理员为用户分配或撤销角色时
- **则** 系统发布访问拓扑或数据权限缓存修订号
- **且** 受影响用户后续请求按新的角色数据范围执行

#### Scenario: 组织归属变更后失效

- **当** 组织能力 provider 更新用户部门归属时
- **则** 系统必须使依赖该归属的数据权限缓存失效
- **且** 部门数据范围查询不得无限期使用旧的部门用户集合

### Requirement: 数据权限错误必须本地化

系统 SHALL 对进入 HTTP API 响应的数据权限错误使用 `bizerr` 封装，并为所有启用运行时语言提供错误翻译。新增或修改用户可见错误文案时，必须同步维护运行时 i18n JSON。若修改 API DTO 文档元数据，还必须同步维护非英文 apidoc i18n 资源和翻译完整性校验。

#### Scenario: 数据不可见错误返回结构化响应

- **当** 用户请求不在其数据权限范围内的记录详情时
- **则** 系统返回结构化业务错误
- **且** 响应包含稳定错误码、messageKey、messageParams 和英文 fallback

#### Scenario: 运行时翻译完整

- **当** 数据权限错误在 `zh-CN`、`en-US` 或 `zh-TW` 语言环境下返回时
- **则** 响应消息使用对应语言翻译
- **且** 不暴露未翻译的 i18n key
