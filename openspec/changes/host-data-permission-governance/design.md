## Context

宿主角色管理已经在 `sys_role.data_scope` 中保存三种数据范围：

- `1`：全部数据
- `2`：本部门数据
- `3`：仅本人数据

当前宿主静态 API 主要通过 `g.Meta permission` 做菜单 / 按钮权限校验。用户访问上下文包含角色、菜单、权限码和超级管理员标识，但没有稳定的数据权限快照。插件资源层已经具备按插件资源声明注入 `userColumn` / `deptColumn` 的实现，宿主内置业务接口尚未形成同等治理能力。

数据权限不能无差别套到所有宿主表。很多宿主表是系统治理数据，例如菜单、角色、字典、配置、插件状态、缓存修订号和锁，它们没有“本人”或“本部门”的业务归属语义；强行过滤会造成管理员看不到基础治理数据、无法修复配置或破坏框架管理能力。

## Goals / Non-Goals

**Goals:**

- 建立宿主统一数据权限组件，集中解析当前调用方的有效数据范围。
- 为适合数据权限治理的宿主资源声明清晰策略，避免每个 service 手写重复规则。
- 首批覆盖用户、文件、用户创建任务、在线会话等有明确用户或部门归属的数据。
- 保证列表、详情、导出、下载、更新、删除、状态变更、执行类操作使用同一数据边界。
- 复用现有 `orgcap` 可选组织能力；组织模块禁用时安全降级。
- 明确记录哪些宿主模块不适合数据权限治理，避免误改系统治理接口。
- 明确 i18n 和缓存一致性影响。

**Non-Goals:**

- 不新增自定义数据范围、部门及以下、跨部门组合等复杂范围。
- 不把菜单、角色基础治理、字典、配置、插件治理、i18n、健康检查、系统信息等全局治理数据按部门或本人过滤。
- 不新增复杂的数据范围配置交互；但当组织管理能力不可用时，角色表单需要隐藏或禁用“本部门数据”选项，避免配置无法生效的数据范围。
- 不在本变更中重做租户隔离；数据权限只解决同租户或单租户内的数据可见范围。
- 不改变插件资源数据权限现有清单声明模型；仅在宿主实现中复用或抽取通用能力。

## Decisions

### 决策一：新增宿主统一数据权限服务，而不是把逻辑塞进 HTTP 权限中间件

数据权限需要按资源表、字段和业务关系注入查询条件，也需要在写操作前检查目标记录归属。HTTP 权限中间件只知道路由和 `permission` 标签，不知道业务查询模型、目标表和关联关系。

因此新增内部服务，例如 `internal/service/dataperm`，负责：

- 从当前 `context.Context` 和 `bizctx` 获取用户身份。
- 解析用户角色的有效 `dataScope`。
- 合并多角色范围：`全部 > 本部门 > 仅本人 > 无权限`。
- 判断超级管理员绕过。
- 根据资源策略生成可复用的约束结果。
- 提供 `ApplyListScope(ctx, model, resourceKey)`、`EnsureRecordVisible(ctx, resourceKey, id)` 或等价窄接口。

替代方案是每个 service 自行读取角色并拼接条件。该方案实现快但分散，后续很难审计是否漏掉详情、导出或删除路径，因此不采用。

### 决策二：采用显式资源策略注册，不做基于列名的自动推断

每个可治理资源必须显式声明策略，例如：

| resourceKey | 资源 | 本人策略 | 部门策略 | 说明 |
| --- | --- | --- | --- | --- |
| `system.user` | `sys_user` | `sys_user.id = currentUserID` | 通过组织能力注入数据库侧半连接，兜底才解析部门用户集合 | 用户表没有 `dept_id` 列 |
| `system.file` | `sys_file` | `created_by = currentUserID` | 上传人属于当前部门用户集合 | 表中只有上传人，没有部门列 |
| `system.job` | `sys_job` | `created_by = currentUserID` 且 `is_builtin=0` | 创建人属于当前部门用户集合且 `is_builtin=0` | 内置任务投影不按数据权限过滤 |
| `system.online-session` | `sys_online_session` | `user_id = currentUserID` | 会话用户属于当前部门用户集合 | 强制下线也要校验 |
| `system.user-message` | 用户消息 | 永远当前用户 | 永远当前用户 | 自隔离，不因角色范围放宽 |

不采用列名自动推断（如看到 `created_by` 就自动过滤），因为不同表的 `created_by` 可能是审计字段、投影字段或系统种子字段，语义不一定等价于业务归属。

### 决策三：首批模块按适用性分层推进

#### A. 首批接入数据权限的宿主业务模块

| 模块 | 适用原因 | 过滤依据 | 受控操作 |
| --- | --- | --- | --- |
| 用户管理 | 用户天然具有组织归属和本人身份 | `sys_user.id` + `orgcap` 部门用户集合 | 列表、详情、导出、更新、删除、批量删除、状态、重置密码、角色授权用户列表 |
| 文件管理 | 文件有上传人 `created_by` | `sys_file.created_by` + 上传人部门集合 | 列表、详情、批量信息、按 ID 下载、删除、后缀/场景聚合；上传文件 URL 访问 `/api/v1/uploads/*path` 暂时公开，仅保留路径和元数据校验 |
| 定时任务管理 | 用户创建任务有创建人 `created_by` | `sys_job.created_by` + 创建人部门集合 | 用户任务列表、详情、编辑、删除、启停、触发、日志查询、日志终止 |
| 在线用户 | 会话有 `user_id` | `sys_online_session.user_id` + 用户部门集合 | 在线列表、强制下线 |

#### B. 已经具备更强自隔离语义的模块

| 模块 | 现有边界 | 本次要求 |
| --- | --- | --- |
| 用户消息 | 只查询和变更当前用户自己的消息 | 保持当前用户边界，不允许 `全部数据权限` 放宽 |
| 当前用户资料 | 只读写当前登录用户 | 保持自隔离，不纳入角色数据范围 |

#### C. 不适合按数据权限过滤的系统治理模块

| 模块 | 不纳入原因 | 治理方式 |
| --- | --- | --- |
| 菜单管理 | 全局权限拓扑数据，没有部门 / 本人归属 | 功能权限 + 内置保护 |
| 角色管理基础 CRUD | 角色是授权治理数据，过滤会破坏授权管理 | 功能权限 + 内置角色保护；`dataScope` 只作为后续业务数据范围输入 |
| 字典管理 | 字典是跨模块枚举基础设施 | 功能权限 + 内置字典删除保护 |
| 配置管理 | 配置是系统运行参数 | 功能权限 + 内置配置删除保护 + 配置缓存一致性 |
| 插件治理 | 插件状态和资源引用是全局运行时治理 | 功能权限 + 插件生命周期保护 |
| i18n / apidoc | 翻译和文档资源是全局交付资源 | 功能权限 + i18n 缓存作用域 |
| 健康检查 / 系统信息 / 公共配置 | 不是业务数据归属模型 | 现有公开或受保护读取规则 |
| 缓存 / 锁 / 集群修订号 | 基础设施状态，不是用户业务数据 | 组件内部一致性与权限边界 |

#### D. 后续评估模块

通知渠道、通知消息等当前更多服务于插件和系统消息投递，表结构同时包含 `sender_user_id` 和 `user_id`。本次只约束用户消息自隔离，不承诺对通知治理后台做数据权限过滤；后续如新增通知管理页面，再按通知业务归属单独设计。

### 决策四：部门范围通过 `orgcap` 下推数据库约束，组织能力缺失时降级为本人

宿主核心用户表没有 `dept_id`，部门关系由可选组织能力提供。因此部门范围不能直接拼接 `dept_id = ?`，必须通过 `orgcap` 解析：

- 当前用户的部门集合。
- 优先将目标资源的归属用户 ID 与组织关系表通过数据库侧 `EXISTS`、`JOIN` 或等价半连接关联。
- 对无法提供数据库侧约束的 provider，才允许返回可见用户 ID 集合兜底；此兜底不得成为大数据量路径的默认实现。

当 `org-center` 未安装、未启用或 `orgcap` provider 不可用时：

- `全部数据权限` 不依赖组织能力，仍正常工作。
- `本部门数据权限` 无法解析部门边界，按 `仅本人数据权限` 降级执行。
- `仅本人数据权限` 不依赖组织能力，仍按当前用户过滤。

这样比“组织缺失时退化为全部数据”更安全，也比直接返回空结果更符合未启用组织模块时的最小可用体验。前端角色管理同步隐藏或禁用“本部门数据”选项，减少不可生效配置。

### 决策五：读写边界统一，禁止只过滤列表

每个接入资源必须覆盖：

- 列表查询和树 / 选项类查询。
- 详情查询。
- 导出、按 ID 下载、批量信息等读取数据的路径。
- 更新、删除、状态切换、重置密码、强制下线、触发任务、终止任务等写操作或执行类操作。

写操作必须先验证目标记录在当前调用方数据范围内。批量操作必须对全部目标校验通过后再执行，不允许部分成功。

### 决策六：数据权限变更复用访问拓扑缓存失效，并保留后续独立域扩展点

角色 `dataScope`、用户角色关系和组织归属会影响数据权限结果。当前已有权限拓扑缓存失效机制，角色和用户角色写路径已触发访问拓扑变化。有效角色数据权限应并入登录令牌访问快照，与角色、菜单和按钮权限共用 `permission-access` 修订号，避免每次数据权限判断重复查询 `sys_user_role` 和 `sys_role`。

本轮缓存只保存“有效数据范围类型”（全部 / 本部门 / 仅本人 / 无权限）和异常状态，不缓存部门用户集合。部门数据范围仍通过组织能力在数据库侧注入半连接或等价约束，避免大部门用户集合长期滞留在进程内存中。动态插件路由和宿主数据服务也应沿用同一身份快照传递有效数据范围，避免插件数据访问路径重新扫描角色表。

如果后续出现独立数据权限缓存（例如按用户缓存部门用户集合），必须声明独立缓存域和显式 scope：

- 权威数据源：`sys_role.data_scope`、`sys_user_role`、组织能力 provider 的用户部门关系。
- 一致性模型：写后在单节点立即失效，集群模式下通过缓存协调修订号在允许窗口内失效。
- 最大可接受陈旧时间：沿用访问拓扑缓存域的窗口；若新增独立域，必须在实现中明确配置。
- 降级策略：无法确认新鲜度且超过故障窗口时失败关闭。

### 决策七：i18n 影响保持最小但必须显式审查

角色表单不新增文案，因此前端运行时语言包通常不需要变化。若新增用户可见业务错误，例如“目标数据不在当前数据权限范围内”，必须：

- 在模块 `*_code.go` 中定义 `bizerr.Code`。
- 为 `zh-CN`、`en-US`、`zh-TW` 补充运行时错误翻译。
- 若修改 API DTO 文档，补充 apidoc 非英文资源和翻译完整性校验。

## Risks / Trade-offs

- [Risk] 数据权限遗漏详情、导出或写操作，造成“列表看不到但能操作”。
  Mitigation：资源接入任务按操作类型逐项列出，测试覆盖列表、详情、导出/下载和写路径。

- [Risk] 部门范围需要关联组织关系，用户量较大时应用层 `WHERE IN` 用户集合会变长。
  Mitigation：用户管理首期即改为数据库侧半连接；组织关系表补充覆盖索引。后续模块接入时优先复用数据库侧约束，不默认拉取大批用户 ID。

- [Risk] 把系统治理数据错误纳入数据权限导致管理员无法维护系统。
  Mitigation：设计中显式列出不纳入模块，规范要求未声明资源策略的宿主治理接口不得自动套用数据权限。

- [Risk] 组织能力禁用时本部门数据权限无法表达部门边界。
  Mitigation：后端按本人范围安全降级；角色新增和编辑表单隐藏或禁用本部门选项，避免继续产生不可生效配置。

- [Risk] 多角色用户的数据范围合并不符合预期。
  Mitigation：采用最宽范围合并并写入规范；测试覆盖 `全部 + 本人`、`本部门 + 本人` 等组合。

- [Risk] 缓存失效未覆盖组织归属变化。  
  Mitigation：组织能力 provider 若维护用户部门关系，必须在关系变更后触发数据权限相关缓存失效；首期没有独立缓存时避免长期缓存部门用户集合。

## Migration Plan

1. 创建宿主数据权限服务与资源策略注册，不改变现有业务行为。
2. 接入首批模块的读路径，先覆盖列表、详情、导出、按 ID 下载。
3. 接入首批模块的写路径和执行类操作。
4. 补充后端单元 / 集成测试与必要 E2E。
5. 运行 `make init`、后端测试和受影响 E2E。
6. 若发现某模块业务归属语义不清晰，保持未接入并在设计中追加“后续评估”记录。

回滚策略：数据权限服务以内部调用方式接入，不新增破坏性数据库结构；回滚时移除接入调用即可恢复原行为。若新增 SQL 索引或字段，必须确保初始化可重建。

## Open Questions

- `本部门数据权限` 是否仅包含当前用户直接所在部门，还是应包含子部门？当前角色文案是“本部门”，本设计按直接部门理解；若要包含子部门，需要新增角色数据范围值或调整文案。
- 用户管理中的“重置密码 / 禁用用户 / 删除用户”是否应允许本部门管理员操作同部门所有用户，还是仍需要额外防止同级管理员互相操作？本变更只定义数据可见范围，管理员层级规则可后续单独治理。
- 文件 `/uploads/*path` 访问已明确作为公开浏览器访问入口，必须稳定从 path 反查 `sys_file` 记录并拒绝裸路径或路径穿越；后续如需权限控制，再单独设计签名 URL 或带鉴权的前端 blob 展示方案。
