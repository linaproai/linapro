## Context

当前宿主数据库初始化入口分散在两层：仓库根`Makefile`暴露`make init`、`make mock`，`apps/lina-core/Makefile`进一步转发到`go run main.go init`、`go run main.go mock`。命令实现本身会扫描宿主和插件目录下的`SQL`文件并直接执行，没有显式确认护栏。

这两个命令都属于高风险写库操作，但它们又必须保留非交互执行能力，因为本地初始化、自动化脚本和测试回归都依赖可脚本化的命令入口。因此，本次设计需要在“默认安全”和“自动化可用”之间取得平衡：默认拒绝执行；只有操作者显式声明本次动作时才继续。

本次变更不涉及数据库表结构、配置文件格式或外部依赖变更，主要影响`Makefile`入口、`GoFrame`命令入参解析、`SQL`执行错误处理和相关测试。

## Goals / Non-Goals

**Goals:**
- 为`init`和`mock`建立默认拒绝、显式确认后才执行的安全模型。
- 保证从根目录`make`入口到后端命令入口都遵循相同的确认语义，避免存在易绕过的默认危险路径。
- 将命令执行语义从“尽力执行”改为“失败可见”，确保任一`SQL`执行失败时命令返回失败状态。
- 保持命令非交互、可脚本化，便于本地开发和自动化流程显式调用。
- 通过自动化测试锁定安全护栏和失败语义，避免后续回归。

**Non-Goals:**
- 本次不引入交互式确认提示或二次输入对话。
- 本次不增加`dry-run`、环境白名单或数据库目标识别等增强能力。
- 本次不改变`init`、`mock`扫描`SQL`目录的既有约定。
- 本次不尝试把所有`SQL`文件包裹为跨文件事务执行。

## Decisions

### 决策一：使用非交互的显式确认值，而不是布尔开关或交互式确认

**决定**：
- `init`命令必须要求传入`confirm=init`或`--confirm=init`。
- `mock`命令必须要求传入`confirm=mock`或`--confirm=mock`。
- 仅接受长参数`confirm`，不提供简写别名。

**原因**：
- 布尔开关如`--yes`、`--confirm`过于宽松，被复制到脚本后容易长期存在，不能有效表达“本次明确知道自己在执行什么动作”。
- 交互式提示虽然更强，但会破坏自动化和无终端场景，不适合作为基础命令默认行为。
- 以命令名作为确认值，能把确认动作和实际危险命令绑定，降低复制粘贴时误用到另一条命令上的概率。
- 不提供短参数，可以减少误敲和阅读歧义。

**备选方案**：
- 使用通用布尔开关`--yes`：放弃，安全强度不足。
- 使用交互式二次确认：放弃，会破坏脚本化调用。
- 使用统一确认值如`confirm=true`：放弃，无法体现具体动作语义。

### 决策二：在`Makefile`和命令实现两层同时校验确认参数

**决定**：
- 仓库根`Makefile`和`apps/lina-core/Makefile`都要求显式传入`confirm`变量；缺失或错误时直接打印用法并失败退出。
- 后端`GoFrame`命令实现继续做最终校验；即使绕过`make`直接执行`go run main.go init/mock`，未确认也不得执行任何`SQL`。

**原因**：
- 仅在`Makefile`拦截，无法防止直接调用`go run main.go init`。
- 仅在命令层拦截，虽然能保证最终安全，但常用的`make`入口仍缺乏即时提示，体验较差。
- 双层校验同时兼顾可发现性和兜底安全。

**备选方案**：
- 只改`Makefile`：放弃，存在直接调用旁路。
- 只改命令实现：放弃，不能在最常用入口给出更早反馈。

### 决策三：`SQL`执行改为首错即停，并向调用方返回失败

**决定**：
- 公共`SQL`执行逻辑改为返回`error`。
- 任一`SQL`文件执行失败时，立即停止后续文件执行，并让命令以失败状态退出。
- 保留成功执行到的文件日志和失败文件日志，便于定位问题。

**原因**：
- 当前仅记录告警后继续执行，会产生“部分成功、部分失败”的隐蔽状态，尤其对初始化类命令风险较高。
- 首错即停能减少错误继续扩散，并让调用方及时感知失败。
- 不引入跨文件事务可以保持实现简单，并兼容不同数据库对`DDL`事务支持不一致的现实。

**备选方案**：
- 继续告警后执行剩余文件：放弃，失败不可见。
- 将所有文件放进一个大事务：放弃，`DDL`与跨文件执行在不同数据库上的事务语义并不稳定。

### 决策四：将帮助信息和错误提示视为安全设计的一部分

**决定**：
- `Makefile`失败提示和命令拒绝执行提示都必须给出正确示例，例如`make init confirm=init`、`go run main.go mock --confirm=mock`。
- 命令的`brief`或帮助输出应体现“需要显式确认”的使用预期。

**原因**：
- 安全护栏若只有拒绝逻辑而没有清晰提示，会增加无谓摩擦，促使使用者寻找绕过方式。
- 明确示例有助于脚本维护者快速迁移已有调用。

**备选方案**：
- 只返回“参数错误”：放弃，可用性差。

## Risks / Trade-offs

- **[命令更啰嗦]** 使用者需要额外输入确认参数，短期内会觉得麻烦 → **Mitigation**：统一确认语义，报错直接提供可复制的正确示例。
- **[旧脚本失效]** 现有未带确认参数的脚本、文档或个人习惯会立即失效 → **Mitigation**：在`Makefile`和命令帮助中输出迁移方式，并补充测试覆盖新的标准调用。
- **[仍可能出现部分写入]** 若前几个`SQL`已成功，后续文件失败，数据库仍可能处于部分变更状态 → **Mitigation**：采用首错即停、返回非零、清晰日志的策略，将损害限制在最小范围，并避免静默继续。
- **[两层校验需保持一致]** 若`Makefile`和命令层示例或确认值不一致，会造成困惑 → **Mitigation**：将确认值设计为与命令名一一对应，并在测试中同时覆盖两层入口文案与行为。

## Migration Plan

1. 为根目录与`apps/lina-core`目录下的`make init`、`make mock`增加`confirm`校验和示例提示。
2. 为后端`init`、`mock`命令增加`confirm`入参校验，并在未确认时拒绝执行任何`SQL`扫描结果。
3. 调整公共`SQL`执行逻辑为返回`error`并首错即停，确保命令失败状态可传递到调用方。
4. 补充命令与`Makefile`相关测试，验证未确认拒绝、确认后可执行、失败时返回非零。
5. 若需要回滚，只需撤回确认校验与错误处理改动；本次不涉及数据库迁移，无额外数据回滚步骤。
