## Context

`media_v2.md` 只提供 MySQL DDL，未给出接口或页面说明。根据表名和字段语义，本次将其收敛为一个媒体治理源码插件：维护策略文本、设置策略作用域绑定，并维护流别名到真实流路径的映射。

项目当前是新项目，数据库初始化允许重建，因此不需要兼容 MySQL 或历史旧表。源码插件按 `apps/lina-plugins/<plugin-id>/` 结构落地，DAO/DO/Entity 由插件本地 `gf gen dao` 生成。

## Goals / Non-Goals

**Goals:**

- 使用 PostgreSQL 原生语法实现插件安装 SQL，替换 MySQL 的 `AUTO_INCREMENT`、`UNSIGNED`、`datetime(3)`、`tinyint(1)` 和表级注释写法。
- 以源码插件形式提供媒体策略、策略绑定、策略解析和流别名管理。
- 后端接口遵循 RESTful：查询用 `GET`，创建用 `POST`，更新/状态切换用 `PUT`，删除用 `DELETE`。
- 前端页面使用中文文案直接展示，不接入本模块 i18n 资源。
- 所有策略生效判断以数据库为权威数据源，不增加缓存一致性负担。

**Non-Goals:**

- 不实现真实媒体服务器推流、拉流、转码或设备接入协议。
- 不解析或校验 YAML 策略内容的业务字段，仅保存文本并做非空校验。
- 不把 `media_v2.md` 的租户字符串强行映射为 LinaPro 宿主多租户主体 ID。
- 不新增运行时 i18n JSON、manifest i18n 或 apidoc i18n 资源。

## Decisions

1. 插件 ID 使用 `media`，源码插件作用域使用 `platform_only`。

   理由：用户明确反馈 media 配置全平台共享，不做 LinaPro 宿主租户隔离。表结构中的 `tenant_id` 仅表示媒体租户策略维度，不作为宿主多租户隔离边界。

2. PostgreSQL 表名保持 `media_*`。

   理由：用户明确指定启动一个 `media` 插件，原始表结构也已经使用该前缀。备选方案是改为 `plugin_media_*`，但会偏离用户提供的业务表名。

3. 策略优先级定义为：租户设备策略 > 设备策略 > 租户策略 > 全局策略。

   理由：更具体的绑定应覆盖更宽泛的绑定。该优先级能从四张绑定表自然推导，并让解析接口有确定结果。

4. 全局策略通过 PostgreSQL 部分唯一索引保证最多一个启用。

   理由：MySQL 表注释说明“只能有一个是 1”。部分唯一索引 `WHERE "global" = 1` 可以直接表达该约束；服务层在设置全局策略时先关闭其他策略再开启目标策略。

5. `global`、`enable`、`auto_remove` 保持数值字段，不转为 PostgreSQL 布尔。

   理由：原策略表使用 `1/2` 表达开关枚举，流别名表使用 `tinyint(1) DEFAULT 0` 表达自动移除的 `1/0` 是/否标记。服务层用 Go 命名类型和常量管理，避免硬编码枚举字符串。

6. 本模块中文-only，不创建 i18n 资源。

   理由：用户明确要求“这个模块不用 i18n，直接全部中文即可”。因此新增功能的 i18n 影响面结论是：本模块所有用户可见文案直接使用中文，插件不提供运行时语言包、manifest i18n 或 apidoc i18n JSON。

7. 不新增缓存。

   理由：策略和别名管理属于低频治理操作，读写均可直接访问 PostgreSQL。权威数据源是数据库；一致性模型为读已提交下的数据库一致性；不存在跨实例本地缓存失效问题。

8. 数据可见性边界由源码插件菜单权限控制，不额外接入宿主租户隔离或组织/部门角色数据范围过滤。

   理由：`media_v2.md` 的表结构只有媒体租户字符串、设备国标 ID、策略和别名字段，没有部门、岗位、用户归属或组织树维度。用户要求 media 配置全平台共享，因此插件表不增加 `host_tenant_id`，服务层不通过 `tenantfilter.Current(ctx)` 注入宿主租户过滤。若后续接入设备台账、组织归属或需要不同平台租户独立配置，应在该迭代中重新评估数据权限和租户隔离。

9. 策略绑定拆分为设备绑定、租户绑定、租户设备绑定三个独立资源。

   理由：三类绑定对应不同自然键和业务含义，页面与接口都不应通过同一个 `scope` 参数混合处理。后端分别暴露 `/media/device-bindings`、`/media/tenant-bindings`、`/media/tenant-device-bindings` 资源；前端分别呈现独立页签和操作入口。策略解析内部仍按既定优先级读取三张绑定表，但解析接口不改变绑定资源边界。

## Risks / Trade-offs

- 原始表缺少完整业务说明 → 以表结构推导最小可用管理逻辑，避免引入媒体协议或 YAML 解析等猜测性能力。
- `global` 是 SQL 常见词 → 所有 SQL 字段和服务查询使用双引号/生成 DAO 列名，避免保留字风险。
- 租户字符串不是宿主租户 ID → 字段命名为 `tenantId`/`tenant_id` 租户标识，API 文档明确其来源由调用方传入，并且不引入 `host_tenant_id`。
- 中文-only 会降低多语言能力 → 本模块是用户显式要求的例外，OpenSpec 和任务中记录该 i18n 决策。

## Migration Plan

1. 新增插件目录和 PostgreSQL 安装 SQL。
2. 执行插件 SQL 初始化本地数据库后运行 `gf gen dao` 生成插件本地 DAO/DO/Entity。
3. 实现后端 API、服务和前端管理页。
4. 运行 Go 单测、前端类型检查、E2E 静态校验和 `openspec validate`。

回滚时卸载或删除 `apps/lina-plugins/media` 接线，并执行插件卸载 SQL 删除插件自有表。
