## Context

本次整改来源于一次全仓库级别的规范审查。审查结果显示，当前项目存在以下跨模块问题：

- OpenSpec 主规范仍混用旧结构，部分文件缺少 `## Purpose` / `## Requirements`，归档过程中会触发兼容性问题
- 后端生产代码中存在手写软删除过滤、API DTO 文档标签不完整、导出符号注释缺失、路径参数风格混用等问题
- 控制器和服务层虽然大体遵循 GoFrame 结构，但缺少稳定的合规基线和批量整改顺序
- 模块可禁用/可降级规范尚未形成可执行实现，当前模块间依赖仍以直接强耦合为主

这是一次典型的横切面治理工作，涉及 OpenSpec、后端 API、服务实现和架构约束，适合先形成统一设计，再按批次执行。

## Goals / Non-Goals

**Goals:**

- 让 OpenSpec 主规范恢复到可校验、可归档、可持续维护的状态
- 让后端生产代码满足 GoFrame v2 技能要求与仓库既有规范
- 统一 API 合同定义方式，降低前后端联调和文档维护成本
- 为模块可禁用能力建立明确的后端实现边界和降级约束
- 为后续迭代提供可重复执行的规范校验流程

**Non-Goals:**

- 不在本次迭代中进行无关业务功能扩展
- 不手工修改 DAO/DO/Entity 等自动生成文件
- 不做一次性的大规模前端视觉或交互改版
- 不以“重写全部历史模块”为目标，而是按风险和收益分批整改

## Decisions

### 1. 先修 OpenSpec 主规范可维护性，再推进代码层整改

先处理主规范结构和归档稳定性问题，再进行代码整改。原因是当前规范库已直接影响变更归档，一旦不先恢复规范系统稳定性，后续整改本身也会难以归档。

备选方案：
- 直接先改代码，再最后统一修规范。放弃原因是归档链路仍然不稳定，整改结果无法顺畅沉淀。

### 2. 后端规范整改按“正确性优先，其次一致性，最后文档补齐”的顺序执行

优先级顺序如下：

- 第一批：修复 ORM/软删除/事务/依赖注入等会影响行为正确性的实现问题
- 第二批：统一 REST 路径、参数绑定、返回结构和模块降级行为
- 第三批：补齐导出符号注释、`dc`/`eg` 标签和历史文档债

备选方案：
- 先批量补注释和标签。放弃原因是会制造大量噪音 diff，但对运行正确性帮助有限。

### 3. 路径参数定义统一采用 `{param}` + `p:"param"` 组合

`g.Meta` 中统一使用 `{id}` 形式声明路径参数；DTO 字段统一用 `p:"id"` 绑定路径或查询参数，避免同一仓库混用 `:id`、`json:"id"`、`p:"id"` 多种方式。

备选方案：
- 保持现状，按模块各自约定。放弃原因是 Swagger 文档、控制器实现和前端调用都更难保持一致。

### 4. 模块可禁用能力采用“集中配置 + 服务层 graceful degradation”方式落地

模块启用状态由集中配置管理；依赖模块的服务在读取扩展数据或执行关联逻辑时需要先判断模块状态，并在模块关闭时返回零值、空集合或跳过关联逻辑，而不是报错。

备选方案：
- 只在前端隐藏入口，不处理后端。放弃原因是后端依赖仍会在接口聚合或任务执行中报错，不满足项目规范。

### 5. 验证采用“静态检查 + 单测 + 相关 E2E”组合，而不是先引入重型新工具

本次先依赖现有 `go test`、OpenSpec 校验、针对性 E2E 以及必要的脚本化检查，不额外引入新的复杂 lint 平台。等规范基线稳定后，再评估是否固化为 CI 规则。

## Risks / Trade-offs

- [接口路径标准化可能影响前端和测试调用] -> 优先评估是否保留兼容路径；若必须变更，需同步更新前端与 E2E
- [主规范清理会触碰历史文件，diff 面积较大] -> 按 capability 分批修正，每完成一批即运行 `openspec validate`
- [注释和标签补齐会带来大量机械改动] -> 严格限定范围，优先修改当前会影响文档或归档质量的文件
- [模块解耦可能暴露隐藏依赖] -> 先补充开关探测和 graceful degradation，再逐步拆硬依赖

## Migration Plan

1. 修复主规范结构和归档兼容问题，确保后续变更可正常校验与归档
2. 按模块批次整改后端代码中的 GoFrame 和项目规范违规项
3. 对涉及接口合同变化的模块同步调整前端调用与 E2E 用例
4. 执行 `openspec validate`、`go test ./...` 以及相关 E2E，确认整改未引入回归
5. 完成整改后执行 `openspec-review`，再进入归档流程

## Open Questions

- 对已有前端依赖的历史路径，是否统一保留一段时间的兼容别名
- 模块启用状态应放在现有 `sys_config` 体系中，还是补充独立的模块配置读取组件
- 测试代码中使用 `g.Map` 是否允许保留，还是也纳入本轮一致性整改范围
