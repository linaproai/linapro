## Context

默认管理工作台的列表页普遍通过`useVbenVxeGrid`接入搜索表单。当前表格封装在创建表单时默认设置`showCollapseButton: true`，表单操作区只根据该布尔值渲染展开/折叠按钮。结果是一行搜索条件页面也会出现展开/折叠入口，但点击后没有额外条件可展示或隐藏。

现有`form-ui`的`useExpandable`已经根据表单容器、断点、`schema`长度和可见性计算折叠保留项。该位置比单个页面更接近真实布局，因为搜索条件是否"一行"受响应式栅格、标签宽度、动态`schema`、语言和按钮区域共同影响。

本变更属于默认管理工作台前端展示适配，不修改`apps/lina-core`核心领域契约、后端服务语义、存储模型或 API 契约。

## Goals / Non-Goals

**Goals:**

- 当搜索表单实际只占一行时，隐藏右侧展开/折叠按钮。
- 当搜索表单存在第二行及以上条件时，继续显示展开/折叠按钮，并保持现有折叠行为。
- 行数判断随断点、搜索表单可见性、动态`schema`和字段布局变化更新。
- 保持搜索、重置、搜索面板显示隐藏、表格刷新和插件工具栏插槽语义不变。

**Non-Goals:**

- 不逐个修改默认工作台列表页的搜索配置。
- 不新增后端接口、路由、DTO、权限、数据库结构、缓存或插件运行时能力。
- 不新增运行时文案或语言包键。
- 不重新设计表单栅格、按钮顺序或列表页整体视觉风格。

## Decisions

### Decision 1: 在公共表单渲染层派生按钮可见性

实现应在`form-ui`的行数计算链路中派生"是否存在可折叠内容"，再传递给操作区决定是否渲染展开/折叠按钮。

理由：

- 真实行数依赖 DOM 布局，页面配置中的字段数量无法可靠代表是否一行。
- 公共表单层已经持有`wrapperRef`、断点、`schema`和可见性状态，可复用现有计算入口。
- 所有宿主列表和使用宿主`useVbenVxeGrid`的源码插件页面都能获得一致行为。

替代方案：

- 逐页设置`showCollapseButton: false`：短期文件少，但无法处理响应式换行和动态`schema`，也会造成重复配置。
- 在`useVbenVxeGrid`里按`schema.length`推断：实现简单，但按钮占位、`wrapperClass`、英文标签和自定义字段宽度都会让字段数与实际行数失配。

### Decision 2: "是否多行"以折叠后是否会隐藏字段为准

按钮可见性应与折叠行为一致：仅当当前布局下存在超出`collapsedRows`的字段时显示按钮。若折叠不会隐藏任何字段，即使`showCollapseButton`配置为`true`，也不渲染展开/折叠按钮。

理由：

- 用户需要的是有可执行收益的控件，而不是固定占位。
- 该定义天然支持`collapsedRows`未来扩展，不把规则写死为"字段数大于列数"。

替代方案：

- 固定判断"第二行存在任意元素"：与默认`collapsedRows=1`一致，但无法覆盖调用方配置更多保留行的情况。

### Decision 3: 保留现有文案和动作入口

实现不新增按钮文案。多行场景继续使用现有`expand`和`collapse`文案，一行场景直接不展示该控件。

理由：

- 变更只调整显示条件，不改变文案语义。
- 避免新增`i18n`资源维护和翻译校验成本。

## Risks / Trade-offs

- [Risk] DOM 行数计算在表单初次不可见或搜索面板被隐藏时不准确。→ Mitigation：沿用并强化现有可见性、断点和`schema`监听，在表单可见后重新计算。
- [Risk] 一行页面隐藏按钮后，操作区宽度和按钮位置轻微变化。→ Mitigation：E2E 覆盖典型一行页面，验证搜索、重置按钮仍在同一行且无重叠。
- [Risk] 多行页面误判为一行会导致用户无法访问折叠后的条件。→ Mitigation：E2E 覆盖典型多行页面，断言展开/折叠按钮仍可见且可展开额外条件。
- [Risk] 源码插件页面通过宿主表格封装继承行为变化。→ Mitigation：该行为属于宿主表格搜索区通用体验；不修改插件目录和插件资源契约，插件如需强制自定义仍可通过现有表单配置和插槽路径处理。
