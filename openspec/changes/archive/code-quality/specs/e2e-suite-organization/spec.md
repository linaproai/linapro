## ADDED Requirements

### Requirement: E2E 质量审查必须验证测试结果有效性
E2E 测试套件治理 SHALL 在审查用户可观察行为、端到端工作流、E2E 测试文件、E2E 辅助资产或功能行为类反馈修复时执行质量审查。质量审查 MUST 判断测试是否能够稳定证明本次业务行为、数据状态、权限结果或用户可观察状态变化正确。质量审查 MUST 只约束测试结果和审查证据，不得在项目规则中强制具体定位器类型、封装模式或编码实现方式。

#### Scenario: 用户可观察行为变更触发质量审查
- **WHEN** 变更涉及用户可观察页面、路由、表单、表格、权限交互、接口联动或端到端工作流
- **THEN** 审查必须执行 E2E 质量审查
- **AND** 审查必须判断本次变更涉及的核心用户路径是否具备有效 E2E 或等价自动化验证

#### Scenario: E2E 测试资产变更触发质量审查
- **WHEN** 变更新增或修改 E2E 测试、E2E fixture、E2E helper、E2E 页面辅助资产、测试数据或执行 manifest
- **THEN** 审查必须执行 E2E 质量审查
- **AND** 审查必须判断变更后的测试是否仍能独立、稳定、可信地证明目标业务行为

#### Scenario: 功能行为类反馈修复触发质量审查
- **WHEN** 功能行为类 bugfix、用户可观察缺陷或跨模块工作流缺陷通过反馈流程处理
- **THEN** 审查必须确认存在能够复现原问题并验证修复结果的自动化测试
- **AND** 缺少有效自动化测试时必须判定为严重问题

#### Scenario: 项目规则不绑定测试实现方式
- **WHEN** 审查执行 E2E 质量审查
- **THEN** 审查不得仅因测试未使用某种具体定位器、封装模式或代码组织方式而判定违规
- **AND** 审查必须以测试能否证明业务结果、覆盖本次变更范围、保持可信稳定为判断依据

### Requirement: E2E 质量问题必须按验收风险分级
E2E 质量审查 SHALL 按问题对本次变更验收、反馈复现、测试可信度和长期维护风险的影响分级。质量问题默认作为警告报告；当问题导致本次变更缺少有效验收证据、功能行为类 bugfix 缺少复现验证、或测试不可独立可信运行时，MUST 升级为严重问题并阻塞任务完成、反馈完成或归档。

#### Scenario: 覆盖缺口影响本次变更验收
- **WHEN** 本次新增、修改或修复的核心用户路径缺少 E2E 或等价自动化验证
- **THEN** 审查必须将该覆盖缺口判定为严重问题
- **AND** 任务不得在缺少明确豁免依据的情况下标记完成

#### Scenario: 无效断言无法证明业务结果
- **WHEN** E2E 断言只验证页面元素存在、弹窗关闭、按钮可点击或页面跳转
- **AND** 这些断言不能证明本次业务结果、数据状态、权限结果或用户可观察状态变化正确
- **THEN** 审查必须将该断言问题判定为质量问题
- **AND** 若该断言是本次变更的唯一验收依据，则必须升级为严重问题

#### Scenario: 测试稳定性影响可信度
- **WHEN** E2E 测试依赖其他测试用例的执行结果、残留数据、执行顺序或与业务语义无关的偶然结构和状态
- **THEN** 审查必须将该稳定性问题判定为质量问题
- **AND** 若问题导致测试无法独立运行或结果不可信，则必须升级为严重问题

#### Scenario: 非本次范围的扩展覆盖缺口
- **WHEN** 审查发现与本次变更无直接关系的异常路径、边界场景或扩展场景缺少 E2E 覆盖
- **THEN** 审查可以将其作为警告或改进建议报告
- **AND** 该建议不得单独阻塞当前任务完成、反馈完成或归档

### Requirement: E2E 质量审查报告必须说明判断依据
E2E 质量审查 SHALL 在审查报告中说明是否触发质量审查、触发原因、覆盖判断依据、发现问题的严重性和建议补充的业务场景。覆盖判断 MUST 基于本次变更范围和可审查证据，不得仅凭主观推测。

#### Scenario: 报告触发原因和证据来源
- **WHEN** 审查触发 E2E 质量审查
- **THEN** 审查报告必须说明触发原因
- **AND** 审查报告必须说明覆盖判断依据，例如 OpenSpec 任务、增量规范、代码变更、页面行为、接口联动或现有测试证据

#### Scenario: 报告质量问题
- **WHEN** 审查发现 E2E 质量问题
- **THEN** 审查报告必须列出文件路径、行号、严重性和问题原因
- **AND** 覆盖缺口必须描述建议补充的业务场景，不得要求采用某种具体实现方式作为唯一修复方案

#### Scenario: 跳过质量审查
- **WHEN** 审查范围不涉及 E2E 测试资产、用户可观察行为、端到端工作流或功能行为类反馈修复
- **THEN** 审查报告必须记录未触发 E2E 质量审查
- **AND** 审查报告必须说明跳过原因
