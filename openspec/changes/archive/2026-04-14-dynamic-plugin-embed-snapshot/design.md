## Context

当前动态插件的作者体验与宿主治理模型存在一处明显割裂：

- 源码插件通过 `go:embed` 暴露 `plugin.yaml`、`frontend`、`manifest` 等资源，并由宿主直接读取 `fs.FS`；
- 动态插件虽然最终也把这些资源带入 `.wasm` 产物，但构建器当前直接扫描插件目录，再把扫描结果写入 Wasm 自定义节；
- 宿主上传、安装、启用、菜单校验、静态资源托管和动态路由治理都只认产物里的快照自定义节，而不依赖 guest 主动提供资源。

因此，真正需要统一的是“插件作者如何声明资源”，而不是“宿主如何消费资源”。如果把宿主的静态资源读取链路直接改成 guest 方法调用，会把原本可在上传前、启用前完成的静态治理校验，变成对未启用 Wasm 代码的提前执行，破坏当前动态插件的治理边界。

## Goals / Non-Goals

**Goals:**

- 让动态插件作者也能像源码插件一样，通过 `go:embed` 统一声明 `plugin.yaml`、`frontend`、`manifest` 等资源。
- 保留宿主现有快照治理模型：上传、安装、启用、菜单绑定、静态资源托管和动态路由治理继续只消费 Wasm 自定义节。
- 让 `hack/build-wasm` 成为作者侧 `go:embed` 声明与宿主侧快照区段之间的唯一转换层。
- 为现有动态插件提供兼容迁移路径，避免一次性打断目录扫描构建方式。

**Non-Goals:**

- 不把宿主静态资源托管改成“每次请求都调用 guest 方法读取资源”。
- 不要求宿主直接解析 Go 编译后的 `embed.FS` 内部实现。
- 不改变当前动态插件上传接口、固定前缀动态路由治理和 `/plugin-assets/...` 访问模型。
- 不把 `plugin.yaml`、安装 SQL、卸载 SQL 的治理真相从产物快照切回 guest 运行时返回值。

## Decisions

### 决策一：统一作者侧资源声明，但不改变宿主消费模型

动态插件新增根级资源声明文件，例如 `plugin_embed.go`，通过 `//go:embed` 暴露资源文件系统。构建器优先消费该声明的 `fs.FS`，再生成宿主已有的自定义节快照。

保留宿主现有消费模型的原因：

- 上传阶段需要先读取嵌入 manifest，才能确定插件 ID、版本和落盘路径；
- 启用前需要校验菜单引用的托管资源是否真实存在；
- `/plugin-assets/...` 的资源托管需要稳定、可缓存、无需执行 guest 代码；
- 安装/卸载 SQL 属于治理数据，不应依赖 guest 运行时动态计算。

备选方案是“宿主通过 guest ABI 读取资源”，但该方案会把静态校验链路变成代码执行链路，且要求宿主在上传或启用前实例化未启用 Wasm，不符合当前动态插件的安全与治理边界，因此不采用。

### 决策二：构建器负责把 `go:embed` 资源转成宿主快照

`hack/build-wasm` 继续是动态插件构建的唯一入口，但资源读取逻辑从“只扫目录”调整为：

1. 优先查找并加载动态插件导出的嵌入文件系统；
2. 从嵌入文件系统中读取 `plugin.yaml`、`frontend`、`manifest/sql` 等资源；
3. 继续生成 `lina.plugin.manifest`、`lina.plugin.frontend.assets`、`lina.plugin.install.sql`、`lina.plugin.uninstall.sql` 等自定义节；
4. 目录扫描仅作为兼容旧样例和迁移期插件的回退路径。

这样可以让宿主继续只消费静态快照，同时把作者侧资源声明统一到 `go:embed` 模型。

### 决策三：动态插件使用显式导出的嵌入文件系统，而不是依赖 Go 编译产物反射

动态插件的 `go:embed` 资源不会通过“解析 Wasm 内部 Go 运行时结构”被宿主读取，而是通过插件代码显式导出一个资源文件系统句柄，供构建器在构建阶段使用。

采用显式导出的原因：

- `embed.FS` 的编译后布局不是稳定对外契约；
- 解析 Go 编译产物内部布局会把构建链路绑定到工具链实现细节；
- 显式导出更接近源码插件 `UseEmbeddedFiles` 的心智模型，也更容易测试和维护。

### 决策四：宿主解析链路保持快照优先，避免运行时双真相

宿主侧的动态插件解析、前端资源托管、菜单校验、动态路由治理和上传接口不直接接触 `go:embed`。宿主只认构建后的自定义节快照，避免出现以下双真相问题：

- 构建器认为某个资源已嵌入，但 guest 返回另一套资源列表；
- 宿主已允许启用，但运行时读资源时又因 guest 实现差异返回缺失；
- 升级回滚时快照与 guest 逻辑不一致，导致旧 release 无法稳定恢复。

### 决策五：兼容旧插件，但把新样例和文档切到统一声明方式

本次迭代将：

- 把 `plugin-demo-dynamic` 切换到新的 `go:embed` 资源声明方式；
- 更新插件开发文档，明确源码插件与动态插件在作者侧资源声明上的统一做法；
- 在构建器中保留目录扫描回退，给已有动态插件预留迁移窗口。

这样可以先统一推荐模式，再视迁移情况决定后续是否移除回退逻辑。

## Risks / Trade-offs

- [风险] 构建器若需要真正加载插件包中的嵌入文件系统，可能引入额外的构建阶段装配复杂度。
  → Mitigation：优先复用现有 Go 包导出约定与构建工具链，只在 `hack/build-wasm` 内实现这一层装配，不把复杂度扩散到宿主运行时。

- [风险] 同时支持“嵌入文件系统优先 + 目录扫描回退”会带来短期双路径维护成本。
  → Mitigation：以新样例和文档推动统一声明方式，并为回退路径补充测试，后续再评估移除窗口。

- [风险] 若作者侧 `go:embed` 声明与源码目录内容不一致，可能导致构建结果与开发者预期不符。
  → Mitigation：构建阶段对缺失的必需资源直接报错，并在文档中要求 `plugin.yaml`、`frontend`、`manifest` 的嵌入路径与目录结构保持一致。

- [风险] 若误把宿主侧也改成 guest 资源读取，会削弱上传和启用前治理能力。
  → Mitigation：在实现和规范中明确“作者侧统一，宿主侧快照不变”的边界，拒绝把静态资源读取并入 guest ABI。

## Migration Plan

1. 为动态插件定义统一的嵌入资源声明约定，并在样例插件中落地。
2. 调整 `hack/build-wasm`，优先从插件导出的嵌入文件系统读取资源，再回退目录扫描。
3. 保持宿主解析自定义节逻辑不变，只补充对新构建产物的兼容验证。
4. 更新插件开发文档与样例说明，明确推荐使用 `go:embed` 声明资源。
5. 通过构建器测试、样例构建测试和宿主已有动态插件测试确认无回归。
