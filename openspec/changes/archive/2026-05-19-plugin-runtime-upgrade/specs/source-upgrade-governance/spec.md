## MODIFIED Requirements

### Requirement: 框架元数据必须集中维护并直接在系统信息中展示

框架 SHALL 将其名称、版本、描述、主页、仓库 URL 和许可证保存在 `apps/lina-core/manifest/config/metadata.yaml` 中。系统信息 API 必须直接返回该元数据，使系统信息页面无需前端硬编码值即可渲染项目卡片。

#### 场景：系统信息 API 返回框架元数据
- **当** 管理工作台请求系统信息时
- **则** 响应包含框架名称、版本、描述、主页、仓库 URL 和许可证
- **且** 每个值来自宿主 `metadata.yaml`

## REMOVED Requirements

### Requirement: 源码升级必须提供统一的开发时入口

**Reason**: 旧开发期升级入口是未实现的指令，不应继续作为框架或源码插件升级的规范要求。源码插件和动态插件的文件覆盖属于开发阶段离线更新，运行时状态和数据升级必须通过宿主启动后的插件运行时升级流程处理。

**Migration**: 删除旧开发期升级入口相关文档、任务和错误提示。源码插件离线文件更新使用 `plugins.update` 或直接覆盖插件目录；插件运行时升级使用插件管理页和 `POST /plugins/{id}/upgrade`。框架自身源码升级不在本插件运行时升级变更范围内。

### Requirement: 源码升级必须在执行前完成安全检查

**Reason**: 该要求专属于旧的开发期升级命令。既然该命令不是当前交付能力，相关 Git 工作树安全检查不得作为不存在命令的规范要求保留。

**Migration**: 插件离线文件覆盖的本地改动保护由 `plugin-workspace-management` 中的 `plugins.update` dirty 阻断和 `force=1` 规则承担。运行时升级的安全检查由插件运行时升级预检、权限校验和确认弹窗承担。

### Requirement: 框架升级必须仅从 hack 配置读取升级元数据

**Reason**: 该要求绑定旧的框架源码升级工具和升级入口，不属于当前已实现能力，也不属于插件运行时升级范围。

**Migration**: 保留框架元数据展示能力。未来若重新设计框架源码升级，应单独提出变更，不复用插件运行时升级流程。

### Requirement: 框架升级必须从第一个文件重放所有宿主 SQL

**Reason**: 该要求绑定旧的框架源码升级工具和宿主 SQL 重放策略，不属于当前插件运行时升级能力，继续保留会误导用户以为存在开发期升级入口。

**Migration**: 插件运行时升级只执行目标插件自己的 `manifest/sql` 升级资源和治理同步，不负责宿主框架 SQL 全量重放。未来框架升级策略需要独立设计和实现。
