## MODIFIED Requirements

### Requirement: 源码插件必须分离有效版本和发现的源码版本

系统 SHALL 区分当前有效的源码插件版本和在源码树中发现的插件版本。`sys_plugin.version` 和 `release_id` 仅代表有效版本，新发现的版本存储为发布记录或发现快照，在运行时升级完成前不得覆盖有效版本。

#### 场景：已安装的源码插件发现更高版本
- **当** 源码插件 `plugin-demo` 有效运行 `v0.1.0` 且其源码中的 `plugin.yaml` 已升级到 `v0.5.0` 时
- **则** `sys_plugin.version` 保持 `v0.1.0`
- **且** 系统记录 `v0.5.0` 源码插件发布快照
- **且** 该新发布不被视为当前有效版本，直到运行时升级完成

#### 场景：已安装的源码插件发现更低版本
- **当** 源码插件 `plugin-demo` 有效运行 `v0.5.0` 且其源码中的 `plugin.yaml` 是 `v0.1.0` 时
- **则** `sys_plugin.version` 保持 `v0.5.0`
- **且** 系统将插件标记为异常状态
- **且** 系统要求管理员人工修复文件或数据库状态后再恢复正常

### Requirement: 源码插件升级必须是显式的运行时操作

系统 SHALL 要求源码插件升级通过宿主启动后的运行时管理 API 显式执行，而非通过开发期升级命令执行，也非在宿主启动期间自动修复。开发阶段只能通过文件覆盖让宿主发现新版本。

#### 场景：显式升级单个源码插件
- **当** 管理员在插件管理页确认升级 `plugin-demo`
- **且** 宿主收到 `POST /plugins/{id}/upgrade` 请求
- **则** 系统仅为 `plugin-demo` 执行运行时升级流程
- **且** 不触发其他源码插件或动态插件的升级

#### 场景：源码插件文件覆盖后等待运行时升级
- **当** 开发者覆盖 `apps/lina-plugins/plugin-demo` 下的源码插件文件
- **且** 插件 `plugin.yaml` 版本高于数据库有效版本
- **则** 宿主启动后将插件标记为待升级
- **且** 系统不得要求运行旧的开发期升级命令

### Requirement: 宿主启动必须标记源码插件升级状态

宿主 SHALL 在启动期间扫描源码插件，然后比较已安装源码插件的发现版本和有效版本。如果发现版本高于有效版本，宿主必须标记插件为待升级并继续启动；如果发现版本低于有效版本，宿主必须标记插件为异常并继续启动。

#### 场景：待处理的源码插件升级不阻塞启动
- **当** 宿主启动并发现 `plugin-demo` 有效运行 `v0.1.0` 而源码发现报告 `v0.5.0` 时
- **则** 启动流程继续
- **且** 插件运行时状态变为待升级
- **且** 插件管理页可显示有效版本、发现版本和升级动作

#### 场景：源码插件发现版本低于有效版本
- **当** 宿主启动并发现 `plugin-demo` 有效运行 `v0.5.0` 而源码发现报告 `v0.1.0` 时
- **则** 启动流程继续
- **且** 插件运行时状态变为异常
- **且** 插件管理页提示管理员人工干预修复

### Requirement: 源码插件升级必须记录 `phase=upgrade` 并同步治理资源

源码插件运行时升级 SHALL 执行升级阶段迁移记账并同步菜单、权限、资源引用、i18n、apidoc、路由和 cron 等治理资源。成功运行后，新发布成为有效发布。

#### 场景：源码插件升级成功
- **当** 管理员升级已安装的源码插件且所有升级回调、SQL 和治理同步步骤成功时
- **则** `sys_plugin.version` 和 `release_id` 更新为新发布
- **且** `sys_plugin_migration` 记录 `phase=upgrade` 条目
- **且** 新发布成为有效发布
- **且** 插件运行时状态变为正常

#### 场景：源码插件升级失败
- **当** 源码插件升级期间插件回调、升级 SQL 语句或治理同步步骤失败时
- **则** 运行时升级流程立即停止
- **且** 保留失败的升级记录和错误信息
- **且** 插件运行时状态变为升级失败
- **且** 系统不自动执行回滚

### Requirement: 动态插件升级必须进入统一运行时升级模型

系统 SHALL 保持动态插件升级在运行时模型上。动态插件新 artifact 被上传或文件覆盖后，若发现版本高于数据库有效版本，系统必须将插件标记为待升级，并通过同一插件管理页升级流程完成有效 release 切换和治理资源同步。

#### 场景：动态插件发现更高版本
- **当** 动态插件 `plugin-demo-dynamic` 有效运行 `v0.1.0`
- **且** 本地发现或上传的动态插件 artifact 版本为 `v0.2.0`
- **则** 系统将插件标记为待升级
- **且** 有效 release 仍保持 `v0.1.0`
- **且** 管理员必须通过插件管理页显式确认升级

#### 场景：动态插件文件版本低于有效版本
- **当** 动态插件 `plugin-demo-dynamic` 有效运行 `v0.2.0`
- **且** 本地发现或上传的 artifact 版本为 `v0.1.0`
- **则** 系统将插件标记为异常
- **且** 系统不得自动降级有效 release

## REMOVED Requirements

### Requirement: 源码插件升级必须是显式的开发时操作

**Reason**: 该要求依赖未实现的开发期升级指令，并错误地把运行时数据库状态、治理资源和插件数据迁移放入开发阶段。插件文件覆盖只能产生发现版本，不能完成运行时升级。

**Migration**: 使用运行时插件管理页和 `POST /plugins/{id}/upgrade` 执行显式升级。开发阶段仅通过 `plugins.update` 或直接文件覆盖更新本地插件文件。

### Requirement: 宿主启动必须验证源码插件升级已完成

**Reason**: 启动阻断会导致管理员无法进入插件管理页执行运行时升级，且无法处理数据库状态和插件文件状态不一致的恢复流程。

**Migration**: 宿主启动改为标记 `pending_upgrade`、`abnormal` 或 `normal` 状态，并保持管理入口可用。待升级插件的业务入口由运行时状态控制。
