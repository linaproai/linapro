## Context

Lina 管理后台当前使用纯无状态 JWT 认证，后端不跟踪用户会话。管理员无法查看在线用户列表，也无法强制下线异常会话。同时系统缺少服务器运行状态的可视化监控能力。

当前技术栈：GoFrame v2 + MySQL + JWT（HS256），前端 Vben5 + Vue 3 + Ant Design Vue。

## Goals / Non-Goals

**Goals:**

- 实现基于 MySQL MEMORY 引擎的会话跟踪机制，支持在线用户列表查询和强制下线
- 设计会话存储抽象层，当前基于 MEMORY 引擎实现，未来可切换为 gcache + Redis
- 实现基于 gopsutil 的服务器指标定时采集，写入 MySQL，支持多节点分布式部署
- 前端新增系统监控模块，包含在线用户和服务监控两个页面

**Non-Goals:**

- 不实现实时推送（WebSocket）刷新监控数据，前端通过手动刷新获取最新数据
- 不实现历史指标趋势图表，仅展示最新一次采集的快照数据
- 不实现报警/告警功能
- 不实现集群管理（自动发现节点等），节点仅上报自身数据

## Decisions

### 决策1：会话存储使用 MySQL MEMORY 引擎

**选择**: MySQL MEMORY 引擎表 `sys_online_session`

**备选方案**:
- A) gcache 内存缓存：性能最优，但进程重启丢失，且不支持跨实例共享
- B) Redis：支持分布式，但引入额外组件依赖
- C) MySQL MEMORY 引擎：通过 MySQL 存储在内存中，性能接近内存缓存，无需额外依赖

**理由**: MEMORY 引擎降低组件依赖复杂度，利用已有 MySQL 基础设施。MySQL 重启后数据丢失可以接受（用户重新登录即可）。通过定义 `SessionStore` 抽象接口，未来可无缝切换到 gcache + Redis 方案。

### 决策2：服务监控采用定时采集 + 数据库存储

**选择**: gopsutil 定时采集 → 写入 `sys_server_monitor` 表 → API 从数据库读取

**备选方案**:
- A) 实时查询（API 请求时现场采集）：简单但不支持多节点，且响应慢
- B) 定时采集写数据库：支持多节点、无状态部署，历史数据可查

**理由**: 定时采集使 Lina 服务保持无状态。每个节点独立采集自身指标并写入数据库，新增节点只需部署服务即可自动上报。API 从数据库读取各节点最新数据。

**采集参数**:
- 采集频率：默认 30 秒
- 数据保留：最近 1 小时（定期清理旧记录）
- 节点标识：hostname + 本机 IP 自动获取

### 决策3：网络流量指标纳入 v0.7.0

gopsutil 的 `net.IOCounters(false)` 实现非常简单（3 行代码），返回 BytesSent/BytesRecv 等累计值。通过两次采集的差值除以时间间隔即可计算速率。实现成本极低，纳入本次迭代。

### 决策4：前端按节点展示监控数据

- 单节点时：直接展示指标，无需选择
- 多节点时：提供下拉选择器切换节点查看

## Risks / Trade-offs

- **[MEMORY 引擎限制]** → MEMORY 引擎不支持 BLOB/TEXT 类型，所有字段需使用定长类型。会话数据字段使用 VARCHAR 即可满足需求。
- **[MySQL 重启丢失会话]** → MySQL 重启后所有在线会话丢失，用户需重新登录。对管理后台场景影响可控。
- **[采集间隔与实时性]** → 30 秒采集间隔意味着前端看到的数据最多有 30 秒延迟。对监控场景可接受。
- **[监控数据表增长]** → 每 30 秒一条记录，单节点每小时 120 条。定期清理保留 1 小时数据，表大小可控。
