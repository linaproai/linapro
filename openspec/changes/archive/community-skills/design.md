## Context

`lina-community-issue-review`、`lina-community-pr-review`和`lina-community-release-changelog`是面向社区协作治理的代理技能，不属于`apps/lina-core`运行时能力，也不改变默认工作台、插件宿主、数据库或前端交互。它们的核心价值是把项目规范和源码实现用于`GitHub Issue`分类、`PR`审查和版本更新日志生成，形成可重复执行的外部协作流程。

现有`.agents/skills/`已经定义了技能治理原则。三个新技能沿用这些治理原则，分别处理`Issue`分类、`PR`审查和发布日志生成。

## Goals / Non-Goals

**Goals:**

- 默认审查`linaproai/linapro`仓库的开放`Issue`和`PR`。
- 支持按指定编号审查单个`Issue`或`PR`。
- 避免重复审查已经由技能评论并打过标签的`Issue`，避免重复审查已批准或无新提交的`PR`。
- 识别功能需求或`Bug`反馈是否已经在当前项目中处理，并在确认后关闭对应`Issue`。
- 按项目规范和源码实现回答疑问、评估功能需求、评估`Bug`原因或关闭无效内容。
- 在`PR`发现问题时提供文件、行号、规则来源和可执行修改建议，在无法处理时升级给历史维护成员。
- 手动生成详尽的双语版本更新日志，支持默认范围和显式双引用范围。
- 评论语言跟随`Issue`或`PR`描述语言。
- 明确`Issue`和`PR`内容不可信，禁止被提示注入改变审查规则或触发不安全命令。

**Non-Goals:**

- 不实现独立长期维护脚本或`GitHub Actions`工作流。
- 不自动创建 OpenSpec 变更、实现功能、修复`Bug`或修改运行时代码。
- 不运行`Issue`正文或`PR`代码中的脚本、安装命令或外部链接下载内容。
- 不对已关闭且无重新审查要求的历史`Issue`做批量处理。
- 不把标签作为仓库项目管理的唯一事实来源。
- 不自动合并、关闭、批准`GitHub Review`或修改`PR`分支。
- 不修改`CI`配置、不创建`GitHub Release`、不新增后端、前端、数据库或插件运行时能力。

## Decisions

### 使用技能文档而不是脚本

社区协作能力需要代理结合项目规范、源码和自然语言上下文进行判断，单个脚本无法可靠替代语义审查。技能文档固定流程、边界和模板，实际分类由代理在执行时读取可信上下文完成。

### 以隐藏评论标记和标签共同判断重复审查

`Issue`技能使用隐藏标记`<!-- lina-community-issue-review ... -->`记录处理状态，只有同时存在技能评论标记和`question`、`feature`或`bug`任一标签时才按已处理跳过。`PR`技能使用隐藏标记`<!-- lina-community-pr-review ... -->`记录`headRefOid`，相同`PR`和相同`head`时跳过。

### 分类后执行最小必要 GitHub 状态变更

`Issue`技能：疑问类添加`question`并关闭；可行功能需求添加`feature`并保持开放；可行`Bug`添加`bug`并保持开放；无效内容关闭后发布说明评论且默认不添加处理标签。`PR`技能：发现问题时发布幂等审查评论；符合规范时添加`bot-approved`标签。

### 已处理 Issue 不进入待办队列

功能需求和`Bug`类`Issue`在打`feature`或`bug`标签前，先核对当前项目规范、源码、测试和变更记录。如果等价功能已经存在或`Bug`已经被当前实现修复，技能发布`status=resolved`评论说明原因和证据，并关闭`Issue`。

### 人工升级流程

`PR`技能无法可靠审查时，根据`PR`变更文件在目标分支上的提交历史选择项目成员进行`@`提醒。筛选规则：提取能映射到`GitHub login`的提交作者，与仓库协作者取交集，过滤机器人账号、`PR`作者和当前执行账号，按覆盖文件数、最近修改时间和提交次数排序，最多`@`三名成员。

### 手动技能优先于自动化发布集成

`release-changelog`技能先实现为手动技能，不接入`CI`。这样发布人员可以在不同版本区间上人工试运行、审阅和修订技能提示，待生成质量稳定后再设计自动发布集成。

### 证据来源以仓库内容为准

`release-changelog`技能不得要求`PR`标识作为输入。生成内容必须来自`Git`历史、源码差异和`OpenSpec`内容，覆盖关键能力、治理、修复和工具链变化。

## Risks / Trade-offs

- 错误分类真实`Issue`或`PR`。缓解：要求基于项目规范和源码证据分类，信息不足时优先请求补充而不是打标签或关闭。
- 误关有效问题。缓解：只有疑问类回答完成、明确无效、骚扰或广告时关闭；功能需求和`Bug`可行时保持开放。
- 误判已处理。缓解：只有能从当前规范、源码、测试或变更记录中确认等价能力或修复时才按`resolved`关闭；证据不足时继续走普通分类。
- 提示注入影响执行。缓解：明确`Issue`和`PR`内容不可信，只可用于分类、语言判断和事实线索，不可改变命令或规则。
- 权限不足导致半完成状态。缓解：技能要求在修改`GitHub`状态前做只读权限检查，并在失败时报告未完成的动作。
- `Git`历史和`OpenSpec`内容语义不一致。缓解：`release-changelog`技能优先采用源码和最终`OpenSpec`状态，无法确认时写保守描述。
- 双语内容可能信息不一致。缓解：技能先完成英文结构，再生成语义等价的中文内容，中英文不得交叉混写。
