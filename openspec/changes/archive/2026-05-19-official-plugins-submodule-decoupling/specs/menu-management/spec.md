## MODIFIED Requirements

### Requirement: 插件管理菜单使用稳定业务标识

系统 SHALL 为插件治理的菜单提供稳定的业务标识 `menu_key`，并以此作为菜单生命周期管理的主要锚点。

#### Scenario:识别插件菜单归属

- **当** 宿主需要判断菜单属于某个插件时
- **则** 宿主优先基于 `sys_menu.menu_key` 解析菜单归属
- **且** `remark` 仅作为备注展示字段，不作为正式功能标识

#### Scenario:插件菜单安装或升级

- **当** 动态插件安装器或源码插件同步流程基于清单菜单元数据创建或更新菜单时
- **则** 宿主按 `menu_key` 执行幂等写入
- **且** 宿主根据 `parent_key` 解析真实的 `parent_id`
- **且** 相同 `menu_key` 在宿主范围内保持唯一

#### Scenario:插件自主选择菜单父级

- **当** 插件 manifest 声明 `parent_key`
- **则** 宿主不得基于插件 ID 强制绑定特定宿主目录
- **且** 宿主不得仅因 `parent_key` 不是宿主稳定目录而拒绝 manifest
- **且** 宿主在同步菜单时只要求该 `parent_key` 对应的父级菜单记录存在

#### Scenario:插件卸载菜单元数据清理菜单

- **当** 动态插件卸载或宿主执行插件菜单清理时
- **则** 宿主根据清单中声明的 `menu_key` 删除对应菜单和角色关联
- **且** 宿主不依赖插件卸载 SQL 手动维护菜单删除语句

### Requirement:默认后台使用稳定的一级目录结构

系统 SHALL 为默认管理后台提供面向项目管理后台主场景的稳定一级目录结构。

#### Scenario:查询默认后台菜单骨架
- **当** 宿主为当前用户投射默认后台菜单时
- **则** 一级目录按以下结构组织：`工作台`、`权限管理`、`组织管理`、`系统设置`、`内容管理`、`系统监控`、`任务调度`、`扩展中心`、`开发中心`
- **且** 这些一级目录对应的宿主稳定父级 `menu_key` 分别为 `dashboard`、`iam`、`org`、`setting`、`content`、`monitor`、`scheduler`、`extension`、`developer`

#### Scenario:一级目录作为宿主稳定目录记录存在
- **当** 宿主初始化或同步默认后台菜单骨架时
- **则** 这些一级目录由宿主创建和拥有
- **且** 插件可以通过 `parent_key` 引用这些宿主目录，但宿主不得强制具体插件只能挂载到某一个固定目录

#### Scenario:默认后台扩展业务模块
- **当** 开发者持续向项目添加业务模块或官方源码插件时
- **则** 新菜单会优先放置在现有稳定目录中
- **且** 无需频繁重构一级导航命名和结构

### Requirement:插件菜单语义化挂载由插件清单声明

系统 SHALL 将插件菜单的语义化挂载位置视为插件 manifest 的产品装配声明，而不是宿主按插件 ID 硬编码的运行时策略。

#### Scenario:插件声明语义化挂载目录
- **当** 插件需要将菜单同步到宿主时
- **则** 插件在自身 `plugin.yaml` 中声明对应 `parent_key`
- **且** 宿主按照该声明解析父级菜单，而不维护官方插件 ID 到父级目录的硬编码映射

#### Scenario:插件声明不存在的父级菜单
- **当** 插件 manifest 声明的 `parent_key` 不存在于宿主菜单记录中
- **则** 宿主同步菜单失败
- **且** 错误说明该 `parent_key` 无法解析
