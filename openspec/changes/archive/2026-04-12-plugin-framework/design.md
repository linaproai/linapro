## Context

Lina 当前的前后端扩展方式仍以“宿主源码直接改造”为主，尚未形成稳定的插件平台能力。结合现有代码与本次探索，当前状态存在以下约束：

- 后端 HTTP 路由与控制器在 `apps/lina-core/internal/cmd/cmd_http.go` 中集中硬绑定，尚无插件注册与装配入口。
- 登录、登出等认证流程由 `apps/lina-core/internal/service/auth/auth.go` 主导，适合作为首批宿主 Hook 的发布位置，但当前没有事件总线与失败隔离机制。
- 数据初始化只扫描 `apps/lina-core/manifest/sql/` 与 `mock-data/`，尚不能管理插件携带的 SQL、静态资源与迁移记录。
- 前端动态路由目前只从宿主本地 `views` 中解析页面，`apps/lina-vben/apps/web-antd/src/router/access.ts` 与 `apps/lina-vben/packages/utils/src/helpers/generate-routes-backend.ts` 均默认页面资源在宿主项目内。
- 宿主前端最终会被打包并嵌入后端二进制，这意味着源码插件与动态插件的前端接入路径必须分离：源码插件可参与宿主构建，动态插件则必须通过宿主额外托管或挂载其前端资源。
- 项目已有菜单、角色、认证、数据权限、分布式锁与主节点选举等通用能力，插件机制必须优先复用这些既有能力，而不是重新发明独立治理体系。

本次变更要解决的不是“单一技术插件”，而是一套能够承载以下场景的插件平台：

1. 开发阶段的源码插件，目录固定在 `apps/lina-plugins/<plugin-id>/`，插件作者仅手工维护 `apps/lina-plugins/lina-plugins.go` 中的显式注册关系。
2. 运行时安装的 `wasm` 插件，支持热启停、热升级与平滑卸载。
3. 可嵌入宿主后台的前后端扩展，包括后端 Hook、前端 Slot、宿主页面挂载、菜单接入与权限治理。
4. 可被 Lina 统一治理的独立前端子应用，包括 `iframe`、新标签页与宿主内嵌挂载三种呈现模式。
5. 多节点部署下的插件代际传播、节点协调与失败回滚，但插件自身的内部一致性策略仍由插件开发者决定。

## 2026-04-11 热升级与回滚收尾补充

本次继续把第三期热升级能力从“设计口径”推进到“已实现并经回归验证”的状态，核心收敛如下：

- 动态插件治理正式区分“期望中的 staged artifact”与“当前正在服务的 active release”。上传新的 `wasm` 只会先覆盖 `plugin.dynamic.storagePath/<plugin-id>.wasm` 这份可变 staging 产物；真正对外服务的版本则固定指向 `plugin.dynamic.storagePath/releases/<plugin-id>/<version>/<plugin-id>.wasm` 归档文件。
- 主节点 Reconciler 负责在安装、启停、升级和卸载时推进 `desired_state/current_state/generation/release_id` 四元状态机，并在成功切换后再推进宿主注册表、release 状态和节点投影；非主节点只收敛本地投影，不重复执行共享迁移和 release 切换。
- 当动态插件升级失败时，宿主会把失败 release 标记为 `failed`，恢复注册表到上一个稳定 release，并继续使用旧 release 对外服务。失败 release 的前端静态资源不会继续通过公共 `/plugin-assets/...` 路径暴露，避免用户误命中已回滚版本。
- 当前生效 release 在重新加载 manifest 时，不再只恢复顶层 manifest 元数据，而是会一并恢复 `wasm` 内嵌的后端 Hook 与通用资源契约。这样即使宿主已经切到归档 release，认证 Hook、通用资源查询和失败隔离链路仍然与当前生效版本保持一致。
- 前端壳层新增了面向动态插件代际变化的页面刷新提示机制。只有当前停留在该插件页面的用户会看到“刷新当前页面”提示；点击后宿主会先重建当前登录用户可访问菜单和动态路由，再在不强制跳离当前插件页的情况下重挂载到新代际资源。停留在非插件页面的用户保持无感，不会收到多余提示或被强制跳转。

结合前面的模板口径调整，本次也同步确认：

- 仓库继续不提供独立 `plugin-template` 目录；`plugin-demo-source` 与 `plugin-demo-dynamic` 两个真实样例就是当前插件开发样板。

## 2026-04-12 集群部署与拓扑边界收敛补充

随着第三期动态插件热升级、多节点代际切换与主节点 Reconciler 落地，插件机制本身已经不再只是“插件目录/清单/页面接入”的问题，而是明确依赖宿主的部署拓扑语义。原先为收敛这部分复杂度而拆出的 `cluster-deployment-toggle` 与 `refine-cluster-service-boundaries`，本质上都属于插件机制的后续反馈，而不是独立 change。

本次合并后，设计边界补充如下：

- 宿主默认按单节点模式运行，仅在 `cluster.enabled=true` 时启用选主与主节点专属行为。这样可以让动态插件的热升级、启停、卸载和后台收敛在单节点场景下同步完成，而不是被多节点治理逻辑拖累。
- `cluster.Service` 作为唯一的宿主拓扑门面，负责暴露 `IsEnabled()`、`IsPrimary()`、`NodeID()` 以及按需启动内部选主实现。`cron`、HTTP 启动链路和插件运行时都通过它读取拓扑语义。
- `plugin.Service` 不再通过包级全局变量保存 `clusterModeEnabled` 或主节点回调，而是显式持有最小拓扑接口。这样可以避免 `plugin` 包在没有依赖注入的情况下偷偷持有宿主级状态。
- 原先独立的 `election` 组件实际已只被 `cluster` 内部使用，因此本次设计上将其视为 `cluster` 的内部实现文件，而非顶层 service 边界的一部分。
- 节点标识生成逻辑也统一收敛到 `cluster`，并复用于动态插件节点投影等多节点治理能力，避免 `cluster` 与 `plugin` 各自生成不同的节点名。

这部分收敛的结果是：插件机制的多节点治理不再散落在多个 service 之间，而是明确依赖同一套 cluster topology abstraction。后续人工校验也应以 `plugin-framework` 为唯一 review 入口，而不是把 cluster 相关 follow-up 分拆成独立 change。

## Goals / Non-Goals

**Goals:**

- 建立统一插件契约，覆盖源码目录、运行时产物、前端资源、后端扩展、权限治理与生命周期管理。
- 支持两类一级插件类型：源码插件与动态插件；当前 `dynamic` 动态插件仅支持 `wasm` 实现。
- 支持三类前端接入模式：`iframe`、新标签页、宿主内嵌挂载，并允许插件内部使用任意前端框架。
- 为插件提供标准后端 Hook 与前端 Slot 扩展机制，明确上下文、顺序、异常隔离与可观测性。
- 复用 Lina 现有菜单、角色、权限、数据权限与模块解耦规范，保证插件在启停与卸载时平滑降级。
- 为动态插件提供热更新与多节点代际切换方案，使普通用户无感，插件当前活跃页面用户只需收到刷新提示。
- 通过分期路线将复杂能力拆解为可控迭代，避免一次性实现跨度过大。

**Non-Goals:**

- 本次变更不将 `external service` 作为一期第一类动态插件，只为后续接入预留模型与扩展位。
- 本次变更不允许插件任意侵入宿主私有实现；只有宿主公开的 Hook、Slot、页面装载协议和治理接口可以被插件依赖。
- 本次变更不要求所有任意前端项目都自动获得宿主内嵌能力；若选择宿主内嵌挂载，插件必须实现宿主约定的前端适配协议。
- 本次变更不在卸载插件时默认删除插件业务数据；默认策略是卸载能力与治理资源，保留业务数据。
- 本次变更不替插件兜底其内部的跨节点业务一致性，只负责宿主级插件装载、状态传播与代际切换。

## Decisions

### 决策一：统一插件契约，一级 `type` 仅保留 `source/dynamic`，动态插件产物再单独表达

插件平台将使用统一清单（建议命名为 `plugin.yaml`）描述插件，不再先按“源码插件”“动态插件”分裂成两套模型。一期先保持模型收敛，但当前已落地清单只保留最小基础字段：

- 基本信息：`id`、`name`、`version`、`description`。
- 插件一级类型：`type = source | dynamic`。
- 可选补充信息：`author`、`homepage`、`license`。
- 动态插件产物：当前仅支持 `wasm`；若后续引入其他运行时交付形态，再作为 `dynamic` 类型下的扩展语义补充。
- 其他资源与接入信息：SQL、前端页面、Slot、菜单、权限与路由均优先通过目录约定、SQL 与插件代码推导，不在清单中重复建模。

选择统一契约的原因：

- 可以让源码插件与动态插件共用同一套基础治理模型与最小清单验证流程。
- 可以避免菜单、权限、页面路径、SQL 文件位置在多个位置重复描述，降低双真相源风险。
- 可以避免把动态插件交付形态误建模成一级插件类型，减少列表筛选、状态治理和文档描述上的歧义。
- 后续若出现 `external service` 或其他执行容器，再新增独立运行维度即可。

备选方案与取舍：

- 方案 A：源码插件、动态插件各定义一套清单。缺点是能力重复、文档分裂、后续演进成本高，放弃。
- 方案 B：只定义最小 manifest，把复杂信息分散到多文件。缺点是安装校验与治理边界模糊，放弃。
- 方案 C：一期同时落地 `distribution.kind` 与 `runtime.kind` 两个持久化字段。缺点是当前运行时能力尚未真正交付，过早持久化细分字段会引入重复表达和演进负担，暂不采用。

### 决策二：源码插件通过目录约定发现资源，并由开发者手工维护显式注册表接入后端

源码插件固定放在 `apps/lina-plugins/<plugin-id>/`。宿主当前采用“前端资源按目录扫描，后端入口显式注册”的折中方案：

- 后端：开发者在 `apps/lina-plugins/lina-plugins.go` 中追加匿名导入，将各插件后端入口编织进宿主插件管理组件。
- 前端：按 `frontend/pages/**/*.vue` 与 `frontend/slots/**/*.vue` 目录约定发现资源，供宿主路由解析、菜单挂载与 Slot 注册使用。
- 打包：源码插件的前端资源参与宿主前端构建；最终仍然由宿主后端二进制嵌入并交付。

这样做的关键价值：

- 插件开发者只需要维护 `apps/lina-plugins/<plugin-id>/` 和集中式的 `apps/lina-plugins/lina-plugins.go`，不需要手工编辑宿主的 `cmd_http.go`、控制器、菜单或前端页面映射。
- 宿主仍然保持静态可编译与类型安全，不需要在源码模式下引入运行时反射式模块装配。
- 源码插件天然随宿主构建交付，不需要再设计安装/卸载态切换。

备选方案与取舍：

- 手工在宿主多个位置分散导入插件：维护成本高且易遗漏，放弃。
- 完全运行时扫描源码目录：Go 编译链不适合直接对源码进行运行时装配，且会破坏构建确定性，放弃。
- 构建期自动生成后端注册表：会额外引入脚本、生成物与调试链路复杂度，当前阶段放弃。

### 决策三：运行时后端插件采用 WASM Host 作为当前唯一执行容器

当前动态插件仅支持一种产物：

1. `wasm`：允许单文件交付；宿主需要支持从 `wasm` 中读取 manifest 与可选资源包，或支持无前端资源的纯后端插件。

后端执行容器选择纯 Go WASM Runtime（建议 `wazero`）而非 Go 原生 `plugin`：

- `wazero` 跨平台、无 CGO、便于与当前单二进制交付模型集成。
- Go 原生 `plugin` 仅适合特定平台与构建方式，版本兼容与部署成本高，不适合作为平台能力。
- `external service` 的隔离性更强，但接入复杂度与治理边界更大，按用户要求保留到后续迭代。

对于 `.wasm` 单文件，宿主当前通过 `plugin.dynamic.storagePath` 统一管理运行时产物；上传流与运维手工拷贝流都落到这个目录，宿主只扫描该目录根下的 `*.wasm` 文件，不再依赖 `apps/lina-plugins/<plugin-id>/plugin.yaml` 做 runtime 发现。

多节点部署补充约束：

- `plugin.dynamic.storagePath` 当前仍是宿主本地文件系统路径语义；宿主不会把上传到某一节点的 `.wasm` 自动复制到其他节点，也不会通过数据库记录反向分发制品内容。
- 同类约束不仅适用于动态插件 `wasm`；凡是落到节点本地磁盘的上传资源或运行时资源，例如宿主 `upload.path` 下的通用上传文件、插件运行时归档文件与其他本地静态资源，当前都不提供内建的跨节点复制能力。
- 因此，多节点部署时必须为 `plugin.dynamic.storagePath`、`upload.path` 及其他需要跨节点读取的资源目录提供共享存储，或在宿主之外完成可靠的制品分发；否则从节点上传的资源可能只在该节点可见，主节点安装、其他节点状态收敛或后续资源访问都可能失败。

对于 `.wasm` 单文件，宿主提供两种兼容方式：

- 纯后端模式：插件只提供后端 Hook、API 或任务能力，不携带前端资源。
- 嵌入资源模式：插件按宿主规定的 ABI 或自定义段嵌入 manifest 与前端资源包，宿主在安装时提取并注册。

### 决策四：前端接入模式与插件内部框架解耦，统一支持 iframe、新标签页与宿主内嵌挂载

插件可以使用任意前端框架，但宿主只对“接入方式”负责，不对插件内部技术栈做限制。平台定义三种标准接入模式：

- `iframe`：宿主提供菜单、权限、上下文令牌与地址挂载，最适合独立子应用或需要强隔离的前端。
- `new-tab`：宿主生成带权限上下文或单点登录令牌的跳转链接，适合外部控制台式页面。
- `embedded-mount`：插件提供宿主约定的前端适配入口（例如标准 ESM 导出 `mount/unmount/update`），宿主在布局或页面容器中挂载；插件内部可自由使用 Vue、React 或其他框架。

此外，动态插件若携带前端资源，宿主必须提供插件静态资源托管能力，将资源挂载到稳定路径（如 `/plugin-assets/<plugin-id>/<release>/...`）。源码插件则继续参与宿主前端构建并嵌入主二进制。

备选方案与取舍：

- 仅支持 `iframe`：最简单，但无法满足用户要求中的宿主内嵌挂载与新标签页模式，放弃。
- 仅支持宿主内嵌挂载：对插件框架与构建产物要求过高，也无法覆盖独立前端场景，放弃。

### 决策五：插件继续复用 Lina 的菜单、角色、权限与数据权限治理，新增资源归属表而不是为插件重建一套权限系统

插件治理将遵循 Lina 现有后台治理体系：

- 菜单仍然使用 `sys_menu`，角色授权仍然使用 `sys_role_menu`。
- 插件权限标识必须命名空间化，例如 `plugin:demo:view`、`plugin:demo:audit:list`。
- 插件运行所需的数据权限上下文继续复用 Lina 现有用户、角色、部门与数据范围概念。

为解决安装、禁用、卸载与升级时的资源归属问题，新增插件元数据模型而不是大范围改造现有业务表：

- `sys_plugin`：插件当前安装态、启用态、代际、插件形态与错误状态。
- `sys_plugin_release`：插件版本、产物校验信息、资源路径与 manifest 快照。
- `sys_plugin_migration`：插件 SQL 与其他迁移的执行记录。
- `sys_plugin_resource_ref`：插件与菜单、配置、字典、文件、静态资源等资源之间的归属引用。
- `sys_plugin_node_state`：多节点下各节点对插件代际的收敛状态、心跳与错误信息。

采用“资源引用表”的原因：

- 能避免为每个通用业务表都新增 `plugin_id` 字段。
- 能为卸载、重装、升级与审计提供统一的治理入口。
- 能让插件停用时保留角色授权关系，重新启用后自动恢复生效。

### 决策六：宿主发布稳定 Hook 与 Slot 契约，插件只能在公开扩展点上扩展

首批后端 Hook 建议包括：

- `auth.login.succeeded`
- `auth.logout.succeeded`
- `system.started`
- `plugin.enabled`
- `plugin.disabled`

首批前端 Slot 建议包括：

- `auth.login.after`
- `layout.user-dropdown.after`
- `dashboard.workspace.after`
- `system.user.detail.after`

每个 Hook/Slot 必须包含：

- 固定名称与版本。
- 输入上下文 schema。
- 允许返回值或副作用类型。
- 超时时间与失败处理策略。
- 顺序规则（按 manifest 显式优先级或默认按插件 ID 排序）。
- 可观测性字段（执行耗时、结果、错误摘要）。

插件扩展失败不会中断宿主主流程：

- 登录成功后的插件 Hook 失败只能写日志、记告警、记插件执行状态，不得导致登录失败。
- 前端 Slot 组件加载失败只能降级为“跳过渲染 + 上报错误”，不得导致整个页面白屏。

### 决策六补充：前后端插槽必须形成“公开目录 + 类型定义 + 文档”三件套

插件平台不再只在设计文档里零散提到 Hook 和 Slot 名称，而是把“插件可安装位置”作为一等模型管理。宿主需要同时提供以下三层产物：

- 设计层：在 `OpenSpec` 中维护宿主公开插槽目录，明确每个插槽的语义、上下文、排序规则、失败隔离与版本边界。
- 代码层：前后端分别使用专门类型定义合法插槽标识，避免在宿主实现、插件示例和插件开发文档里散落硬编码字符串。
- 文档层：在 `apps/lina-plugins/README.md` 中输出开发者可直接消费的插槽清单、使用方式和示例代码。

当前将插槽模型拆为两类：

| 类别     | 含义                             | 宿主代码形态                  | 当前状态   |
| -------- | -------------------------------- | ----------------------------- | ---------- |
| 后端插槽 | 宿主关键业务事件上的 Hook 挂载点 | Go `type` + `const`           | 一期已发布 |
| 前端插槽 | 宿主公开 UI 容器上的内容插入点   | TypeScript `const` + 联合类型 | 一期已发布 |

一期先把当前真实可用的插槽固化为正式目录。

当前已发布的后端 Hook 插槽：

| 插槽                    | 触发时机            | 说明                                 |
| ----------------------- | ------------------- | ------------------------------------ |
| `auth.login.succeeded`  | 用户登录成功后      | 适合登录审计、登录后初始化等扩展逻辑 |
| `auth.logout.succeeded` | 用户登出成功后      | 适合会话回收、审计补录等扩展逻辑     |
| `system.started`        | HTTP 服务启动完成后 | 适合启动后预热、注册和自检类逻辑     |
| `plugin.installed`      | 动态插件安装完成后  | 仅投递给目标插件自身                 |
| `plugin.enabled`        | 插件启用完成后      | 仅投递给目标插件自身                 |
| `plugin.disabled`       | 插件禁用完成后      | 仅投递给目标插件自身                 |
| `plugin.uninstalled`    | 动态插件卸载完成后  | 仅投递给目标插件自身                 |

当前已发布的前端 UI 插槽：

| 插槽                         | 宿主位置               | 说明                                   |
| ---------------------------- | ---------------------- | -------------------------------------- |
| `layout.user-dropdown.after` | 后台右上角用户菜单左侧 | 适合追加轻量入口、状态提示或快捷操作   |
| `dashboard.workspace.after`  | 工作台主内容区底部     | 适合追加工作台卡片、统计信息或快捷入口 |

后续新增插槽时，必须同步满足以下约束：

1. 先更新 `OpenSpec` 规格和设计，再开放宿主实现。
2. 先新增类型常量，再在宿主模板或插件示例中使用，禁止直接写裸字符串。
3. 先更新 `apps/lina-plugins/README.md`，再视为对插件开发者正式发布。

### 决策七：热更新采用“期望状态 + 代际切换”模型，普通用户无感，插件活跃页面收到刷新提示

宿主对插件的运行时管理采用期望状态模型：

- 数据库记录插件的 `desired_state`、`current_state`、`generation`、`release_id`。
- 负责执行迁移的主节点完成安装或升级后，写入新 `generation`。
- 各节点后台 Reconciler 监听插件代际变化，加载新插件版本并在本地原子替换注册表。
- 新请求只会进入新代际；旧请求允许自然结束。

前端侧同步策略：

- 宿主维护插件注册表版本号或代际摘要。
- 用户停留在非插件页面时，插件更新对其无感。
- 用户停留在当前插件页面时，如果当前页面所属插件代际已变化，宿主显示“插件已更新，请刷新当前页面”的提示。

该模型既满足“无重启热更新”，也与现有项目中的主节点选举机制契合。

### 决策八：将实现分为三期推进，先打底座，再落运行时，再补集群与开发体验

**第一期：插件契约与源码插件底座**

- 建立 `apps/lina-plugins/` 目录规范、最小化 manifest schema 与目录约定。
- 实现源码插件的目录发现、后端显式注册、前端构建接入。
- 打通插件菜单、权限、角色授权与基础治理。
- 发布首批 Hook/Slot 契约。
- 提供示例插件 `plugin-demo`，验证“插件路由接入 + 宿主左侧菜单页展示 + 工作台 Slot 展示”。

**第二期：运行时 wasm 插件**

- 实现运行时产物安装器、校验器、资源提取器与插件静态资源服务。
- 实现 WASM Host、插件 ABI、Hook 执行隔离与安装/启停/卸载/升级 API。
- 打通 `iframe`、新标签页、宿主内嵌挂载三种 UI 模式。
- 让示例插件同时支持源码模式与运行时 `wasm` 模式。

**第三期：多节点热更新与开发者体验**

- 集成主节点迁移、节点状态收敛、失败回滚与代际回退。
- 增加前端插件代际感知、当前插件页刷新提示与审计视图。
- 补充插件开发文档、`wasm` 打包工具、WASM-only 指引与多节点注意事项。
- 预留 `external service` 接入协议，但不在本期实现。

### 决策九：`plugin.yaml` 保持最小化，插件菜单一期以 SQL 为单一真相源

插件资源继续按目录与代码约定组织，不再由 `plugin.yaml` 重复索引。对于插件菜单，一期采用以下约束：

- `plugin.yaml` 只保留基础元数据，不再声明插件菜单、权限前缀、前端页面、Slot 与 SQL 文件列表
- 插件菜单的安装、授权和卸载统一收敛到 `manifest/sql/*.sql` 与 `manifest/sql/uninstall/*.sql`
- 插件菜单 SQL 必须使用 `menu_key` 作为稳定业务标识，并通过父菜单 `menu_key` 解析真实 `parent_id`
- 插件安装、升级、卸载和启停过滤都基于 `menu_key`，不再依赖 `remark`、`id` 或硬编码 `parent_id`

选择该方案的原因：

- `remark` 是备注字段，不适合承载稳定功能语义
- `perms` 是权限语义，不适合作为菜单身份；目录菜单甚至可能没有权限码
- `menu_key` 能让菜单安装、升级、卸载和幂等重放具备稳定主键
- 一期菜单已完全由 SQL 执行链路治理，再保留 `menus.json` 会形成双真相源并增加漂移风险

阶段性实现约束：

- 一期先为受插件治理的菜单引入 `sys_menu.menu_key`
- 插件菜单安装 SQL 必须通过父菜单 `menu_key` 解析真实 `sys_menu.id`
- `remark` 仅保留展示备注用途，不再作为插件菜单归属和过滤的主要依据

## Risks / Trade-offs

- [风险：WASM ABI 设计过早固化] → 通过 ABI 版本号、manifest 版本号和宿主兼容矩阵控制演进，避免后续破坏性升级。
- [风险：任意前端框架的宿主内嵌挂载复杂度高] → 将“任意框架”与“宿主内嵌”解耦，要求内嵌模式必须实现宿主适配协议，其他情况使用 `iframe` 或新标签页。
- [风险：插件菜单与权限治理与宿主耦合过深] → 保持菜单、角色、权限复用，但通过 `sys_plugin_resource_ref` 做所有权追踪，避免污染宿主核心表结构。
- [风险：热更新期间插件页面资源与后端代际不一致] → 采用前端代际检测与页面刷新提示，保证普通用户无感、活跃插件用户可控恢复。
- [风险：多节点升级时迁移重复执行] → 只允许主节点执行迁移与 release 切换，其他节点只执行本地收敛与状态上报。
- [风险：多节点环境下本地磁盘资源未共享] → 当前实现只同步元数据、代际与节点状态，不负责跨节点文件分发；部署时必须为 `plugin.dynamic.storagePath`、`upload.path` 等目录配置共享存储或外部分发流程。
- [风险：插件 Hook 阻塞主流程] → 所有 Hook 采用超时、熔断、错误隔离与观测埋点，关键主流程只接收“异步失败”而不接收“主链路失败”。
- [风险：源码插件接线关系遗漏] → 通过集中式 `apps/lina-plugins/lina-plugins.go`、最小化清单校验与示例文档降低遗漏概率，并把插件扫描限制在 `apps/lina-plugins/` 下。
- [风险：单文件 wasm 同时携带 UI/SQL/资产时实现成本过高] → 平台必须支持，并通过 `wasm` 资源嵌入约定和工具链逐步完善开发体验。

## Migration Plan

1. 在宿主中引入插件元数据表、清单校验器与空注册表，不改变现有业务模块行为。
2. 增加源码插件扫描与示例插件；源码插件首次同步后默认启用，但仍允许管理员后续显式禁用，确保开箱即可验证插件接入效果。
3. 为认证、菜单、角色与前端路由增加插件感知逻辑，并通过开关控制仅对已启用插件生效。
4. 在单节点环境下上线运行时 `wasm` 安装能力，验证安装、启用、停用、卸载、回滚的完整链路。
5. 将主节点选举与节点状态收敛接入插件代际切换流程，在多节点环境完成热升级演练。
6. 形成插件开发文档与打包文档，作为后续功能实现与第三方插件开发的唯一依据。

回滚策略：

- 插件安装或升级失败时，宿主回滚到上一 `release_id` 与上一 `generation`。
- 若迁移已部分执行，则依赖 `sys_plugin_migration` 与资源引用记录执行安全回滚或标记人工介入。
- 前端资源切换失败时，宿主保留旧资源路径并阻止新代际进入生效状态。
