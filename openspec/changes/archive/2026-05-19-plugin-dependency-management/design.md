## Context

当前插件生命周期以单个插件为操作单元：`Install(ctx, pluginID, options)` 先读取目标插件期望清单，再分别进入源码插件或动态插件安装路径。`plugin.yaml` 的 `Manifest` 类型尚未包含依赖声明；源码插件和动态插件清单均只表达自身身份、租户治理、菜单、宿主服务、SQL、前端和运行时资源。

插件数量增加后，以下问题会变成生命周期风险：

- 插件使用另一个插件提供的页面、菜单、数据表、host service provider 或业务能力，但安装时无法声明和校验。
- 插件需要特定 LinaPro 框架版本能力，但清单中无法表达约束。
- 手动安装顺序依赖操作者经验，缺失依赖时错误通常出现在 SQL、路由或运行时调用阶段。
- 卸载一个基础插件可能破坏已安装的下游插件。
- 启动自动启用和源码插件升级路径可能绕过依赖校验，导致启动或升级后系统进入不一致状态。

本设计把依赖治理放入插件清单和插件生命周期编排层，而不是放入单个插件业务代码中。

## Goals / Non-Goals

**Goals:**

- 为源码插件和动态插件提供一致的 `dependencies` 清单语义。
- 支持 LinaPro 框架版本约束和插件依赖版本约束。
- 在安装、启动自动启用、卸载、源码插件升级和动态插件安装/升级路径中执行依赖校验。
- 对已发现且版本满足的硬依赖支持自动安装，并以确定性拓扑顺序执行。
- 通过 API/UI 暴露依赖检查结果、自动安装计划和阻断原因。
- 复用现有插件 runtime revision/event 机制，保证多插件状态变化后的缓存一致性。

**Non-Goals:**

- 不在本变更中实现远程插件 marketplace、依赖下载或外部仓库解析。
- 不在第一阶段实现依赖插件自动升级；已安装依赖版本不满足时阻断安装并给出升级建议。
- 不默认自动启用依赖插件；依赖安装和启用仍保持分离，除非依赖插件本身也在 `plugin.autoEnable` 中。
- 不新增业务模块数据权限模型，不调整租户隔离语义。
- 不改变源码插件目录结构和动态插件 WASM 打包主流程，只扩展清单内容和校验。

## Decisions

### 决策 1：在 `plugin.yaml` 使用结构化 `dependencies` 对象

建议清单形状：

```yaml
dependencies:
  framework:
    version: ">=0.6.0 <0.7.0"
  plugins:
    - id: multi-tenant
      version: ">=0.1.0"
      required: true
      install: auto
    - id: org-center
      version: ">=0.1.0"
      required: false
      install: manual
```

字段语义：

- `framework.version`：LinaPro 框架版本约束。
- `plugins[].id`：依赖插件 ID，必须使用既有插件 ID 命名规则。
- `plugins[].version`：依赖插件版本范围，使用语义化版本约束。
- `plugins[].required`：默认 `true`。硬依赖不满足时阻断安装；软依赖仅作为提示和能力提示。
- `plugins[].install`：默认 `manual`。`auto` 表示安装目标插件时可自动安装该依赖。

备选方案是使用顶层 `frameworkVersion` 和 `pluginDependencies` 两个字段。该方案字段更少，但扩展性差，未来难以为依赖增加作用域、原因、能力名或安装策略，因此不采用。

### 决策 2：依赖解析器作为独立内部组件

新增插件依赖解析组件，输入为目标插件 ID、已发现 manifest 集合、当前 registry/release 状态和框架版本，输出为：

- 依赖检查结论。
- 自动安装计划。
- 阻断原因列表。
- 依赖链和拓扑顺序。
- 反向依赖列表。

该组件不执行安装 SQL、不写 registry、不刷新缓存，只负责纯解析和校验。安装 facade 调用解析器后再按计划复用现有安装生命周期。

备选方案是在 `Install` 方法中直接递归安装依赖。该方案短期代码少，但容易把图解析、错误聚合、循环检测和生命周期副作用混在一起，不利于测试和 UI 预检查复用，因此不采用。

### 决策 3：自动安装只处理已发现且版本满足的硬依赖

当依赖声明为 `required: true` 且 `install: auto` 时，系统只会自动安装当前 catalog 已发现、版本满足且未安装的依赖插件。依赖插件不存在、版本不满足、清单无效或存在循环依赖时，目标插件安装被阻断。

软依赖不参与自动安装阻断；UI 可以提示软依赖未满足导致部分能力不可用，但插件生命周期不得因此失败。

### 决策 4：自动安装按拓扑排序逐个复用现有生命周期

依赖解析器构建有向图：目标插件依赖的节点先安装，目标插件最后安装。拓扑排序必须稳定，除依赖关系外按插件 ID 排序，避免不同节点或不同平台顺序不一致。

安装过程中每个插件继续复用现有源码/动态安装路径。若第 N 个依赖安装失败，系统停止后续安装，并返回已经成功安装的依赖列表、失败插件 ID 和失败依赖链。第一阶段不承诺跨插件全局事务回滚，因为源码/动态插件安装已经包含 SQL、菜单、资源引用、发布归档和外部 runtime 状态，强行跨插件回滚风险高于收益。

### 决策 5：卸载使用反向硬依赖保护

卸载插件前，系统扫描已安装插件的硬依赖声明。如果存在已安装插件依赖当前插件，默认阻断卸载并返回下游插件列表。强制卸载不作为第一阶段公开能力；如果后续确需支持，必须要求操作者先禁用或卸载下游插件，并通过独立变更定义风险边界。

### 决策 6：升级路径必须校验新版本依赖

源码插件升级命令和动态插件安装/升级路径在切换有效版本前，都必须对新版本 manifest 执行依赖校验：

- 新版本框架约束不满足时拒绝升级。
- 新版本新增或提高依赖版本约束时，当前已安装依赖必须满足。
- 如果新版本会使其他已安装插件的反向依赖不满足，必须拒绝切换有效版本。

第一阶段不自动升级依赖插件；错误中返回需要先升级的依赖插件 ID 和版本要求。

### 决策 7：缓存一致性复用插件 runtime revision/event

依赖自动安装会在一次请求中改变多个插件的 registry、release snapshot、菜单、权限、资源引用、enabled snapshot、frontend bundle 和 i18n bundle 状态。单机模式下可在每个生命周期完成后本地刷新；集群模式下必须复用现有插件 runtime revision/event、enabled snapshot 和相关 scope 失效机制，不允许只刷新当前节点内存状态。

依赖检查本身为只读操作，不写缓存状态；安装、卸载、升级成功后才触发对应插件 scope 的失效和事件发布。

### 决策 8：前端展示依赖计划但不承载依赖决策

后端提供安装前检查 API 或在安装响应错误中返回结构化依赖信息。前端插件管理页展示：

- 框架版本检查结果。
- 待自动安装依赖列表。
- 需要手动处理的依赖。
- 循环依赖或版本不满足等阻断原因。
- 卸载阻断的下游插件列表。

依赖图、版本判断和阻断语义以服务端为权威，前端只做展示和确认。

## Risks / Trade-offs

- [风险] 自动安装多个插件时中途失败，系统可能留下部分已安装依赖。缓解：不做隐式全局回滚，返回已安装依赖列表和失败点；后续可由操作者卸载不需要的依赖。
- [风险] 依赖版本约束解析引入新的库或复杂实现。缓解：优先选择轻量且可测试的 semver constraint 实现；若新增依赖，需评估许可证和跨平台构建影响。
- [风险] 自动依赖安装在集群中重复执行。缓解：沿用现有集群 primary/shared lifecycle 边界；从节点只观察共享状态并刷新本地投影。
- [风险] 软依赖语义被误用为运行时硬依赖。缓解：文档明确运行时必需能力必须声明硬依赖；软依赖只用于增强能力提示。
- [风险] UI 依赖状态展示新增文案造成 i18n 漏项。缓解：任务中显式维护前端运行时语言包、宿主 manifest i18n 和 apidoc i18n。
- [风险] 反向依赖扫描依赖最新 manifest，插件工作区缺失时可能无法完整判断源码插件依赖。缓解：已安装插件必须保留 release manifest snapshot；卸载保护优先读取已安装发布快照，发现 manifest 不可用时保守阻断并提示修复。

## Migration Plan

1. 扩展 manifest 类型、动态插件 artifact manifest 和清单校验，保持未声明 `dependencies` 的现有插件合法。
2. 实现依赖解析器和单元测试，覆盖框架版本、插件版本、自依赖、循环依赖、缺失依赖、软依赖、自动安装计划和反向依赖。
3. 将解析器接入手动安装、启动自动启用、卸载和升级路径。
4. 扩展插件管理 API 和前端展示，补齐 i18n 与 apidoc i18n。
5. 为至少一个官方插件样例添加依赖声明，验证 source/dynamic 清单解析一致。
6. 运行后端单元测试、插件生命周期测试、相关 E2E、OpenSpec 校验和缓存一致性相关回归。

Rollback 策略：如果依赖安装编排出现阻塞，可先保留 manifest 字段和只读依赖检查，暂缓自动安装接入；未声明 `dependencies` 的插件仍按既有生命周期运行。

## Open Questions

- LinaPro 框架版本的权威来源应使用构建时版本、配置版本还是已有 release metadata。
- `install: auto` 是否允许安装动态插件依赖，还是第一阶段只允许自动安装源码插件和已上传动态插件。
- 插件启用阶段是否需要额外校验依赖插件已启用，还是只要求已安装。
- 反向依赖保护是否需要提供受控的 force 能力，以及 force 是否应要求同时禁用下游插件。
