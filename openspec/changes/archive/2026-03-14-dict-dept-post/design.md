## Context

v0.1.0 建立了基础的用户管理 MVP，包含 sys_user 单表、JWT 认证、CRUD 和导入导出。当前系统缺少组织架构（部门、岗位）和数据枚举（字典）等管理后台核心基础模块。v0.2.0 将补齐这些模块，同时扩展用户管理以支持部门和岗位关联。

前端严格参考 ruoyi-plus-vben5 项目（`/Users/john/Workspace/gitee/dapppp/ruoyi-plus-vben5`）的对应模块实现，保持 UI 和交互一致。

## Goals / Non-Goals

**Goals:**
- 实现字典管理模块的完整功能，包括 Tag 样式系统（预设色 + 自定义色 + CSS 类）
- 实现部门管理模块的完整功能，包括树形层级结构和负责人选择
- 实现岗位管理模块的完整功能，包括部门关联和部门树筛选
- 通过独立关联表（sys_user_dept / sys_user_post）建立用户与部门/岗位的关联
- 扩展用户管理页面，增加部门树筛选和部门/岗位选择
- 提供全局 DictTag 组件和 DeptTree 组件供各模块复用

**Non-Goals:**
- 不修改 sys_user 表结构（不在 user 表上加 dept_id 等字段）
- 不实现角色管理和权限控制（后续版本）
- 不实现菜单管理（后续版本）
- 不实现字典数据的导入功能（仅导出）

## Decisions

### D1: 用户与部门/岗位使用独立关联表

**决策**: 使用 sys_user_dept 和 sys_user_post 关联表，而非在 sys_user 表上添加 dept_id 字段。

**理由**: sys_user 作为基础表保持轻量，其他模块通过独立表扩展关联关系，降低模块间耦合，便于后续模块的独立开发和部署。

**替代方案**: 在 sys_user 表上直接添加 dept_id 字段（参考项目的做法）。虽然查询更简单，但会让 user 表不断膨胀，且每次新增关联模块都需要 ALTER TABLE。

**影响**: 后端查询用户部门信息时需要 JOIN 关联表，但对前端完全透明——API 返回结构与参考项目一致（包含 deptId、deptName、postIds）。

### D2: 字典管理采用双面板布局

**决策**: 字典管理页面采用左右双面板布局（左侧字典类型列表，右侧字典数据列表），面板间通过 Mitt 事件通信。

**理由**: 与参考项目保持一致，用户操作直观——点击类型即可查看/管理其数据，无需页面跳转。

### D3: Tag 样式完整实现

**决策**: 完整实现 TagStylePicker 组件，支持 12 种预设色和自定义 hex 颜色，使用 vue3-colorpicker 库。

**理由**: 用户要求完整实现，与参考项目保持一致。预设色覆盖常见场景（success/danger/warning/info 等），自定义色满足个性化需求。

### D4: 部门树形结构使用 VXE-Grid treeConfig

**决策**: 部门列表使用 VXE-Grid 的 treeConfig 渲染树形结构，通过 parentField 自动构建层级，无分页。

**理由**: 部门数据量通常不大（几十到几百条），全量加载 + 树形展示是最佳体验。与参考项目一致。

### D5: DeptTree 作为可复用组件

**决策**: 将 DeptTree 组件放在 `src/views/system/user/dept-tree.vue`（与参考项目一致），供用户管理和岗位管理共同使用。

**理由**: 用户管理和岗位管理的左侧部门树在功能上完全一致（搜索、选择、刷新），复用同一组件避免重复代码。

### D6: 部门 ancestors 字段存储祖先路径

**决策**: sys_dept 表包含 ancestors 字段，存储从根到当前节点的路径（如 "0,1,2"），用于快速查询子树。

**理由**: 查询某部门及其所有子部门时，使用 `ancestors LIKE '0,1,%'` 比递归查询高效。与参考项目数据结构一致。

### D7: 字典数据通过 dict_type 字符串关联

**决策**: sys_dict_data 通过 dict_type 字符串字段（而非外键 ID）与 sys_dict_type 关联。

**理由**: 字典类型的 type 字段是业务标识（如 "sys_normal_disable"），直接用它关联更直观，前端缓存和查询也更方便。与参考项目一致。

### D8: 开发顺序

**决策**: 字典管理 → 部门管理 → 岗位管理 → 用户管理扩展。

**理由**:
1. 字典管理无外部依赖，且完成后 DictTag 组件可供后续模块的 status 字段渲染
2. 部门管理依赖字典（status 渲染），且 DeptTree 组件需先实现
3. 岗位管理依赖部门（dept_id 关联 + DeptTree 筛选）
4. 用户管理扩展依赖部门和岗位两个模块都完成

## Risks / Trade-offs

**[关联表 JOIN 性能]** → 用户列表查询需要 LEFT JOIN sys_user_dept 获取部门信息，比直接字段查询稍慢。在当前数据量（百级别用户）下可忽略，大数据量场景可通过索引优化。

**[部门负责人选择依赖关联数据]** → 编辑部门时，负责人下拉需要查询该部门下的用户（通过 sys_user_dept 关联表）。新增部门时无法选择负责人（因为还没有用户关联到该部门），仅编辑时可用。与参考项目行为一致。

**[字典缓存一致性]** → 前端 Pinia Store 缓存字典数据，修改字典后需要刷新缓存。参考项目通过手动"刷新缓存"按钮解决，我们采用相同方案。

**[vue3-colorpicker 依赖]** → TagStylePicker 需要引入 vue3-colorpicker 第三方库。该库体积小且成熟，风险可控。
