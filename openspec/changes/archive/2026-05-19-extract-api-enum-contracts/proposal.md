# 抽象 API 公共枚举契约

## 背景

`apps/lina-core/api` 中存在多处语义相同但分散定义的枚举和值类型，例如列表排序方向、租户覆盖模式、菜单类型、插件桥接访问模式以及大量 `0/1` 状态字段。当前重复定义增加了接口 DTO、控制器和服务层之间的契约漂移风险，也容易让新增模块继续硬编码相同字面值。

参考 Kubernetes 的处理方式，本次变更不建立大而全的全局枚举包，而是按稳定契约边界进行小组件抽象：真正跨 API 复用的契约进入 `apps/lina-core/pkg`，领域私有枚举仍留在领域包内，已有稳定公共包通过类型别名复用。

## 目标

- 新增小型公共契约组件，承载列表排序方向、租户覆盖模式和通用 `0/1` 状态标志。
- 让 API DTO 复用已有的菜单类型和插件桥接契约，并仅将确认为跨模块稳定的值类型提升到公共包，减少重复字面值。
- 保持现有 JSON 字段名、示例值和外部 API 取值不变，避免前端和插件调用方感知破坏性变更。
- 为公共契约组件补齐注释和轻量测试，确保后续模块可安全引用。

## 非目标

- 不新增 `pkg/enums` 或其他大一统枚举包。
- 不改变数据库枚举、字典类型或前端运行时字典来源。
- 不调整 REST 路由、权限、数据权限、缓存策略或业务行为。
- 不修改自动生成的 `api` 骨架、`dao`、`do`、`entity` 文件。

## 影响范围

- 后端 API DTO 类型定义和相关控制器类型转换。
- `apps/lina-core/pkg` 下新增或复用的公共契约组件。
- 后端编译与 API 契约一致性测试。

## 风险

- API 类型替换可能影响控制器赋值和服务层转换，需要通过包级编译测试覆盖。
- 过度抽象可能造成不同业务语义被误合并，因此本次仅抽象已跨模块稳定复用的契约，并保留领域私有枚举边界。
