## Context

`lina-openspec-archive-consolidate`当前已经定义了按功能领域聚合归档的流程：默认读取`openspec/changes/archive/`下日期前缀原始归档，生成非日期前缀的领域归档目录，并在验证后清理原始目录。这个流程能降低归档碎片数量，但它仍把`tasks.md`视为需要语义合并的任务列表，缺少面向长期维护的摘要压缩边界。

当前历史归档中的`tasks.md`通常混合了低价值执行流水和高价值维护证据。低价值内容包括普通 checklist、文件级搬迁流水和重复验证命令；高价值内容包括`FB-*`反馈闭环、根因、修复说明、废弃方案、验证证据、审查结论、`i18n`、缓存、数据权限、DI、跨平台影响等治理记录。若不区分这些内容，归档聚合会持续增长；若直接清空`tasks.md`，又会丢失未来排障和设计追溯所需的证据。

## Goals / Non-Goals

**Goals:**

- 让`lina-openspec-archive-consolidate`在聚合归档时自动执行高价值摘要压缩。
- 明确信息分层：背景进入`proposal.md`，设计决策进入`design.md`，最终契约进入`specs/`，执行与反馈证据摘要进入`tasks.md`。
- 将`tasks.md`从任务流水压缩为维护摘要，保留反馈、根因、验证和审查证据。
- 在清理原始归档前增加语义覆盖门禁，避免不可逆信息丢失。
- 让月度无人值守归档聚合复用同一套压缩规则，并在无法安全压缩时失败。

**Non-Goals:**

- 本变更不直接压缩或删除现有`openspec/changes/archive/`历史目录。
- 本变更不改变 OpenSpec CLI 的归档机制。
- 本变更不引入外部存储、数据库或新的运行时依赖。
- 本变更不修改业务代码、HTTP API、前端 UI、数据库结构或运行时用户文案。

## Decisions

### 1. 在既有技能中内建压缩，而不是新增独立清理技能

`lina-openspec-archive-consolidate`已经拥有归档输入边界、分组规则、清理边界和月度自动化入口。把摘要压缩放进该技能，可以让“聚合”和“裁剪”在同一次语义判断中完成，避免另一个清理工具在不了解聚合结果的情况下删除高价值历史。

替代方案是新增`lina-archive-compact`技能。该方案职责更单一，但需要重新定义输入边界、清理安全、报告和月度 CI 集成，容易与现有聚合流程产生重复或冲突。

### 2. `tasks.md`保留维护摘要，不保留完整任务流水

聚合归档中的`tasks.md`应包含：

- 关键交付阶段摘要。
- `FB-*`反馈闭环摘要。
- 根因、修复路径和最终状态。
- 自动化验证、手工验证和审查结论摘要。
- `i18n`、缓存一致性、数据权限、DI、跨平台和测试策略等治理影响判断。
- 未迁移或未压缩的原因。

普通 checklist、重复命令、逐文件搬迁清单和已被`design.md`或`specs/`完整覆盖的执行流水应被合并或裁剪。这样`tasks.md`仍保留维护价值，但不会无限增长。

替代方案是清空`tasks.md`或只保留一句指针。该方案空间收益最大，但会丢失排障证据，不适合当前项目强调可持续交付和审查闭环的定位。

### 3. 高价值信息按承载目标迁移

压缩流程不应把所有信息都塞回`tasks.md`：

| 信息类型 | 聚合后承载位置 |
|---|---|
| 背景、痛点、目标 | `proposal.md` |
| 架构决策、方案演进、废弃方案 | `design.md` |
| 最终需求和验收场景 | `specs/<capability>/spec.md` |
| 反馈闭环、根因、验证、审查、治理影响 | `tasks.md` |

这能让归档目录保留普通 OpenSpec 结构，同时让维护者按问题类型快速定位历史信息。

### 4. 清理原始归档前必须通过语义覆盖门禁

技能只有在满足以下条件后才能清理参与聚合的原始日期前缀目录：

- 每个输入归档的`proposal.md`背景和影响已经进入聚合`proposal.md`或设计约束。
- 每个输入归档的关键设计决策和方案演进已经进入聚合`design.md`。
- 每个输入归档的增量规范已经合并到对应`specs/`。
- 每个输入归档的`FB-*`、根因、验证、审查和治理影响已经进入聚合`tasks.md`摘要，或在报告中说明无此类信息。
- 最终报告列出保留的信息类别、裁剪的信息类别、未裁剪原因和验证结果。

若无法确认覆盖，技能应保留原始目录并在报告中说明阻断原因。

### 5. 默认读取边界保持不变，压缩模式显式扩展

未指定变更列表时，技能仍只读取日期前缀原始归档目录，避免重复处理已聚合的功能目录。新增“压缩既有聚合目录”的能力必须由用户显式指定目标目录或明确要求压缩既有非日期归档目录，且默认不删除这些非日期目录。

这保留了现有安全边界，同时允许维护者按需对既有`plugin-framework/`、`system-config/`等功能归档执行二次压缩。

## Risks / Trade-offs

- 语义摘要可能遗漏细节 → 使用高价值信息分类、覆盖门禁和最终报告降低风险；无法确认时保留原始归档。
- 压缩结果不如原始任务流水可追溯 → 保留`FB-*`、根因、验证和审查摘要，并依赖 Git 历史追溯完整原文。
- 月度 CI 中 AI 摘要质量不稳定 → 共享提示词必须要求失败优先；无法安全压缩时不得清理原始归档或创建归档 PR。
- 技能说明可能变长 → 将规则组织为明确步骤和表格，避免引入额外参考文件，保持触发后可直接执行。

## Migration Plan

1. 更新`lina-openspec-archive-consolidate`技能说明，加入摘要压缩模式、信息分层、`tasks.md`高价值抽取规则、清理门禁和报告格式。
2. 更新月度归档聚合提示词，明确无人值守执行时必须遵守压缩门禁，无法确认语义覆盖时失败。
3. 新增或更新 OpenSpec 增量规范，定义人工调用和月度自动化调用的行为要求。
4. 运行`openspec validate improve-archive-consolidation-compaction --strict`。
5. 通过静态检索确认技能说明包含关键规则：`FB-`、根因、验证、审查、语义覆盖、压缩报告和非日期目录保护。

## Open Questions

- 是否需要为大规模归档压缩补充确定性脚本，先抽取`FB-*`和验证段落作为 AI 摘要输入索引？当前提案先不引入脚本，保留为后续优化空间。
