## Why

当前项目已经建立了运行时 i18n 资源加载、语言协商和前端语言包聚合能力，但前后端实现逻辑中仍大量存在直接返回中文、中英混排或透传后端原始文案的情况。错误提示、导入失败原因、Excel 导出内容、插件桥接错误、插件生命周期结果、部分前端页面标签和服务端日志/审计展示文案如果继续直接写死语言，会让未来新增语言时只能依赖人工逐处排查，也会导致同一请求在不同语言环境下展示不一致。

本次迭代需要在现有 i18n 基础上补齐“运行时可见文案治理”能力，把用户可见文案、机器可读错误码、开发/运维日志和用户输入内容的边界区分清楚，形成可扫描、可测试、可维护的多语言实现约束。

## What Changes

- 新增运行时文案国际化治理规范，覆盖后端 API 错误、业务校验失败、导入/导出文件、插件桥接/宿主服务错误、插件生命周期结果、前端静态页面文案和请求错误透传。
- 建立后端结构化错误模型：业务错误必须包含稳定错误码、翻译键、英文源文案、参数和默认回退文案；HTTP 响应按请求语言本地化，同时保留机器可读错误码。
- 明确日志与用户可见文案边界：运维日志采用稳定英文和结构化字段，不能依赖本地化文案；审计/操作日志、导入失败报告和导出文件中面向用户展示的名称、状态、结果必须按请求语言投影。
- 为 Excel 导出、导入模板和导入失败原因定义统一的本地化辅助能力，禁止在 `user_excel.go`、`dict_*`、插件导出等业务文件中直接硬编码表头、状态、性别、成功/失败等展示文本。
- 为插件平台与插件示例定义统一错误返回契约：插件桥接协议与宿主服务调用返回稳定错误码和英文开发者源文案，管理端可见结果通过 i18n key 本地化，避免“network request URL 非法”“解析 data list request tag 失败”等中英混排。
- 前端默认管理工作台继续优先使用 `$t` 和运行时语言包；对仍未接入的监控页、在线用户页、插件页面等补齐翻译键，并规范请求拦截器对后端 `errorCode/messageKey/messageParams/message` 的处理顺序。
- 增加自动化扫描和测试门禁，区分注释、测试 fixture、用户数据示例、技术协议常量和真正运行时可见文案；新增或修改运行时文案时必须同步维护 `zh-CN`、`en-US`、`zh-TW` 运行时语言包和插件语言包。
- 本次只制定方案和规范，不直接清理所有硬编码实现；后续 `/opsx:apply` 按任务清单分批落地。

## Capabilities

### New Capabilities

- `runtime-message-i18n-governance`: 运行时可见文案的识别、错误码与翻译键建模、本地化响应、导入导出本地化、插件错误契约、前端透传处理和自动化治理。

### Modified Capabilities

- 无。此次通过新增跨模块治理能力约束所有运行时可见文案，避免在多个业务能力规范中重复维护相同国际化规则。

## Impact

- **后端宿主**：影响 `apps/lina-core/internal/service/**`、`apps/lina-core/internal/controller/**`、`apps/lina-core/pkg/pluginbridge/**`、`apps/lina-core/pkg/pluginfs/**`、`apps/lina-core/pkg/excelutil/**` 等错误返回、导入导出和插件平台能力。
- **源码插件**：影响 `apps/lina-plugins/*/backend/internal/service/**` 中的业务错误、导出表头、操作状态映射和插件示例错误响应；插件需要把运行时用户可见文案放入各自 `manifest/i18n/<locale>/*.json`。
- **前端工作台**：影响 `apps/lina-vben/apps/web-antd/src/api/request.ts`、监控/在线用户页面、上传/导出/插件管理等页面的错误展示和硬编码标签处理。
- **语言资源**：需要维护宿主与插件运行时语言包，新增错误消息、导入导出表头、枚举展示和前端页面文案键；不复用 `apidoc` 资源。
- **测试与治理**：需要新增硬编码文案扫描脚本、缺失翻译校验、后端错误本地化单元测试、导入导出语言测试和关键前端页面 i18n 测试。
- **性能**：本地化查找必须复用现有运行时翻译缓存，不得在错误热路径或批量导出循环中反复构建整包语言资源。
