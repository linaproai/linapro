## Context

当前 LinaPro 登录页由 `apps/lina-vben/apps/web-antd/src/views/_core/authentication/login.vue` 组合 `@vben/common-ui` 的 `AuthenticationLogin` 组件实现。该共享组件已经内置了忘记密码、注册账号、手机号登录、扫码登录和第三方登录等展示开关，但当前业务页没有显式关闭这些入口；同时前端底层偏好系统已经支持 `authPageLayout`，可在 `panel-left`、`panel-center`、`panel-right` 之间切换，不过项目默认值仍是 `panel-right`，后端 public frontend 配置白名单中也尚未暴露对应的宿主系统参数。

因此，这次变更不是从零设计登录页布局，而是在现有宿主配置链路和 Vben 布局能力之上，完成三件事：收敛未实现登录方式的暴露面、把 LinaPro 默认登录框位置改为居中、并把该默认位置纳入宿主 `sys_config` 的公共前端参数治理。

## Goals / Non-Goals

**Goals:**
- 让登录页在当前阶段只暴露账号密码登录能力，不再展示未落地的认证入口。
- 将 LinaPro 登录框默认位置调整为居中展示，并复用现有 `authPageLayout` 能力而不是重复造一套布局枚举。
- 把登录框位置作为宿主内置 public frontend 参数统一管理，允许管理员通过系统参数维护默认布局。
- 让未登录页面能够通过现有公共前端配置接口读取并应用登录框位置配置。
- 为该行为补齐前后端回归验证，避免后续再把未实现入口误放出。

**Non-Goals:**
- 本轮不新增忘记密码、注册账号、短信验证码登录、扫码登录或第三方登录的真实业务能力。
- 本轮不改造登录接口、JWT、会话治理或权限返回结构。
- 本轮不新增独立的登录页配置中心，也不引入新的公开配置接口。
- 本轮不重做登录页视觉风格，仅调整入口暴露策略与布局默认值。

## Decisions

### 1. 复用 Vben 既有 `authPageLayout` 枚举，统一以 `panel-center` 作为 LinaPro 默认值

登录页位置能力在底层偏好系统中已经存在，支持 `panel-left`、`panel-center`、`panel-right` 三个取值，且 `AuthPageLayout` 已经基于该枚举实现了左侧、居中、右侧三种版式。此次不再发明新的“登录框位置”字段或 CSS 自定义字符串，而是直接沿用这组三态枚举，并将 LinaPro 默认值从 `panel-right` 调整为 `panel-center`。

这样做的原因有两点：
1. 可以直接复用现有布局切换控件与渲染逻辑，避免宿主侧维护一套与 Vben 不一致的映射关系。
2. “默认居中”可以通过修改默认偏好值和宿主系统参数默认值同时达成，语义清晰，后续扩展左右布局也无需新增前端代码结构。

备选方案：
- **方案 A：硬编码居中，不保留枚举配置**。优点是改动少；缺点是无法满足“前端可调整位置、系统参数可配置默认值”的要求。
- **方案 B：新增自定义位置字段再映射到样式类**。优点是字段命名可以更贴业务；缺点是会复制已有能力，并增加宿主与前端适配成本。

### 2. 未实现登录方式同时在“页面入口”和“路由入口”两层收口

仅关闭登录卡片中的按钮还不够，因为当前路由仍然注册了 `code-login`、`qrcode-login`、`forget-password`、`register` 等页面，用户仍然可以通过直接访问 URL 进入这些未落地页面。为避免继续暴露“看起来支持、实际上不可用”的能力，本轮采用双层收口策略：

- 在登录页装配 `AuthenticationLogin` 时显式关闭忘记密码、注册、手机号登录、扫码登录和第三方登录入口。
- 在认证路由层移除或重定向未实现认证页面，让这些地址不再成为正式可访问入口。

这样可以保证从视觉入口和可访问路由两个层面都符合“当前只有账号密码登录能力”的产品语义。

备选方案：
- **仅隐藏按钮**：实现最简单，但仍保留死页面入口，不能满足“还没有这些能力”的治理目标。
- **彻底删除页面文件**：可以最彻底地收口，但会提高未来恢复这些能力时的重复开发成本；因此优先保留实现文件，仅取消当前接入。

### 3. 登录框位置纳入现有 public frontend 配置白名单，新增 `sys.auth.loginPanelLayout`

登录框位置属于未登录页也要读取的展示配置，最适合放入现有 `config.GetPublicFrontend()` 的白名单体系中统一暴露，而不是新增专用接口。新增受保护系统参数键建议为 `sys.auth.loginPanelLayout`，值域限定为 `panel-left`、`panel-center`、`panel-right`，归属 `auth` 配置分组，并随公共前端配置接口一起返回给前端。

前端运行时同步逻辑读取该值后，将其映射到 `preferences.app.authPageLayout`，与现有 `themeMode`、`layout` 等公共前端设置保持同一条生效链路。这样管理员刷新系统参数后，登录页在下一次加载时即可使用新的默认布局。

之所以放在 `auth` 分组而不是 `ui` 分组，是因为该参数只作用于登录/注册类未登录页面，不影响工作台布局；放入 `auth` 更利于语义聚合，也避免与 `sys.ui.layout` 的工作台布局概念混淆。

备选方案：
- **放入 `ui` 分组**：实现上也可行，但容易和后台布局配置混淆。
- **只改前端默认值，不新增宿主参数**：无法满足管理员可通过系统参数调整默认位置的要求。

### 4. 系统参数作为宿主默认值来源，登录页现有前端布局切换控件继续保留

Vben 登录页工具栏已经自带布局切换控件，允许在左/中/右三种布局之间切换。本轮不移除这一能力，而是让宿主系统参数决定登录页初始默认布局。也就是说：

- LinaPro 自带默认值改为 `panel-center`；
- 若宿主 `sys_config` 中维护了 `sys.auth.loginPanelLayout`，则登录页启动时以宿主值为准；
- 登录页工具栏仍可用于前端即时切换布局，便于体验验证和后续扩展。

这样既满足“前端应该有设置项可以调整登录框位置”的现状确认，也满足“默认设置项应由系统参数配置”的治理要求。

### 5. 回归验证覆盖前后端两个平面

这次变更横跨前端页面装配、前端运行时配置同步、后端受保护参数元数据和白名单投影，因此需要至少覆盖两类验证：

- **后端配置单测**：校验新参数的值域验证、白名单投影和默认值输出。
- **前端运行时/页面测试**：校验登录页默认隐藏未实现入口、默认布局为居中，并能根据公共前端配置 payload 应用左右/居中布局值。

这样可以避免后续只改某一端而导致参数存在但页面不生效，或页面默认调整后遗漏后台元数据的问题。

## Risks / Trade-offs

- **[风险] 书签或外链仍指向旧的认证子路由** → **缓解**：对未实现认证路由统一回退到标准登录页，避免出现 404 或误导性页面。
- **[风险] 宿主系统参数与前端本地布局切换都能影响登录页位置，优先级需要稳定** → **缓解**：明确宿主公共前端配置在页面启动时提供默认布局，前端运行期切换只影响当前浏览体验，后续刷新重新以宿主默认值为准。
- **[风险] 新增 public frontend 参数但遗漏初始化 SQL 或内置元数据会导致系统参数页不可维护** → **缓解**：同时更新宿主内置参数清单、参数校验和公共前端输出测试。
- **[风险] 未来重新放开多种登录方式时容易遗漏路由注册** → **缓解**：保留页面文件但以规格明确“当前阶段只接入账号密码登录”，后续能力上线时按新变更重新接入即可。

## Migration Plan

1. 更新宿主 public frontend 参数元数据与默认值，新增 `sys.auth.loginPanelLayout=panel-center`。
2. 扩展公共前端配置白名单响应与前端运行时同步逻辑，将 `auth.panelLayout` 映射到 `preferences.app.authPageLayout`。
3. 调整登录页与认证路由装配，隐藏未实现入口，并把未实现认证页面统一回退到标准登录页。
4. 增加后端配置测试与前端运行时/页面测试，验证默认居中和系统参数切换行为。

## Open Questions

- 当前登录页工具栏中的布局切换结果是否需要在未来迭代中持久化为“用户个人偏好”并覆盖系统默认值？本轮先按“宿主默认值优先于刷新后的展示结果”落地，不扩展到个性化偏好治理。
