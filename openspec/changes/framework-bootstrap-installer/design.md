## Context

这次变更不是单纯补一个下载脚本，而是要为整个 `LinaPro` 仓库建立一个“首次落地入口”。需求同时跨越仓库分发方式、跨平台脚本入口、目标目录安全策略、环境依赖体检和仓库根文档，因此适合先在设计阶段把边界讲清楚，避免后续实现时把“源码下载器”“环境安装器”“项目初始化器”混成一件事。

当前仓库已经有根目录 `hack/scripts/` 用于承载仓库级脚本，但还没有任何“快速获取源码并开始使用”的标准入口。首次接触 `LinaPro` 的用户需要自行决定：是 `git clone`、下载 ZIP、还是复制现有目录；同时还需要自己判断 `Go`、`Node.js`、`pnpm`、`MySQL`、`make` 等依赖是否齐备。这个链路既不统一，也很难在 `macOS`、`Linux`、`Windows` 三个平台上提供一致体验。因此，这次安装入口进一步收敛到 `hack/scripts/install/` 子目录，避免和其他仓库级脚本混在同一层级。

另外，用户给出的目标入口示例是 `curl -fsSL https://linapro.ai/install.sh | bash`。这说明对外希望有一个“单命令启动”的官方分发方式；但就当前仓库实现范围而言，真正需要先落地的是仓库内的脚本和参数契约。官方托管地址应当复用仓库脚本内容，而不是在仓库外维护另一套逻辑，避免版本漂移。

## Goals / Non-Goals

**Goals:**
- 在仓库根 `hack/scripts/install/` 下建立面向整个 `LinaPro` 的快速安装脚本入口，而不是仅面向 `apps/lina-core` 子项目。
- 为 `macOS/Linux` 与 `Windows` 分别提供可直接执行的脚本入口，并保持核心参数语义一致。
- 统一采用 GitHub/Codeload 源码压缩包下载模式，支持仓库地址与 `ref` 参数，避免依赖 `git clone`。
- 支持“解压到当前目录”与“解压到指定目录”两类落地模式，并默认采用安全的非覆盖策略。
- 在安装完成后输出本机环境体检结果与下一步建议，缩短首次上手路径。
- 在根 `README.md` 与 `README.zh_CN.md` 中建立正式的快速安装说明。

**Non-Goals:**
- 本次不负责自动安装 `Go`、`Node.js`、`pnpm`、`MySQL`、`make` 等系统依赖，只做检查与提示。
- 本次不自动执行 `make init`、`make mock`、`make dev` 等会修改本地环境或数据库的命令。
- 本次不在仓库内实现 `linapro.ai` 站点发布基础设施；只约定仓库脚本为官方托管入口的内容源。
- 本次不引入基于 `git clone` 的安装主路径，也不处理凭证型私有仓库登录流程。

## Decisions

### 决策一：采用双入口脚本，而不是尝试用单个 Shell 文件兼容三端

**选择**：在仓库根 `hack/scripts/install/` 下新增两个入口：`install.sh` 负责 `macOS/Linux`，`install.ps1` 负责 `Windows PowerShell`。两者共享一致的核心参数语义，例如仓库、引用、目标目录、是否允许覆盖、是否落地到当前目录等。

**原因**：
- `curl | bash` 适合 Unix-like 环境，但不适合原生 Windows 用户；PowerShell 需要原生下载与解压能力，强行统一到单脚本会把平台差异藏进大量兼容分支里，维护成本更高。
- `Windows` 原生具备 `Invoke-WebRequest` / `Expand-Archive`，`macOS/Linux` 则更适合 `curl` / `wget` + `tar`。拆分入口后，脚本逻辑更清晰，也更容易让用户理解推荐命令。

**备选方案**：
- 单个 `bash` 脚本兼容三端。拒绝，因为这会间接要求 `Windows` 用户安装 `Git Bash` 或 `WSL`，与“主流操作系统开箱即用”的目标不符。
- 单个 `Python` 脚本。未采用，因为首次安装前不能假定用户本机已有满足要求的 `Python` 环境。

### 决策二：源码获取统一走归档下载，不把 `git clone` 作为安装主路径

**选择**：安装脚本默认根据平台从 GitHub/Codeload 下载归档文件，Unix 优先 `tar.gz`，Windows 优先 `zip`，并支持通过参数覆盖仓库与 `ref`。

**原因**：
- 归档下载比 `git clone` 更轻量，不需要仓库历史，也不依赖本机预装 Git。
- `tar.gz` 在 `macOS/Linux` 更通用，`zip` 在 `Windows` 上由系统原生命令支持更好。
- 通过仓库与 `ref` 参数，可以同时支持官方默认仓库、指定分支、指定标签和开发版分发，而无需为不同分支维护多份脚本。

**备选方案**：
- 直接使用 `https://github.com/<repo>/archive/refs/heads/<branch>.zip`。可行，但优先 `codeload.github.com` 更适合做纯下载入口，跳转更少，分发语义更明确。
- 优先 `git clone --depth 1`。拒绝，因为仍然要求 Git 可用，且体积与错误面都更大。

### 决策三：目标目录采用“显式模式选择 + 安全默认值”

**选择**：默认把源码落地到当前工作目录下的新子目录；当用户显式传入“当前目录模式”时，才允许直接把解压结果铺到当前目录；当用户显式传入目标路径时，脚本将源码落地到指定目录。所有模式下，只要目标目录非空，就默认拒绝继续，除非用户显式传入覆盖参数。

**原因**：
- 用户明确提出既要支持当前目录，也要支持指定目录，但如果默认直接把压缩包内容解压进当前目录，风险很高，容易污染已有文件。
- 使用“当前目录新建子目录”作为默认模式，更符合初次安装的安全预期，也能兼容 `curl ... | bash` 这类无交互命令。
- GitHub 归档顶层目录名会随着分支或 tag 变化，脚本需要在临时目录解压后动态识别唯一根目录，再移动到最终位置，不能把 `linapro-main` 之类名称硬编码进逻辑。

**备选方案**：
- 默认直接解压到当前目录。未采用，因为误覆盖风险太高。
- 只支持 `--dir`，不支持“当前目录模式”。未采用，因为不符合用户对“当前目录 / 指定目录”双模式的明确要求。

### 决策四：安装脚本第一版只做“环境体检 + 下一步提示”，不做系统依赖自动安装

**选择**：脚本在源码落地完成后，统一检查 `Go`、`Node.js`、`pnpm`、`MySQL`、`make` 等关键依赖是否存在，并输出版本或缺失提示；但不自动调用 `brew`、`apt`、`yum`、`choco`、`winget` 等包管理器安装依赖。

**原因**：
- 真正阻碍首次上手的常见问题是“不知道自己缺什么”，而不是“必须由安装脚本代装一切”。
- 自动安装系统依赖会引入管理员权限、网络镜像、包管理器差异和企业环境限制，复杂度远高于本次想解决的问题。
- 先把依赖检查和下一步命令指引做好，就已经能显著降低 onboarding 成本，同时保持脚本可预测、可审计。

**备选方案**：
- 自动按平台安装所有依赖。拒绝，因为跨平台差异与风险都过高。
- 完全不检查依赖，只提示“请查看 README”。未采用，因为会让快速安装体验断在一半。

### 决策五：仓库脚本作为唯一事实来源，官方分发 URL 只做薄包装

**选择**：`hack/scripts/install/install.sh` 与 `hack/scripts/install/install.ps1` 作为仓库内唯一的安装逻辑来源；对外公开的 `https://linapro.ai/install.sh` / `install.ps1` 应复用同一套内容，最多只做薄包装或重定向映射，不允许站点托管脚本与仓库脚本长期分叉。

**原因**：
- 安装入口是用户最先接触到的能力之一，若站点副本和仓库脚本逻辑不一致，会放大故障排查成本。
- 把仓库脚本作为单一事实来源，也更符合后续通过 release/tag 发布稳定版本的治理方式。

**备选方案**：
- 网站单独维护更短的 bootstrap 脚本，再由其拼装逻辑。未采用，因为容易形成第二套实现。

## Risks / Trade-offs

- [Risk] 不自动安装依赖时，部分用户仍需手动补环境。→ Mitigation：输出清晰的缺失项、推荐版本与下一步命令，优先解决“看不懂卡点”的问题。
- [Risk] 不同平台的下载与解压工具可用性存在差异。→ Mitigation：Unix 侧优先 `curl`、回退 `wget`，Windows 侧使用 PowerShell 原生命令，并在缺失必要能力时给出明确失败提示。
- [Risk] 当前目录模式容易误伤已有文件。→ Mitigation：将其设计为显式参数触发，并在目录非空时默认拒绝执行。
- [Risk] 官方托管 URL 若发布延迟，可能与仓库脚本版本短暂不一致。→ Mitigation：明确仓库脚本为源头，并在文档与发布流程中约束同步方式。

## Migration Plan

1. 在仓库根 `hack/scripts/install/` 下新增 `install.sh` 与 `install.ps1`，并约定统一的参数模型与输出格式。
2. 实现源码归档下载、临时目录解压、动态顶层目录识别与目标目录落地逻辑。
3. 在安装流程末尾增加环境依赖检查与下一步引导输出。
4. 更新仓库根 `README.md` 与 `README.zh_CN.md`，补充快速安装示例与平台说明。
5. 为脚本补充自动化验证或最小可执行回归，确保参数解析、目录策略与错误提示行为稳定。
6. 后续若需要对外提供 `linapro.ai` 托管入口，可直接复用仓库脚本内容，不再额外定义第二套实现。

## Open Questions

- 第一版是否需要支持 `repo` 之外的企业镜像源参数（如 GitHub Enterprise 或自建镜像）？当前设计先聚焦公开 GitHub 仓库。
- 环境体检结果是否需要输出机器可解析格式（如 `json`），以支持后续安装器 UI 或诊断工具复用？首版可先保留为人类可读文本。
