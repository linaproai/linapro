## Context

当前仓库已经具备插件宿主、动态扩展、前后端一体化工程组织、`OpenSpec` 协同流程和 AI 辅助研发入口，但项目入口说明、系统元数据、接口文案和规范描述仍大量沿用“后台管理系统”心智。这种叙事偏差会影响三个层面：

1. **产品认知偏差**：仓库根规范、系统信息页、登录页、OpenAPI 标题和脚本 banner 会持续向新贡献者暗示 `Lina` 只是一个后台项目，而不是一个可扩展的全栈开发框架。
2. **架构边界偏差**：`lina-core` 已经出现菜单路由投影、树选择器、当前用户工作台启动数据等面向具体前端工作台的聚合输出。如果不显式定义“核心宿主能力”和“工作台适配”的边界，后续页面需求会继续驱动核心接口和 DTO 结构调整。
3. **文档治理偏差**：仓库目前没有根 `README.md`，已有子目录 `README.md` 共 24 份，且缺少统一的中英文镜像约束；现有 Markdown 规范默认中文，也与后续国际化仓库协作目标冲突。

本次变更是一个明显的跨模块治理变更：会同时影响项目规范、活跃 OpenSpec 主规范、前后端元数据、目录级说明文档以及一批用户可见文案，但不以业务功能重构为目标。

## Goals / Non-Goals

**Goals:**
- 统一 `Lina` 的项目定位为“`AI驱动的全栈开发框架`”，并将该口径落到仓库规范、系统元数据和用户可见入口文案。
- 将 `apps/lina-core` 的核心宿主边界上升为顶级要求，明确“通用能力”与“工作台适配输出”的职责分层。
- 为现有工作台导向接口建立明确分类规则，避免后续继续把页面结构直接下沉到核心领域契约。
- 建立全仓库 `README.md`（英文）+ `README.zh_CN.md`（中文镜像）的长期治理规范，并为根目录补齐入口文档。
- 将现有存量 README 纳入渐进式治理清单，确保后续新增目录说明文档统一遵循同一规则。

**Non-Goals:**
- 本次不对现有业务功能做大规模重构，不追求一次性重命名所有菜单、模块或数据库表。
- 本次不立即对现有工作台导向 API 做破坏性路径调整；若接口仍被当前前端使用，则优先通过分类、注释和规范边界收敛。
- 本次不重写 `openspec/changes/archive/` 下的历史提案和设计文档，只调整活跃主规范与当前生效的项目规范。
- 本次不引入自动翻译流水线；中英文 README 采用人工同步维护，优先保证技术准确性和内容一致性。

## Decisions

### 决策一：全仓库统一使用单一定位词“AI驱动的全栈开发框架”

**决定**：所有项目级说明、系统级元数据和用户可见介绍统一使用“`AI驱动的全栈开发框架`”作为 `Lina` 的正式定位；不再使用“管理后台系统”“中后台管理系统”“Admin API”这类把项目整体收缩为后台产品的表述。

**原因**：
- 该表述来自用户明确确认，属于本次变更的最高优先级输入。
- 当前仓库已经落地插件宿主、动态插件、AI 协作研发流程和全栈工程组织，该定位与事实一致。
- 使用单一定义可以减少后续前后端、文档和 OpenSpec 产物之间的漂移。

**备选方案**：
- 使用“AI 驱动、Spec-Driven 的全栈开发框架”作为正式定位：放弃。虽然仓库确实采用 `OpenSpec` 流程，但用户已明确要求正式定位不包含该短语。
- 继续使用“后台管理系统”作为主定位，并在局部补充插件/AI 说明：放弃。这样无法从根本上纠正后续实现与文档心智。

### 决策二：将“默认管理工作台”与“核心宿主服务”明确拆分

**决定**：
- `apps/lina-vben` 及其统一布局、菜单、工作台、系统信息页等能力，统一表述为 `Lina` 的**默认管理工作台**。
- `apps/lina-core` 统一表述为 `Lina` 的**核心宿主服务**，提供通用模块接口能力、组件能力、系统治理能力和插件扩展能力。
- 当接口输出明显服务于具体工作台装配（如菜单路由投影、当前登录用户工作台启动数据、树选择器/下拉选项投影）时，允许其继续存在，但必须被标注为**工作台适配接口**或等价语义，而不是被误写成通用领域能力。

**原因**：
- 这样可以在不打断现有前端联调的前提下，先收敛命名和职责边界。
- 页面结构变化时，团队可以优先在适配层解决，而不是修改通用 service 和领域模型。

**备选方案**：
- 立即重构所有工作台导向 API 的路径和 DTO：放弃。本次是治理收口，不适合同时引入大面积兼容风险。
- 完全不承认工作台适配接口的存在：放弃。当前仓库已经存在这类接口，直接否认会让规范脱离现实。

### 决策三：README 国际化治理采用“英文主文档 + 中文镜像”模式

**决定**：
- 仓库内所有目录级主说明文档统一使用 `README.md` 作为英文主文档。
- 同目录必须提供 `README.zh_CN.md` 作为中文镜像文档。
- 两份文档结构一致、信息一致，仅语言不同。
- 不引入自动翻译脚本；更新 README 时要求人工同步维护双语版本。

**原因**：
- 英文主文档更适合作为国际化项目的默认入口。
- 中文镜像能兼顾当前中文主协作环境。
- 人工同步虽有成本，但比自动翻译更能保证技术名词和上下文准确性。

**备选方案**：
- 只在仓库根目录维持双语，子目录保持单语：放弃。用户已明确要求全仓库 README 都逐步统一。
- 继续以中文 `README.md` 为默认，再单独维护英文副本：放弃。这与国际化入口目标相反。
- 用单一英文 README，不提供中文镜像：放弃。与当前团队协作习惯不符，也不满足本次需求。

### 决策四：README 存量治理采用分阶段落地，而不是一次性重写全部说明文档

**决定**：将 README 治理拆成三层推进：
1. **规则与入口优先**：先更新项目规范、文档规范、OpenSpec 上下文，并补齐根 `README.md` / `README.zh_CN.md`。
2. **用户可见入口对齐**：同步修正系统信息页、登录页、OpenAPI 标题/描述、脚本 banner 等项目级入口文案。
3. **存量 README 渐进治理**：把当前 24 份子目录 `README.md` 纳入统一清单，逐步补齐对应的 `README.zh_CN.md`，并将现有中文内容转写为英文主文档。

**原因**：
- README 存量较多，且分布在插件、脚本、前端包等不同目录，一次性重写会显著扩大 review 面。
- 先补规则和入口，能够立即阻止新的文案漂移；后续再按清单分批收口存量内容。

**备选方案**：
- 一次性完成全部 README 中英文双份补齐：放弃。成本高，且会把本次治理变更做得过大。

### 决策五：活跃主规范只修改当前生效能力，不回写 archive 历史记录

**决定**：
- 对需要改变规范行为或规范表述的能力，新增新的 capability spec 或针对 `openspec/specs/` 写 delta spec。
- 历史归档文档保持原样，仅在当前生效主规范中体现新要求。

**原因**：
- 归档文档的价值是保留当时决策上下文，不适合为了统一措辞而回写历史。
- 当前行为约束应以主规范和项目规范为准，足以指导后续实现和 review。

### 决策六：项目级元数据修改优先以源文件为准，并同步考虑打包副本

**决定**：
- 后端配置与组件信息优先修改 `apps/lina-core/manifest/config/` 下的源文件。
- 若实现阶段需要更新嵌入到二进制中的配置副本，则通过现有打包准备脚本同步刷新 `apps/lina-core/internal/packed/manifest/`，避免直接只改副本。

**原因**：
- 当前 `internal/packed/manifest/` 是构建嵌入产物的准备目录，不是主编辑入口。
- 如果只改副本而不改源，会再次引入配置漂移。

## Risks / Trade-offs

- **[范围膨胀]** README、项目规范、OpenSpec 主规范、前后端文案会同时被触达，容易把治理类变更做成大杂烩 → **Mitigation**：按“规则与入口优先、README 存量分批治理”的顺序拆任务，并在任务清单中单列 README 存量治理阶段。
- **[术语混用]** 在部分页面或 DTO 注释改成“工作台”后，其他残留文案仍可能继续使用“后台” → **Mitigation**：在规范中明确词汇表，并把用户可见入口、OpenAPI、测试断言列为首批必改项。
- **[边界收敛不彻底]** 本次不会立即重构所有工作台导向 API，因此仓库中仍会保留一批历史接口形态 → **Mitigation**：通过新 capability 明确“允许存在，但必须被分类和限界”，为后续迭代继续收口提供依据。
- **[双语 README 漂移]** 若仅新增 `README.zh_CN.md` 而缺少同步规则，后续容易出现中英文内容不一致 → **Mitigation**：将“结构一致、信息一致、同步维护”写入顶级规范，并在后续新增 README 时强制检查。
- **[打包资源遗漏]** 后端 OpenAPI/组件配置若只改源文件但未刷新打包副本，运行时嵌入内容可能仍旧过时 → **Mitigation**：在实现任务中明确调用现有打包准备流程，或至少验证嵌入副本与源配置一致。

## Migration Plan

1. 先更新项目规范、OpenSpec 上下文和新增治理 capability，锁定后续实现与文档的统一口径。
2. 再调整仓库根 README、系统信息页、OpenAPI 元数据、登录页和脚本 banner 等高曝光入口。
3. 最后按清单渐进治理子目录 README，并同步补齐对应中文镜像文档。
4. 本次变更不涉及数据库迁移和接口路径破坏性调整；若中途发现术语调整影响过大，可通过回退文案与规范改动直接撤销，不存在数据回滚问题。

## Open Questions

- 当前无阻塞性开放问题；README 存量治理的批次划分将在 `tasks.md` 中显式拆解，而不是额外保留为设计未决项。
