## Context

当前插件体系已经具备`capability.Services`、`pluginhost.Services`、动态`hostServices`协议、`data`服务、`orgcap`、`tenantcap`和`ai`能力族，但官方源码插件中仍存在直接生成或读取宿主`sys_*`核心表的生产路径。典型问题包括插件生成`sys_user`、`sys_dict_data`、`sys_online_session`、`sys_plugin`等`DAO/DO/Entity`，以及在插件服务中直接按宿主表完成展示名称、候选搜索、会话投影、字典标签、插件状态或租户关系查询。

这类做法把插件契约绑定到宿主存储模型，绕过宿主统一数据权限、租户边界、缓存失效和接口性能治理。由于本项目没有历史兼容负担，本次变更采用一次性迁移和强治理方案，不保留旧接口、旧`host service`方法或旧`DAO`生成范围作为生产兼容入口。

## Goals / Non-Goals

**Goals:**

- 建立插件访问宿主数据的统一领域能力模型，所有插件只能通过领域接口、源码插件`pluginhost.Services`或动态插件`hostServices`协议访问宿主业务数据和治理状态。
- 一次性定义并落地`usercap`、`authzcap`、`dictcap`、`filecap`、`sessioncap`、`configcap`、`notifycap`、`plugincap`、`jobcap`、`infracap`，并按相同模型重整`orgcap`、`tenantcap`和`ai`。
- 清零插件生产代码对宿主`sys_*`核心表、宿主`DAO/DO/Entity`、宿主内部 service 和官方能力插件表的直接依赖。
- 将动态`data`服务收窄为当前插件自有表访问，并在 manifest 校验、运行时授权和治理扫描中阻断核心表授权。
- 用`CapabilityContext`统一表达`pluginID`、actor、tenant、调用来源、系统调用标识、授权快照和审计信息。
- 让源码插件通过`pluginhost.Services.Admin()`直接获得类型化管理目录，动态插件通过安装授权后的`hostServices`方法调用管理能力，同时保留领域数据权限、租户边界、状态机、数量上限和审计校验。
- 建立关键运行时数据的统一缓存一致性策略，使用共享修订号和事务后失效覆盖权限、租户、插件状态、字典、组织树和运行时配置。

**Non-Goals:**

- 不提供宿主核心表的通用表查询、通用写入、通用 SQL 或跨插件表白名单能力。
- 不保留旧`orgcap`、旧`tenantcap`、旧`ai`根方法、旧动态`host service`方法或插件侧宿主表`DAO`生成作为生产兼容层。
- 不把具体管理工作台页面的列、筛选项或路由装配形状固化到`apps/lina-core`核心领域契约中。
- 不在本变更中改变项目定位、插件目录基本结构或源码插件与动态插件的共同后端分层要求。

## Decisions

### 领域能力作为唯一宿主数据边界

插件需要的是用户、授权、组织、租户、字典、会话、文件、通知、插件状态和任务等领域能力，不是宿主表行。因此所有宿主核心表和官方能力插件表都视为领域 owner 私有事实；插件公共契约只暴露领域接口、输入输出`DTO`、投影结构和稳定错误语义。

替代方案是继续扩展`data`表授权或新增表级白名单。该方案实现短期简单，但会持续泄漏宿主存储结构，并且难以强制数据权限、缓存失效和批量装配边界，因此拒绝。

### 一次性全领域契约落地

本变更一次性建立全部领域能力契约和治理规则，任务按领域拆分执行。这样可以统一`Service`、`AdminService`、`ScopeService`、`Provider`、`StatusReporter`、领域`ID`类型、动态方法名、错误码和缓存策略，避免每迁移一个插件就重新定义一套局部接口。

替代方案是只先做`usercap`、`authzcap`和`dictcap`。该方案风险较低，但会让会话、插件状态、通知、任务和配置继续保留旧通道，导致治理扫描无法形成单一规则。

### 源码插件管理能力采用类型化完整目录

源码插件是随宿主编译嵌入的受信任扩展，管理能力通过`pluginhost.Services.Admin()`提供完整类型化`AdminService`目录，不再维护字符串式管理能力声明。插件业务服务仍应通过构造函数接收最窄接口，避免把整个目录长期保存在业务对象中。

`AdminService`方法不做额外插件级字符串授权校验，但必须执行`CapabilityContext`解析、租户边界、目标数据边界、状态机、数量上限、系统 actor 和审计治理。后台任务、生命周期和 provider 回调等无用户场景必须使用宿主创建的系统 actor，插件不得自行伪造。

### 动态插件安装授权替代插件级菜单/RBAC 方法校验

动态插件通过`plugin.yaml hostServices`声明领域服务和方法，安装或启用阶段确认授权后形成运行时授权快照。运行时调用先校验快照中的`service + method + resource/projection`，不再额外校验当前用户是否拥有某个工作台菜单或按钮权限。

这不等于绕过领域安全边界。领域方法仍必须校验租户、数据可见性、目标资源约束、状态机、批量上限、存在性侧漏策略和审计来源。这样可以让插件级授权模型简单稳定，同时满足数据权限规则要求。

### 插件可见`ID`使用领域命名类型并以字符串编码

插件契约不得暴露数据库自增主键类型作为长期边界。每个领域定义自己的命名`ID`类型，动态协议统一编码为字符串；领域实现内部可以映射到数据库主键、业务键或组合键。

替代方案是继续暴露`int`主键。该方案方便当前实现，但会把插件契约绑定到宿主表结构，并影响后续跨存储、迁移和组合资源表达。

### `data`服务只访问当前插件自有表

动态`data`服务的表名必须匹配当前插件自有表命名空间。`sys_*`表永久拒绝；官方能力插件表也不得通过`data`服务授权给其他插件，必须由所属领域能力发布投影或命令。

测试、Mock 数据、安装 SQL 和迁移 SQL 可以作为受控例外接触宿主表，但必须限定在测试或安装治理职责中，不进入插件生产运行路径。

### 批量缺失语义隐藏不可见原因

`BatchGet*`返回可见`Items`和不可解释的`MissingIDs`，不区分真实不存在和不可见。命令场景必须使用`Ensure*`，默认任一不可见、不可用或越权目标导致整体失败，除非领域规范显式定义逐项处理、响应结构和审计方式。

### 缓存一致性采用共享修订号和事务后失效

权限、租户成员、插件状态、字典、组织树、运行时配置等关键运行时数据统一采用共享修订号与事务后失效。单机模式可以使用本地缓存和本地失效，但仍走同一修订号抽象；集群模式接入共享后端、事件广播或等价协调机制。失效必须幂等、可重试、可观测，并具备回源重建或明确能力不可用错误。

### `i18n`标签语义由领域能力稳定返回

领域能力默认返回稳定值和`labelKey`。当调用方确实需要`label`时，领域能力按当前请求 locale 解析，同时保留`labelKey`。字典、菜单、权限、插件资源和错误消息的用户可见文本必须遵守宿主和插件的`i18n`启用边界。

## Risks / Trade-offs

- [Risk] 一次性全领域契约范围大，容易出现任务上下文过长或局部实现漂移。→ Mitigation：任务按契约、协议、领域适配、插件迁移、治理扫描和验证分层拆分；实现时先定契约，再迁移调用方。
- [Risk] 动态插件管理方法安装授权后可能被误解为绕过数据权限。→ Mitigation：规范明确安装授权只替代插件级菜单/RBAC 方法校验，领域方法仍强制执行租户、数据可见性、目标状态和审计校验。
- [Risk] 清零插件`sys_*`生产访问会影响多个官方插件并导致短期编译失败。→ Mitigation：先建立领域契约和宿主适配器，再逐插件迁移`backend/hack/config.yaml`、生产代码和测试替身，最后运行治理扫描。
- [Risk] 领域`ID`字符串化需要迁移现有整型 ID 调用路径。→ Mitigation：领域边界使用命名类型，适配器内部集中完成字符串 ID 与数据库主键映射，不让插件侧参与转换。
- [Risk] 缓存失效统一到共享修订号会改变部分本地缓存行为。→ Mitigation：保留单机本地实现分支，但使用相同修订号接口，并补充单机与集群分支测试。
- [Risk] 新增治理扫描可能依赖平台脚本。→ Mitigation：扫描能力使用 Go 工具或`linactl`跨平台入口实现，根`Makefile`和`make.cmd`只作为薄包装。

## Migration Plan

1. 定义`CapabilityContext`、领域`ID`类型、公共错误码、批量缺失语义、`labelKey`/`label`语义和领域接口方法族。
2. 新增或重整领域契约包，先完成`Service`、`AdminService`、`ScopeService`、`Provider`和`StatusReporter`边界，不访问数据库。
3. 扩展`pluginhost.Services`和`AdminServices`目录，确保源码插件普通消费面和管理面分离。
4. 扩展动态`hostServices`描述符、能力推导、安装授权快照、WASM host service 分发和 guest 目录。
5. 将动态`data`服务 manifest 校验和运行时执行收窄为当前插件自有表，并拒绝核心表与官方能力插件表。
6. 按领域实现宿主适配器，使用投影查询、批量查询、数据库侧权限过滤、共享缓存修订号和事务后失效。
7. 迁移所有官方插件生产代码，删除插件`backend/hack/config.yaml`中宿主核心表生成项并重新生成 DAO。
8. 增加治理扫描，阻断生产代码中宿主表 DAO、直接`sys_*`查询、旧领域接口和旧动态方法。
9. 运行变更范围编译、协议、权限、缓存、批量性能和治理扫描验证。

本项目不需要兼容旧生产接口，因此回滚策略是回滚整个 OpenSpec 变更分支；不提供新旧接口并行开关。
