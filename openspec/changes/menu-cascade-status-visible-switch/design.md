## Context

菜单更新当前只写入目标行的 `status`/`visible`，子级保持原值。列表页通过字典渲染展示状态，无行内切换。插件管理、角色管理、用户管理等页面已有 `Switch` 行内切换先例。

## Goals / Non-Goals

**Goals:**

- 写入 status 或 visible 时，后代节点在同一事务内同步收敛到相同目标值
- 列表页可通过开关快速切换状态与显示，交互对齐插件管理页
- 保持现有 `PUT /menu/{id}` 契约，不新增专用状态接口
- 成功后失效权限拓扑缓存并刷新前端可访问菜单

**Non-Goals:**

- 不改变菜单删除级联语义
- 不新增独立状态/显隐 REST 资源
- 不引入新的字典类型或 i18n 键

## Decisions

1. **级联方向双向**
   - 选择：写入 `status` 或 `visible` 时，将该字段目标值级联到全部后代（含启用/显示）
   - 原因：管理员将父级停用再启用时，期望子树整体恢复；仅负向级联会造成“父开子关”的困惑
   - 备选：仅负向级联 — 已按用户反馈否决

2. **在既有 Update 服务内完成级联**
   - 选择：`Update` 成功写入当前行后，若负向变更则批量 `WhereIn` 更新后代对应字段
   - 原因：抽屉全量编辑与列表开关走同一路径，语义一致；`getDescendantIds` 已存在
   - 备选：新增 `UpdateStatus`/`UpdateVisible` API — 否决，增加契约面且价值有限

3. **列表开关复用 `menuUpdate` 部分字段**
   - 选择：放宽 `UpdateReq` 的 `name`/`type` 必填约束；开关只提交目标 `status` 或 `visible`
   - 原因：列表返回的 `name` 可能是本地化展示标题，回写会污染治理数据；专用状态接口收益有限
   - 备选：开关先 `menuInfo` 再全量更新 — 否决，多一次请求且仍易误写其他字段

4. **事务边界**
   - 选择：当前行更新与后代批量更新放在同一数据库事务中，失败整体回滚
   - 原因：避免父级已停用而子级仍启用的中间态

5. **前端刷新**
   - 选择：开关成功后 `tableApi.query()` + `refreshAccessibleState`，失败回滚本地开关值
   - 原因：与编辑/删除后刷新链路一致，保证侧栏即时收敛

## Risks / Trade-offs

- **[Risk] 深层菜单树批量更新成本** → 使用一次后代 ID 收集 + 单次 `WhereIn` 更新，避免 N+1 逐行写
- **[Risk] 列表开关误触** → 与角色/插件页一致采用即时切换；失败回滚并提示错误
- **[Risk] 现有 E2E 用 `.ant-switch.first()` 定位级联删除开关** → 同步收紧选择器，避免与行内开关冲突

## Migration Plan

- 无数据迁移；上线后仅对新的停用/隐藏操作生效
- 历史“父停用/隐藏而子仍启用/显示”的数据不自动回填；管理员再次切换父级即可收敛

## Open Questions

- 无。启用/显示不级联已按用户描述与常见治理习惯确定。
