## Context

当前 E2E 套件已经通过 `hack/tests/config/execution-manifest.json` 将部分文件放入串行队列，并通过 `run-suite.mjs` 将 full/smoke/module 拆分成并行阶段和串行阶段。完整回归中仍暴露出一个关键缺口：测试文件虽然能被分到串行或并行，但缺少可审查的全局状态变更分类，也缺少对运行时缓存、插件生命周期、系统参数、字典、菜单权限等共享状态的治理说明。

本变更针对测试基础设施和测试用例治理，不改变业务接口和产品运行时行为。需要继续遵守现有 E2E 命名规则、模块目录边界、OpenSpec 任务追踪规则和 i18n 持续治理要求。

## Goals / Non-Goals

**Goals:**

- 将“全局共享状态”定义为可分类、可校验、可审查的 E2E 执行属性。
- 让运行脚本和校验脚本能够发现高风险共享状态用例没有进入串行边界的问题。
- 让插件、mock 数据、语言包缓存、系统参数、字典和菜单权限相关测试具备稳定的前置条件和清理策略。
- 将缓存/ETag 类测试改为协议语义验证，允许全量回归期间由于合法全局版本刷新而返回新资源。
- 形成文档化的测试冲突案例和治理规则，便于后续新增用例时复用。

**Non-Goals:**

- 不重写现有 Playwright 框架，不引入新的测试运行器。
- 不改变业务模块的真实数据模型、API 契约或运行时 i18n 行为。
- 不为了测试便利关闭插件生命周期、缓存失效、权限同步等真实业务机制。
- 不要求所有 E2E 都串行执行；并行能力仍然保留给无全局状态竞争的文件。

## Decisions

### 决策 1：在执行清单中引入全局状态分类

在 `execution-manifest.json` 中为串行条目或测试文件维护机器可读分类，例如 `pluginLifecycle`、`runtimeI18nCache`、`systemConfig`、`dictionaryData`、`permissionMatrix`、`sharedDatabaseSeed`。运行脚本继续使用串行文件集合执行，但校验脚本额外检查分类完整性。

备选方案是仅靠文件路径或人工约定判断串行边界。该方案成本低，但无法解释为什么某个文件必须串行，也无法在新增高风险用例时自动提醒，因此不采用。

### 决策 2：用静态启发式检测补充人工分类

`validate-e2e.mjs` 应扫描高风险 API、helper 或关键词，例如插件 `install/enable/disable/uninstall/sync`、系统参数写入、字典导入/修改、菜单权限修改、运行时语言包缓存断言、`localStorage` ETag 缓存等。若命中高风险模式但不在串行边界或未声明分类，校验应失败并给出修复建议。

启发式不追求完美语义分析，只承担防漏的工程门禁。复杂误报允许通过显式分类或 allowlist 处理，但每条 allowlist 必须说明原因。

### 决策 3：插件和 mock 数据前置条件统一 fixture 化

依赖源码插件或动态插件状态的测试必须通过统一 fixture/helper 准备插件状态。fixture 需要做到幂等安装、启用、必要 mock SQL 加载和前端插件投影刷新。测试文件不应隐式依赖其他文件先安装插件，也不应直接假设某个插件数据表或 mock 数据一定存在。

这会增加少量前置步骤时间，但能换来单文件可运行和全量回归稳定性。

### 决策 4：缓存类测试验证协议语义而不是固定响应码

运行时 i18n ETag、public config 缓存、插件前端资源代际等测试应验证“请求是否携带正确条件头”和“服务端响应是否符合当前资源版本”。例如带旧 ETag 重载时，`304` 表示版本未变，`200 + 新 ETag + body` 表示版本已合法刷新，两者都可正确。

只有在测试本身独占资源版本且无并发全局状态变化时，才断言固定 `304`。

### 决策 5：业务状态断言优先使用稳定字段

测试业务数量、状态和权限时优先使用 API 中的稳定字段、ID、code、labelKey、permission key，而不是从本地化 UI 文本反推业务状态。UI 文案仍要测试，但应作为单独的展示断言，不参与跨语言业务状态计算。

这能避免语言切换、繁简体切换或文案调整导致业务测试误判。

## Risks / Trade-offs

- [Risk] 串行分类过宽导致全量回归变慢 → Mitigation：分类只覆盖真实全局状态变更；只读审计、局部唯一数据和无共享状态文件继续并行。
- [Risk] 启发式检测误报 → Mitigation：支持带原因的 allowlist，并在审查中要求说明为什么并行安全。
- [Risk] fixture 自动加载 mock SQL 可能掩盖缺失前置条件 → Mitigation：fixture 只处理测试所需 demo/mock 数据，并保持 SQL 幂等；业务安装 SQL 仍通过插件生命周期验证。
- [Risk] 缓存类测试接受 `200` 和 `304` 后变松 → Mitigation：`200` 分支必须断言新 ETag 与旧 ETag 不同且响应体存在，仍然验证协议正确性。
- [Risk] 新增分类和校验后短期需要整理既有用例 → Mitigation：分批迁移，先覆盖已知冲突文件和高风险模块，再逐步收敛其它命中项。
