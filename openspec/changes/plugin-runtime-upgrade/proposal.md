## Why

当前插件文件更新和运行时插件状态升级被混在一起描述，导致旧开发期升级入口、启动阻断和运行时数据迁移边界不清。源码插件或动态插件的新版本文件可以在开发阶段离线覆盖，但数据库中的有效版本、治理资源、运行时状态和插件自有数据只能在宿主启动后通过显式运行时升级流程处理。

## What Changes

- 新增插件运行时升级能力：宿主启动扫描插件文件元数据与数据库有效元数据，只标记状态，不自动升级，也不因待升级状态阻断宿主启动。
- 区分插件有效版本和发现版本：数据库有效版本低于文件版本时标记为待升级；数据库有效版本高于文件版本时标记为异常，要求人工修复。
- 插件管理页新增待升级和异常状态展示：待升级插件的安装动作切换为升级动作，升级前展示升级内容；异常插件展示人工修复入口或诊断信息。
- 新增运行时升级 API 和服务编排，显式执行依赖预检、插件自定义升级回调、升级 SQL、治理资源同步、版本切换、缓存失效和集群通知。
- 扩展源码插件生命周期接口，新增升级请求参数，至少包含升级前 manifest 快照和目标 manifest 快照，供插件执行自定义数据迁移、状态清理和兼容处理。
- 删除旧 lifecycle guard 与 `Can*` 插件契约，统一使用 `Before*`/`After*` 生命周期回调模型，支持安装、升级、禁用、卸载、租户禁用、租户删除和安装模式切换等操作在执行前被插件拦截。
- 移除旧的未实现开发期升级指令语义：不得继续把旧开发期升级入口作为源码插件或框架升级的规范入口。

## Capabilities

### New Capabilities

- `plugin-runtime-upgrade`: 定义插件文件覆盖后的运行时升级状态、管理 API、管理页面行为、升级编排、生命周期回调和集群一致性要求。

### Modified Capabilities

- `plugin-upgrade-governance`: 将源码插件升级从开发期命令和启动阻断模型修正为运行时显式升级模型，并统一动态插件升级边界。
- `source-upgrade-governance`: 删除旧开发期升级指令要求，保留框架元数据展示要求，避免引用未实现指令。
- `plugin-workspace-management`: 明确 `plugins.install` / `plugins.update` 只负责开发阶段离线文件覆盖，不执行运行时数据库、治理资源或插件状态升级。

## Impact

- 影响插件管理 API：需要新增运行时升级请求、升级预览/详情字段，并扩展插件列表返回的运行时状态、有效版本和发现版本。
- 影响插件管理前端：需要根据 `pending_upgrade`、`abnormal`、`upgrade_failed` 等状态展示不同操作和提示。
- 影响插件启动扫描与同步：启动阶段需要从 fail-fast 校验调整为状态标记和受控业务入口策略。
- 影响源码插件接口：需要新增可选升级回调和统一生命周期前置回调注册方式，并移除旧 lifecycle guard 与 `Can*` 兼容层。
- 影响动态插件运行时：需要将上传/文件覆盖后的新版本纳入同一运行时升级状态模型，避免自动切换有效版本。
- 影响缓存和集群：升级成功后必须使用显式作用域失效、共享修订号或事件广播同步插件运行时状态、菜单权限、i18n 和路由缓存。
- 影响测试：需要新增后端单元测试覆盖版本漂移状态和升级编排，新增或更新插件管理 E2E 覆盖待升级、异常和升级确认流程。
