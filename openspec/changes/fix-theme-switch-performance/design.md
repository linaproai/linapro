## Context

当前主题按钮在支持`View Transition API`的浏览器中捕获整个文档的新旧状态，并把主题响应式更新放进新快照生成前的关键路径。主题模式变化还会重复生成约八十个颜色变量、切换`Ant Design Vue`全局算法、销毁并重建所有活动`ECharts`实例，以及让`VXE`重复刷新语言状态。验证表明现有`defu`合并已保持未更新顶层分支的对象引用，因此该部分只需建立回归测试，无需重写合并算法。

该问题属于后台管理端共享前端能力，影响分析页、表格页以及未来复用相同主题基础设施的插件页面。实现不得通过设备性能判断或运行时降级掩盖问题，必须移除高成本根路径，并保持主题语义、主题持久化和现有组件视觉一致性。

浏览器本地偏好属于单浏览器状态，权威数据源是运行时`PreferenceManager.state`。`preferences-theme`与`preferences-locale`是首屏加载提示值，完整`preferences`是持久化快照；它们不涉及服务端集群或跨实例同步。

## Goals / Non-Goals

**Goals:**

- 保留原有圆形主题揭幕视觉及其方向：亮转暗从左下向右上显现，暗转亮从右上向左下展开。
- 保持活动图表实例和表格语言状态稳定，只更新与主题真正相关的状态。
- 保持偏好分支引用稳定，并将主题变量更新限制到本次变更字段。
- 保持用户显式主题优先级、立即刷新恢复和公共前端默认值语义不变。
- 通过单元测试与`E2E`建立确定性的性能回归门禁。

**Non-Goals:**

- 不修改主题配色、布局结构、业务页面内容和用户可见文案。
- 不修改后端接口、数据库、权限、插件协议或第三方依赖版本。
- 不引入基于设备能力、帧率或超时的兜底分支。

## Decisions

### 1. 保留圆形揭幕的双向方向

支持`View Transition API`且用户未要求减少动态效果时，主题按钮继续从按钮中心启动圆形揭幕，鼠标和键盘触发使用相同圆心。亮转暗时把`::view-transition-old(root)`置于上层，并从覆盖视口的圆收缩到按钮位置，视觉上让暗色从左下向右上显现；暗转亮时把`::view-transition-new(root)`置于上层，并从按钮位置扩展到覆盖视口，视觉上让亮色从右上向左下展开。两者均使用原有`450ms ease-in`节奏。

动画结束后由原生`View Transition`生命周期自然清理快照，不再调用`skipTransition()`强制跳过尾帧，避免合成层在结束边界闪烁。

模式更新开始时，根节点进入持续一个绘制周期的主题事务；该事务暂停按钮以外页面内容已有的`transition`，保证新快照中的组件颜色在同一帧落到目标主题。按钮自身保留原有太阳、月亮遮罩及光束动画；减少动态效果偏好下直接切换主题并关闭局部动画。

主题按钮为每次点击分配递增序号，并以尚未提交的目标主题计算下一次点击。旧的快照回调与`ready`回调发现序号已过期后不再提交状态或启动动画，保证重叠点击最终落到最后一次用户意图。

用纯色遮罩模拟揭幕无法逐步呈现目标主题的真实内容，视觉与原交互不同，因此不采用。主题更新关键路径通过定向变量更新、实例复用和持久化拆分收窄，原生快照只承担用户要求的揭幕视觉。

### 2. 使用`ECharts 6`动态主题并同步恢复模型

活动图表首次完整渲染时不调用`clear()`，而是通过`setOption(full, { notMerge: true })`建立`ECharts 6`动态主题恢复所需的完整基线，并在应用侧保存一份完整原始业务配置`canonicalOptions`。`updateData()`只接受“已有系列、稳定且唯一的显式`id`、仅包含`id`与`data`”的增量更新，并直接把最新数据覆盖进`canonicalOptions`，因此内存和主题切换耗时不随历史更新次数增长。任意结构变化必须通过完整配置和`notMerge: true`替换基线，含糊补丁会明确报错。

主题切换时，先把`canonicalOptions`和目标主题覆盖以`notMerge: true`、`lazyUpdate: true`提交为新的恢复基线，再立即调用`setTheme()`；`ECharts 6`消费待处理更新并完成唯一一次目标主题更新。随后通过公开的`ZRender`元素动画接口把本次换肤创建的动画推进到终态，并在新快照捕获前统一`flush()`一次。模型始终保留业务自身的`animation`配置，主题切换后不再通过第二次`setOption()`恢复动画。`getOption()`包含主题默认项、组件默认项和交互状态，不能作为业务配置回灌源；通用补丁也无法按`id`、`name`和剩余索引三阶段规则安全压缩，因此不采用历史日志或近似顶层合并。整个过程保持同一个实例，不调用`clear()`、`dispose()`、`init()`或`resize()`。已被`KeepAlive`停用的图表不进入切换关键路径，重新激活时只应用一次当前主题。

显式拼装所有图表颜色会把主题知识扩散到每个业务图表，维护成本更高。项目已固定使用支持动态主题的`ECharts 6`，因此采用实例级主题接口，并仅复用共享适配器已经缓存的业务配置恢复模型。

### 3. 按主题补丁收窄副作用

保留`updatePreferences()`现有的分支级`defu`合并，并用测试锁定未变化分支的引用稳定性。主题更新器同时接收完整当前主题和本次主题补丁，以补丁字段决定`DOM`更新范围。

模式变化只切换根节点`.dark`。仅当内置主题的明暗主色确实不同，或颜色字段、本地主题类型发生变化时生成对应色阶；圆角和字体大小分别按自身字段更新。颜色变量直接更新到根节点已有自定义属性，避免替换整张运行时样式表。

### 4. 分离共享组件监听职责

`VXE`使用两个独立监听器：主题监听器只调用`setTheme()`，语言监听器只加载和设置语言。`Ant Design Vue`继续根据实际主题和设计变量更新，因为这是正确渲染所必需的工作。

### 5. 缩短点击关键路径上的持久化

主题按钮在调用原生`startViewTransition()`前同步写入目标主题提示值和用户显式选择标记；运行时主题仍只在快照更新回调中提交。这样即使点击后立即刷新，也不会在回调尚未执行时恢复旧主题。常规主题模式和语言更新同样同步写入提示值，完整偏好快照使用现有`150ms`防抖保存。防抖保存只写完整快照键，不回写主题或语言提示，避免更早排队的旧状态覆盖最新点击意图；重置操作也通过同一个同步提示路径提交。加载完整快照时，专用主题和语言提示值具有更高优先级，从而消除防抖窗口内刷新读取旧快照的风险。

该一致性模型是单浏览器最终一致：专用提示值立即一致，完整快照最大陈旧时间为`150ms`。页面刷新会由提示值恢复，不需要服务端或集群失效机制。

## Risks / Trade-offs

- `[风险]`业务图表显式配置的颜色可能覆盖动态主题默认值。→ 保持现有业务配置语义，并通过三态画布像素断言和分析页截图验证亮色、暗色及切回亮色图表可读性。
- `[风险]`主题补丁与完整状态含义混淆可能遗漏初始化变量。→ 初始化继续传入完整主题补丁，增量更新传入原始补丁，并补充引用稳定性和定向变量单元测试。
- `[风险]`专用主题提示与完整快照短暂不同。→ 加载时始终以专用提示覆盖完整快照，并测试立即刷新场景。
- `[风险]`原生文档快照仍有固定合成成本。→ 把主题回调内的变量、图表、表格和持久化工作收窄，并用`E2E`记录回调耗时，防止业务计算重新进入动画关键路径。
- `[风险]`连续快速点击产生重叠原生快照。→ 使用递增序号使过期回调失效，并通过单元测试锁定“最后一次意图生效”。

## Migration Plan

1. 先替换共享主题状态与组件更新路径，并运行单元测试。
2. 再恢复整页圆形揭幕的双向快照层级与方向，新增分析页主题切换`E2E`。
3. 运行类型检查、相关单元测试、`E2E`截图审查和`OpenSpec`严格校验。
4. 回滚时按文件恢复本变更；无需数据迁移或缓存清理，现有本地偏好键保持兼容。
