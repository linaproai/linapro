## Context

当前上传默认值已分散在多个层次：`apps/lina-core/manifest/sql/007-config-management.sql` 将 `sys.upload.maxSize` 初始化为 16，`apps/lina-core/manifest/config/config.template.yaml` 也暴露 16，而 `apps/lina-core/internal/service/config/config_upload.go` 的静态回退值仍是 10。上传链路会同时使用运行时配置、配置文件回退和请求体保护中间件，因此默认值不一致会让 fresh init、未覆盖配置运行和错误提示出现不同表现。项目当前是全新项目，本次调整无需兼容旧迁移链路，可直接整理现有默认来源并通过重新初始化验证。

## Goals / Non-Goals

**Goals:**
- 将宿主上传大小默认值统一到 20MB。
- 让数据库初始化、配置模板、静态回退和上传校验链路对同一默认值达成一致。
- 确保相关自动化测试和用户可见错误提示同步更新。

**Non-Goals:**
- 不改变 `sys.upload.maxSize` 继续被管理员修改的能力。
- 不重新设计文件上传模块或新增其他运行时参数。
- 不改变非 `multipart` 请求仍使用既有请求体保护策略的范围。

## Decisions

### 1. 直接整理现有默认来源，而不是新增兼容迁移层

本次变更直接修改现有宿主初始化 SQL、配置模板和后端静态回退值，使 20MB 成为唯一默认基线。这样可以与“项目为全新项目、允许直接修改现有 SQL 并重新初始化”的约束保持一致。

- 备选方案：新增一个新的 SQL 迭代文件只覆盖默认值。
- 不采用原因：该方案会让默认值来源继续分散，也不符合当前项目无需为历史兼容额外保留迁移层的前提。

### 2. 以 `sys.upload.maxSize` 为统一业务语义，并让所有 fallback 与之对齐

`sys.upload.maxSize` 已是宿主上传大小治理的运行时参数，因此 20MB 的默认值不仅要写入配置管理初始化数据，也必须同步到配置模板和 `config_upload.go` 的静态回退值，确保数据库未覆写、配置文件直读和上传链路校验看到的是同一个默认值。

- 备选方案：仅修改初始化 SQL 中的默认值。
- 不采用原因：仅改 SQL 会让未初始化或未覆写配置场景继续落回 10MB/16MB，无法真正消除分裂。

### 3. 同步更新用户可见错误提示与派生产物

上传超限时的友好错误提示、请求体大小保护相关断言，以及任何由 manifest 派生出的嵌入/打包产物都必须同步更新到 20MB。这样可以避免源码已调整但构建结果或测试基线仍保留旧值。

- 备选方案：只改源码，不检查派生产物与基线断言。
- 不采用原因：这会留下构建产物与测试基线不一致的隐患，容易在后续回归中再次暴露旧默认值。

## Risks / Trade-offs

- [Risk] 派生打包产物仍残留旧默认值 -> Mitigation: 实现时同步更新或重生成嵌入资源，并同时检查源码与派生产物中的默认值。
- [Risk] 只调整一种默认来源会继续产生分裂行为 -> Mitigation: 实现时逐项校对 SQL seed、配置模板、静态回退常量和错误提示断言。
- [Risk] 已手工修改 `sys.upload.maxSize` 的本地环境不会自动变为 20MB -> Mitigation: 本次变更以统一默认值为目标，验证采用重新初始化或显式重置配置的干净环境。

## Migration Plan

- 直接修改现有默认来源，并在本地通过重新初始化宿主数据验证 `sys.upload.maxSize` 的初始值为 20。
- 更新并执行受影响的后端自动化测试，确认上传校验和友好错误提示都以 20MB 为默认基线。
- 若需回滚，只需恢复相关默认值并重新初始化验证环境，无需引入额外升级元数据。
