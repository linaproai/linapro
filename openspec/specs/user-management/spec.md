# 用户管理

## 目的

定义宿主用户管理模块对 `org-center` 可选组织能力的查询、维护、角色关联和协作规则，确保组织插件启用或缺失时用户管理都能稳定工作。

## 需求
### 需求：用户列表查询

系统 SHALL 提供用户列表分页查询接口，支持多字段排序、增强条件筛选和角色信息聚合。当 `org-center` 已安装并启用时，系统额外支持按部门筛选并返回部门字段；当插件缺失时，宿主忽略组织扩展筛选并保持用户列表主功能可用。用户列表 SHALL 接入宿主数据权限治理：全部数据范围可查询所有用户；本部门数据范围仅查询当前用户所在部门范围内的用户；仅本人数据范围仅查询当前用户自己的用户记录。

#### 场景：组织插件可用时按部门筛选用户列表

- **当** `org-center` 已安装并启用，且查询时传入 `deptId` 时
- **则** 系统通过组织插件提供的组织关系筛选属于该部门的用户
- **且** 返回的用户数据可包含 `deptId` 和 `deptName` 字段

#### 场景：组织插件缺失时查询用户列表

- **当** `org-center` 未安装或未启用，且查询用户列表时
- **则** 系统仍返回用户分页列表和角色信息
- **且** 部门相关筛选和字段被安全忽略或省略

#### 场景：本部门数据范围限制用户列表

- **当** 普通用户角色数据范围为本部门数据
- **且** `org-center` 已安装并启用
- **则** 用户列表仅返回当前用户所属部门范围内的用户
- **且** 即使请求参数未传入 `deptId`，系统仍应用该数据权限边界

#### 场景：仅本人数据范围限制用户列表

- **当** 普通用户角色数据范围为仅本人数据
- **则** 用户列表仅返回当前登录用户自己的记录

#### 场景：本部门范围叠加显式部门筛选

- **当** 普通用户角色数据范围为本部门数据
- **且** 查询参数传入不属于当前用户可见范围的 `deptId`
- **则** 系统返回空列表
- **且** 不返回其他部门用户

### 需求：创建用户
系统 SHALL 提供创建用户接口并始终支持角色关联；当 `org-center` 已安装并启用时，系统额外支持关联部门和岗位；当插件缺失时，这些组织扩展字段不阻塞用户创建。

#### 场景：组织插件缺失时创建用户
- **当** `org-center` 未安装或未启用且管理员创建用户时
- **则** 系统仍成功创建用户并处理角色关联
- **且** 缺少部门和岗位信息不会导致创建失败

### 需求：更新用户信息

系统 SHALL 提供更新用户信息接口并始终支持角色关联；当 `org-center` 已安装并启用时，系统额外支持更新部门和岗位关联；当插件缺失时，这些组织扩展字段不阻塞用户更新。用户更新、状态变更、重置密码和角色关联变更 SHALL 接入宿主数据权限治理，调用方只能变更其数据权限范围内的目标用户；既有内置管理员、当前用户删除保护和事务规则继续生效。

#### 场景：组织插件缺失时更新用户

- **当** `org-center` 未安装或未启用且管理员更新用户时
- **则** 系统仍成功更新用户基本信息和角色关联
- **且** 与部门和岗位相关的字段被安全忽略

#### 场景：拒绝更新范围外用户

- **当** 普通用户角色数据范围为仅本人数据
- **且** 该用户尝试更新其他用户的基本信息、状态、密码或角色关联
- **则** 系统返回结构化数据不可见错误
- **且** 目标用户记录和关联关系保持不变

#### 场景：本部门范围允许更新部门内用户

- **当** 普通用户角色数据范围为本部门数据
- **且** 目标用户位于当前用户可见部门范围内
- **则** 系统可在现有功能权限和保护规则通过后更新该用户

### 需求：查看用户详情

系统 SHALL 提供用户详情查询接口。当 `org-center` 已安装并启用时，返回关联的部门和岗位信息；当插件缺失时，仍返回基本用户信息和角色信息。用户详情 SHALL 接入宿主数据权限治理，调用方只能查看其数据权限范围内的目标用户。

#### 场景：组织插件缺失时查询用户详情

- **当** `org-center` 未安装或未启用且调用 `GET /api/v1/user/{id}` 时
- **则** 系统返回用户的完整基本信息（不含密码）和角色信息
- **且** `deptId`、`deptName`、`postIds` 等组织扩展字段被省略、设为零值或设为空集

#### 场景：拒绝查看范围外用户详情

- **当** 普通用户角色数据范围为仅本人数据
- **且** 调用 `GET /api/v1/user/{id}` 查询其他用户
- **则** 系统返回结构化数据不可见错误
- **且** 不返回目标用户详情

#### 场景：本部门用户详情可见

- **当** 普通用户角色数据范围为本部门数据
- **且** 目标用户位于当前用户可见部门范围内
- **则** 系统返回目标用户详情

### 需求：删除用户

系统 SHALL 支持删除单个用户并完整事务性清理所有关联数据。软删除用户记录、移除组织分配（当 `org-center` 已安装并启用时）和移除 `sys_user_role` 中的条目必须在单个数据库事务内发生。关联清理的任何失败必须导致整个删除回滚。访问拓扑变更通知必须仅在事务成功提交后发出。删除用户 SHALL 接入宿主数据权限治理，调用方只能删除其数据权限范围内的目标用户，且仍不得删除内置管理员或当前登录用户。

#### 场景：删除用户原子清理关联数据

- **当** 调用方删除用户
- **且** 所有清理步骤成功
- **则** 系统软删除用户记录
- **且** 当 `org-center` 已安装并启用时，移除部门/岗位分配
- **且** 移除匹配的 `sys_user_role` 行
- **且** 提交后通知访问拓扑变更

#### 场景：关联清理失败回滚用户删除

- **当** 调用方删除用户
- **且** 组织或 `sys_user_role` 清理在事务内失败
- **则** 用户软删除必须回滚
- **且** 操作返回底层错误
- **且** 不发出访问拓扑通知

#### 场景：拒绝删除范围外用户

- **当** 普通用户角色数据范围为本部门数据
- **且** 目标用户不在当前用户可见部门范围内
- **则** 系统拒绝删除
- **且** 不清理目标用户的任何关联数据

### 需求：用户部门树接口
系统 SHALL 在 `org-center` 已安装并启用时提供用户管理左侧筛选的部门树接口；当插件缺失时，宿主不再暴露组织扩展接口。

#### 场景：组织插件可用时获取用户部门树
- **当** `org-center` 已安装并启用，且调用 `GET /api/v1/user/dept-tree` 时
- **则** 系统返回部门树结构数据，每个节点包含 id、label、children、userCount
- **且** 树的第一级仍可包含 `未分配` 虚拟节点

#### 场景：组织插件缺失时用户部门树不可用
- **当** `org-center` 未安装或未启用时
- **则** 宿主不再将 `GET /api/v1/user/dept-tree` 作为默认用户管理依赖接口暴露
- **且** 用户管理主流程不依赖此接口正常工作

### 需求：用户管理前端部门树筛选
系统 SHALL 仅在 `org-center` 已安装并启用时在用户管理页面左侧显示 `DeptTree` 筛选区域；当插件缺失时，页面降级为全宽用户列表。

#### 场景：组织插件缺失时页面布局降级
- **当** `org-center` 未安装或未启用，且管理员打开用户管理页面时
- **则** 页面不显示 `DeptTree` 组件
- **且** 用户列表区域以单列全宽布局显示

### 需求：用户编辑表单添加部门和岗位字段
系统 SHALL 仅在 `org-center` 已安装并启用时在用户编辑表单中显示部门选择和岗位多选字段；当插件缺失时，这些字段被隐藏。

#### 场景：组织插件缺失时隐藏部门岗位字段
- **当** `org-center` 未安装或未启用且管理员打开用户编辑抽屉时
- **则** 表单中不显示部门字段和岗位字段
- **且** 用户仍可完成基本信息和角色信息的编辑

### 需求：用户列表添加部门名称列
系统 SHALL 仅在 `org-center` 已安装并启用时在用户列表表格中显示部门名称列；当插件缺失时，该列被隐藏。

#### 场景：组织插件缺失时隐藏部门列
- **当** `org-center` 未安装或未启用且管理员查看用户列表表格时
- **则** 表格中不显示 `部门` 列
- **且** 其余核心用户列继续正常显示

### 需求：用户列表角色名称必须与后端本地化角色显示匹配
用户管理列表 SHALL 使用后端返回的角色显示名称，并保持内置角色显示与角色管理在同一语言下一致。

#### 场景：用户列表以英文显示管理员角色
- **当** 管理员以 `en-US` 打开用户管理时
- **则** `admin` 用户关联的管理员角色显示与角色管理相同的英文名称
- **且** 前端不维护基于中文角色名称或角色键的额外映射

#### 场景：角色选择器保持治理语义
- **当** 管理员打开用户创建或编辑表单时
- **则** 角色选择器继续使用后端角色选项数据
- **且** 保存用户角色关系仍提交稳定的角色 ID 而非本地化显示文本

### 需求：用户批量删除

系统 SHALL 提供 RESTful 批量删除端点，在单个请求中删除多个用户，共享与单用户删除相同的保护规则、数据权限边界和原子性。

#### 场景：批量删除成功

- **当** 拥有 `user:remove` 权限的调用方调用 `DELETE /api/v1/user?ids=2,3,4`
- **且** 没有 id 匹配内置管理员或当前登录用户
- **且** 所有目标用户都位于当前调用方数据权限范围内
- **则** 系统在单个事务内软删除所有指定用户
- **且** 原子清理其组织分配和 `sys_user_role` 关联
- **且** 返回成功
- **且** 事务提交后通知一次访问拓扑

#### 场景：批量删除拒绝内置管理员 id

- **当** 调用方调用 `DELETE /api/v1/user?ids=1&ids=2&ids=3`
- **且** id `1` 属于内置管理员
- **则** 整个批量必须以 `bizerr` `CodeUserBuiltinAdminDeleteDenied` 被拒绝
- **且** 不删除任何用户，不清理任何关联

#### 场景：批量删除拒绝当前用户 id

- **当** 调用方调用 `DELETE /api/v1/user?ids=...`
- **且** id 列表包含当前登录用户的 id
- **则** 整个批量必须以 `bizerr` `CodeUserCurrentDeleteDenied` 被拒绝
- **且** 不删除任何用户

#### 场景：空 id 列表在验证时被拒绝

- **当** 调用方调用 `DELETE /api/v1/user?ids=` 时
- **则** 系统必须以验证错误拒绝请求
- **且** 不启动事务

#### 场景：批量删除拒绝范围外用户

- **当** 调用方批量删除多个用户
- **且** 任一目标用户不在当前调用方数据权限范围内
- **则** 整个批量删除被拒绝
- **且** 不删除任何用户

### 需求：sys_user 表必须携带常用查询索引
系统 SHALL 在 `sys_user` 表上维护 `KEY idx_status (status)`、`KEY idx_phone (phone)` 和 `KEY idx_created_at (created_at)`，使按状态、手机或创建时间范围筛选的用户列表查询避免全表扫描。

#### 场景：init 后 sys_user 索引存在
- **当** `make init` 完成数据库初始化时
- **则** `SHOW INDEX FROM sys_user` 除现有主键和 `username` 唯一键外，返回 `idx_status`、`idx_phone` 和 `idx_created_at` 条目
