# 用户消息规范

## 目的

定义用户消息数据结构、未读数查询、列表查询、已读处理、删除、通知铃铛行为、消息面板行为、公告详情导航和前端 Store 轮询。

## 需求

### 需求：用户消息数据库表设计

系统 SHALL 提供 `sys_user_message` 存储用户消息数据。

#### 场景：sys_user_message 表结构

- **当** 检查 `sys_user_message` 表结构时
- **则** 表包含 `id`（BIGINT 主键自增）、`user_id`（BIGINT 接收者用户 ID）、`title`（VARCHAR(255) 消息标题）、`type`（TINYINT 消息类型：1=通知、2=公告）、`source_type`（VARCHAR(50) 来源类型）、`source_id`（BIGINT 来源 ID）、`is_read`（TINYINT 已读标记：0=未读、1=已读）、`read_at`（DATETIME 阅读时间）和 `created_at`（DATETIME 创建时间）
- **且** 在 `user_id` 和 `is_read` 上创建复合索引

### 需求：查询未读消息数

系统 SHALL 提供查询当前用户未读消息数的端点。

#### 场景：获取未读消息数

- **当** 调用方调用 `GET /api/v1/user/message/count` 时
- **则** 系统返回当前登录用户的未读消息数 `{count: number}`

#### 场景：无未读消息

- **当** 当前用户没有未读消息时
- **则** 系统返回 `{count: 0}`

### 需求：查询消息列表

系统 SHALL 提供查询当前用户消息列表的端点。

#### 场景：获取消息列表

- **当** 调用方调用 `GET /api/v1/user/message` 并传入分页参数时
- **则** 系统返回当前用户按创建时间倒序排列的消息
- **且** 每条消息包含 `id`、`title`、`type`、`sourceType`、`sourceId`、`isRead`、`readAt` 和 `createdAt`

### 需求：标记消息为已读

系统 SHALL 提供标记消息为已读的端点。

#### 场景：标记单条消息为已读

- **当** 调用方调用 `PUT /api/v1/user/message/{id}/read` 时
- **则** 系统将该消息的 `is_read` 设置为 1，`read_at` 设置为当前时间
- **且** 仅当前登录用户自己的消息可被修改

#### 场景：标记所有消息为已读

- **当** 调用方调用 `PUT /api/v1/user/message/read-all` 时
- **则** 系统将当前登录用户的所有未读消息标记为已读

### 需求：删除消息

系统 SHALL 提供删除消息的端点。

#### 场景：删除单条消息

- **当** 调用方调用 `DELETE /api/v1/user/message/{id}` 时
- **则** 系统物理删除该消息记录
- **且** 仅当前登录用户自己的消息可被删除

#### 场景：清空所有消息

- **当** 调用方调用 `DELETE /api/v1/user/message/clear` 时
- **则** 系统物理删除当前登录用户的所有消息

### 需求：消息通知铃铛组件

系统 SHALL 在顶部导航栏提供消息通知铃铛组件。

#### 场景：铃铛图标显示

- **当** 用户登录并查看顶部导航栏时
- **则** 显示铃铛图标
- **且** 存在未读消息时，图标右上角显示未读数徽章

#### 场景：无未读消息

- **当** 用户没有未读消息时
- **则** 铃铛图标不显示计数徽章

#### 场景：定期轮询

- **当** 用户登录时
- **则** 前端定期调用未读数端点更新徽章

### 需求：消息面板

系统 SHALL 提供用户点击铃铛图标时打开的消息面板。

#### 场景：打开消息面板

- **当** 用户点击铃铛图标时
- **则** 打开弹出式消息面板并显示消息列表
- **且** 每条消息显示标题、消息类型和时间
- **且** 未读消息有加粗文本或圆点等视觉标记

#### 场景：点击消息导航到详情

- **当** 用户在消息面板中点击某条消息时
- **则** 该消息自动标记为已读
- **且** 用户导航到公告详情页 `/system/notice/detail/{sourceId}`

#### 场景：全部标为已读

- **当** 用户在消息面板中点击"全部标为已读"时
- **则** 系统调用全部标为已读端点并更新面板中所有消息状态

#### 场景：清空消息

- **当** 用户在消息面板中点击"清空"时
- **则** 显示确认对话框，确认后调用清空端点，面板变空

#### 场景：删除单条消息

- **当** 用户在消息面板中点击某条消息的删除按钮时
- **则** 系统调用删除单条端点并从面板中移除该消息

### 需求：公告详情页

系统 SHALL 提供公告详情页。

#### 场景：详情页展示

- **当** 用户打开 `/system/notice/detail/{id}` 时
- **则** 页面显示公告标题、字典渲染的类型、创建者和创建时间
- **且** 以富文本形式显示公告内容

#### 场景：从消息面板导航

- **当** 用户在消息面板中点击消息导航到详情时
- **则** 该消息自动标记为已读

### 需求：Pinia 中的 Message Store

系统 SHALL 提供 Pinia Store 管理用户消息状态。

#### 场景：初始化轮询

- **当** 用户登录并进入宿主工作区时
- **则** 消息 Store 启动间隔轮询获取未读消息数

#### 场景：停止轮询

- **当** 用户退出登录时
- **则** 消息 Store 停止轮询

#### 场景：未读数响应式更新

- **当** 轮询获取到新的未读数时
- **则** Store 中的 `unreadCount` 响应式更新，铃铛徽章相应更新

#### 场景：预留 SSE 扩展点

- **当** 检查消息 Store 轮询实现时
- **则** 轮询逻辑封装在独立方法中，便于未来 SSE 监听替换轮询而不改变外部 Store 接口

### 需求：用户消息轮询必须感知页面可见性

用户消息的未读数轮询 SHALL 监听页面可见性事件。当 `document.visibilityState === 'hidden'` 时，轮询必须暂停。页面重新可见时，消息 Store 必须立即刷新一次未读数并恢复定期轮询。用户退出或 Store 销毁时，底层定时器必须显式停止。

#### 场景：页面隐藏时未读数轮询暂停

- **当** 用户切换到其他标签页时
- **则** 消息 Store 必须暂停未读数轮询
- **且** 不产生 `GET /api/v1/user/message/count` 请求

#### 场景：页面重新可见时立即刷新一次

- **当** 用户切换回当前应用时
- **则** 消息 Store 必须立即触发一次未读数刷新
- **且** 之后恢复定期轮询

#### 场景：退出登录时停止轮询定时器

- **当** 用户退出登录时
- **则** 消息 Store 必须显式停止轮询定时器
- **且** 之后不产生未读数请求
